# Deploy — GS_PROJECT_NAME

[English](README.md) | [中文](README_CN.md)

Kubernetes deploy scaffolding generated by `gs k8s`. Everything here is an
**editable starting point**, not a runtime dependency — tune it for your cluster.

## Layout

```
Dockerfile                 # multi-stage build -> distroless/static:nonroot
conf/app-k8s.properties    # k8s profile: JSON logs to stdout, drain windows
deploy/k8s/                # (default) Kustomize
  base/                    # Deployment, Service, HPA, ServiceMonitor
  overlays/dev             # 1 replica, dev- prefix, :dev image
  overlays/prod            # 3 replicas, tighter resources
  monitoring/              # prometheus-adapter + custom-metric HPA samples
deploy/helm/               # (gs k8s --format helm) Helm chart
  Chart.yaml, values.yaml
  templates/               # deployment, service, hpa, servicemonitor, adapter
```

`gs k8s` emits **one** packaging format. The default is Kustomize (`deploy/k8s`);
`gs k8s --format helm` emits a Helm chart (`deploy/helm`) instead.

## Build & apply

```sh
docker build -t GS_IMAGE:latest .

# Kustomize:
kubectl apply -k deploy/k8s/overlays/dev     # or overlays/prod

# Helm:
helm install GS_APP_NAME deploy/helm         # or: helm template deploy/helm | kubectl apply -f -
```

Dry-run without a cluster:

```sh
kubectl kustomize deploy/k8s/base            # render Kustomize manifests
kubectl apply -k deploy/k8s/overlays/dev --dry-run=client
helm template deploy/helm                    # render Helm manifests
```

## Health probes

Probes hit the actuator **management port (GS_MGMT_PORT)**:

- `startupProbe` → `/startup` — passes once the app has finished starting, so a
  slow boot is not killed by the liveness probe.
- `livenessProbe` → `/health` — process liveness only; a degraded dependency
  does not trigger a restart.
- `readinessProbe` → `/readiness` — fails (503) during graceful drain so the
  pod is removed from Service endpoints before servers stop.

### Tuning the startup budget

The startupProbe **budget** ≈ `periodSeconds × failureThreshold`. The default
(`3 × 40 = 120s`) is deliberately generous. Only after it passes do the liveness
/ readiness probes take over.

What determines cold start is **not** bean wiring (reflection wiring of a few
hundred beans is sub-millisecond). It is the startup-time I/O that starters do
**on purpose** to fail fast: DB/Redis dials, config-center first fetch,
discovery informer sync, and — if you use `starter-migration-gorm` — schema
migrations. Size the budget against your **p99 boot**, not p50.

| App profile | Boot dominated by | `periodSeconds` | `failureThreshold` | budget |
|---|---|---|---|---|
| Small (few starters, no migration) | process init + a couple of dials | 3 | 10 | ~30s |
| Typical microservice (DB + Redis + config-center + discovery) | dials + config fetch + informer sync | 5 | 12 | ~60s |
| Large (many deps, informers, cache warmup) | many dials + warmup | 5 | 24 | ~120s |
| Runs DB migrations at boot | migration DDL (can be minutes) | 10 | ≥ longest migration ÷ 10 | ≥ 300s |

Rules of thumb:

- **Over-provision the startup budget** — it costs nothing when boot is fast
  (the probe passes on the first success). A budget that is too tight kills a
  legitimately slow boot (cold DB, big migration) into a CrashLoop.
- Keep `periodSeconds` small (3–5s) so a fast boot is detected quickly; scale
  the budget with `failureThreshold`, not with a long `initialDelaySeconds`.
- Don't fold readiness into startup: `readinessProbe` should gate on
  `health.Indicator` (e.g. DB reachable) so traffic only arrives once
  dependencies are up. Startup only protects against liveness killing a slow boot.

## Graceful shutdown

`terminationGracePeriodSeconds` (30s) and the container `preStop` sleep (5s)
align with `app.shutdown.timeout` / `app.shutdown.pre-stop-delay` in
`conf/app-k8s.properties`. Rolling updates drain in-flight requests losslessly.

## Metrics

`servicemonitor.yaml` scrapes `/metrics` on the management port and requires the
Prometheus Operator. `/metrics` is only served when **starter-otel** is wired
into the app — otherwise delete the ServiceMonitor.

## Autoscaling on custom metrics

The default HPA scales on CPU. To scale on a business metric (e.g.
requests-per-second) the chain is:

```
actuator /metrics -> ServiceMonitor -> Prometheus -> prometheus-adapter
  -> custom.metrics.k8s.io -> HPA
```

- **Kustomize**: install prometheus-adapter, apply
  `deploy/k8s/monitoring/prometheus-adapter-configmap.yaml` into its namespace,
  then apply `deploy/k8s/monitoring/hpa-custom-metric.yaml` **instead of**
  `base/hpa.yaml` (both would give the Deployment two autoscalers).
- **Helm**: set `autoscaling.customMetric.enabled=true` (the HPA gains a Pods
  metric) and, if you want the chart to render the adapter rule too,
  `prometheusAdapter.enabled=true`.

The adapter ConfigMap maps `http_requests_total` to a per-pod
`http_requests_per_second`; adjust the series/query to match your app's metrics.

## Pod metadata

The Deployment injects Pod fields via the Downward API (`GS_POD_NAME`,
`GS_POD_NAMESPACE`, `GS_POD_IP`, `GS_NODE_NAME`, `GS_POD_SERVICE_ACCOUNT`) and
mounts labels/annotations at `/etc/podinfo`. Read them in the app with
`go-spring.org/spring/podinfo`:

```go
gs.Object(&podinfo.PodInfo{})
```
