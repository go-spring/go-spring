# Claude 最佳实践

沉淀在 Go-Spring 里和 Claude（及其它 AI 编码助手）协作的经验，按固化程度分两类：

- **硬约定**：无需判断即可照做，维护在 `CLAUDE.md` / Skill / hooks 里自动生效。
- **软约定**：依赖判断、随场景变化，写死反而误伤，留在文档里供人参考。

## 硬约定

已经固化的规则统一维护在根目录 `CLAUDE.md`：[go-spring/go-spring · CLAUDE.md](https://github.com/go-spring/go-spring/blob/master/CLAUDE.md)。

一条经验若“无需上下文、无需权衡，照做即可”，就沉淀到那里，并从软约定中移除。

## 软约定

依赖判断、随场景变化的经验，刻意不固化成规则：

- 拆 PR 还是合并成一个，看改动的内聚性，而非行数。
- 是否抽象：三行相似代码通常好过一个过早的抽象，收敛时机看具体情况。
- 先追问还是直接动手，看任务的模糊程度和可逆性。

## 维护约定

- 软约定尽量标注来源（哪次协作、什么问题触发），便于判断是否过时。
- 成熟、可机械执行的软约定，搬进 `CLAUDE.md` 或 Skill 后从本文档移除。
- 某条规则频繁误伤，就从 `CLAUDE.md` 降级回软约定，并记录原因。
