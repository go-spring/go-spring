Go-Spring 实战第 16 课 —— 容器运行流程：注册、解析、注入、运行、关闭 ​

前面几篇文章，我们已经把 Go-Spring 依赖注入的使用方式全部讲完了：从字段注入和构造函数注入，到单 Bean 注入和集合注入，再到结构体指针、构造函数、函数指针，以及 Bean 注册、Bean 元数据和条件注册。

本篇咱们来梳理一下 Go-Spring 容器的运行流程。在我们理解了这套流程之后，就能把前面讲过的使用方式和容器内部的动作对应起来。这样在遇到问题时，也知道应该按照什么样的流程进行排查。

运行流程 ​

Go-Spring 容器的运行流程可以拆成如下五个阶段：

  • 注册阶段：收集项目中所有的 Bean 和 Module 定义。
  • 解析阶段：根据配置、条件和配置类，决定本次容器运行保留哪些 Bean。
  • 注入阶段：从 Root Bean 出发，创建对象、注入依赖、绑定配置、执行 Init，并收集 Destroy。
  • 运行阶段：所有对象已经处理完毕，业务代码使用普通的 Go 引用，容器没有运行时依赖注入。
  • 关闭阶段：容器执行已收集的 Destroy 回调，按照依赖顺序释放 Bean 持有的资源。

从这个流程可以看出，Go-Spring 的依赖装配集中发生在启动期。等应用真正运行起来以后，业务代码拿到的是普通对象和接口引用，正常调用路径上不会再有容器相关的反射成本。

注册阶段 ​

注册阶段的职责，是把各种注册入口提交的 Bean 定义收集起来。

Go-Spring 提供了以下几种注册 Bean 的方式：

  • gs.Provide()：将 Bean 注册到全局注册表。
  • gs.Module()：将 Module 注册到全局注册表，我们可以在 Module 里面根据配置注册一些 Bean。
  • gs.Group()：gs.Module() 的特化版本，用于专门注册一组类型相同的 Bean。
  • app.Provide()：将 Bean 注册到当前 App 的容器。
  • .Configuration()：把一个 Bean 标记为配置类，后续会扫描它的方法生成子 Bean。

在这个阶段，容器还不会判断这些 Bean 和 Module 最终是否生效，也不会创建对象。这样设计的原因是：

很多 Bean 和 Module 的注册发生在 Go 的 init() 阶段。此时配置还没有加载进来，当前应用到底会不会使用这些 Bean 或 Module 也还不知道。如果注册时就创建对象，导入一个包就可能打开连接、读取配置、启动后台逻辑，副作用就会很难控制。所以这里先收集对象的定义，而不是创建对象本身。

解析阶段 ​

在解析阶段开始前，容器必须已经设置好了配置。随后，条件判断、模块激活和后续配置绑定都会以这份配置为依据。

解析阶段仍然不会创建对象，而是主要做以下五件事：

第一，合并 Bean 定义。

gs.Provide() 将 Bean 注册到全局注册表，app.Provide() 将 Bean 注册到当前容器。也就是说，Bean 定义存在两种不同的注册位置。因此，解析阶段开始后，容器首先会把这两部分合并成一组 Bean 定义。

第二，执行满足条件的 Module。

gs.Module() 本质上是一个在解析阶段执行的回调。用户可以在回调里面根据传入的配置，选择性注册当前应用需要的 Bean。Module 在执行之前会先进行配置条件判断，只有配置条件成立才会调用 Module 函数。这种机制非常适合 Starter：一个 Starter 通常要注册一组完成某项功能的 Bean，并且通过配置开关决定这组 Bean 是否进入当前应用。

第三，扫描 Configuration 配置类。

接下来，容器会找到带有 Configuration 标记的 Bean，然后扫描它的方法，并根据规则创建和注册子 Bean。这些子 Bean 依赖于配置类 Bean 的存在，因此会自动添加一个依赖父 Bean 存在的条件，也就是 OnBeanID(配置类 Bean)。这里所谓的子 Bean 和其他 Bean 并无不同，区别只是子 Bean 由配置类 Bean 产生。

第四，执行 Bean 条件裁剪。

接下来，容器会检查每个 Bean 的条件，只有条件成立的 Bean 才会保留下来。条件不成立的 Bean 会被标记为删除。这个步骤会把配置开关、Bean 存在性条件，以及配置类子 Bean 自动附加的条件统一收束起来。裁剪完成之后，后续注入阶段就只需要面对本次应用真正生效的 Bean 集合了。

第五，检查重复 Bean。

在 Go-Spring 里，类型和名称一起构成 Bean 的唯一标识。在容器中，每个 Bean 也应该是唯一的。但是注册阶段可能会出现类型和名称相同、但由不同条件控制的 Bean。不过，经过条件裁剪之后，最终保留下来的 Bean 必须是唯一的。因此，容器会在解析阶段的最后进行重复 Bean 检查。

注入阶段 ​

到了注入阶段，容器才会真正创建对象。不过，Go-Spring 不是创建所有已经解析过的 Bean，而是从 Root Bean 出发，沿着依赖关系递归创建。

为什么要这样做呢？因为 Go 里面存在空白导入。一个包被空白导入之后，里面的 Bean 定义可能已经完成注册，但这并不代表它一定会被当前应用使用。因此，这里采用了类似 GC Roots 的概念：重要的 Bean 一定可以从根出发访问到。如果访问不到，就说明它对当前应用不重要，也就不需要被创建。

在 Go-Spring App 的运行模型里，Runner 和 Server 类型的 Bean 是默认的 Root Bean。App 持有 Runner 和 Server 数组，容器会从这些入口开始创建对象，并沿着依赖关系向下递归。理论上，当前应用需要的其他 Bean，都应该能从 Runner 或 Server 出发被访问到。

看一下每个 Bean 在处理时都会经历哪些步骤：

  1. 循环依赖检测（包括 DependsOn 和构造函数依赖两种情况）
  2. 创建对象（解析构造函数参数，执行构造函数，获取返回值）
    1. 如果构造函数返回失败，会阻止容器继续启动
  3. 为对象注入依赖和配置
    1. 如果遇到 gs.Dync[T] 类型的字段，会登记到动态配置对象列表里。
    2. 如果发现 lazy 标签的字段，会先记录下来。
  4. 执行对象的 Init 回调
  5. 将 Bean 标记为 wired

在所有 Bean 都注入完成之后，容器还会做两件事：

  1. 执行所有 lazy 标签字段的注入。
  2. 对所有的 Destroy 回调进行排序。

在 Bean 的初始化过程中，不要依赖 lazy 标签的字段。此时这些字段还没有被注入，通常仍是零值，直接使用可能导致 nil 访问或者逻辑错误。我们可以在运行时访问它们，因为容器会在注入阶段的最后统一处理这些字段，确保它们在运行阶段开始前完成注入。

运行阶段 ​

Go-Spring 不提供运行时注册和获取 Bean 的操作。一方面，对于启动框架而言，业务通常不太需要运行时注册 Bean。另一方面，如果确实需要在运行时使用某个 Bean，也可以在启动期就准备好它。容器只关心启动期注入，也就可以避免在运行期处理并发装配的问题。

对于每个请求到来之后，如果需要做某种请求级对象的注入，那应当属于 Web 框架或业务框架的能力，而不是 IoC 容器本身的职责。

使用启动期模型，可以大大减少对运行时性能的担忧。正常业务调用路径使用的是已经注入好的对象引用，不需要再经过容器反射查找和装配。

不过，我们说 Go-Spring 容器没有运行时依赖注入，并不代表它完全不参与运行期能力。声明为 gs.Dync[T] 类型的字段，可以在运行期根据配置变化触发动态刷新。

关闭阶段 ​

IoC 容器关闭是程序退出的一个关键环节。容器会在这个阶段执行所有已收集的 Destroy 回调。

通常来说，如果 A 依赖 B，那么关闭时会先执行 A 的 Destroy，再执行 B 的 Destroy。这个顺序来自业务依赖关系本身：A 的运行建立在 B 可用的基础上，所以退出时也应该先让 A 停止使用 B，并完成自己的收尾动作，然后再释放 B。如果反过来先销毁 B，A 在关闭过程中就可能访问到已经失效的下游资源。

另外，Destroy 回调执行失败也不会中断后续 Destroy，容器只会记录日志并继续向后执行。
