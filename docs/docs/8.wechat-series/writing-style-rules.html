Go-Spring 微信文章写作规约 ​

本规约用于约束 Go-Spring 微信正文文章的编写和改写。

目标是技术准确、主题集中、表达自然。标题优化、转场润色、示例调整和语气修改，都不能改变已有 API、配置项、生命周期顺序、优先级、错误语义和设计边界。

1. 结构 ​

  1. 主标题使用统一格式：
text
Go-Spring 实战第 N 课 —— 大主题：具体能力或问题

  1. 开头从上一篇内容或真实工程问题进入。可以简短说明范围，但不要写成目录式预告，避免 本文将介绍...、本篇要解决的问题有...、首先来看目录... 这类句式。

  2. 每篇只围绕一个明确主题展开。主题可以包含多个分支或多个 API 入口，但它们必须共同回答同一个问题。

  3. 小节标题要具体，可以是对象名、能力名、阶段名或问题名。避免 总结、小结、最佳实践、注意事项 这类空泛标题。

  4. 结尾要收住当前主题，说明这块能力在 Go-Spring 整体模型里的位置。最后一个小节已经自然收束时，可以不单独加结尾段。不要写下一篇预告。

2. 内容 ​

  1. 正文必须有语义主线，不能只有 API 清单。机制型文章讲清规则和边界；概念型文章讲清普通 Go 写法的约束，以及 Go-Spring 接管和不接管的部分；API 入口型文章讲清入口解决的问题、对应语义和适用场景。

  2. 示例必须服务于当前小节。示例前后至少要说明它证明什么；示例后解释 Go-Spring 语义、适用场景或边界，不要逐行复述代码。

  3. 规则型主题可以先给总规则，再按来源、数据形态或阶段拆开解释。机制讲清后，可以补充建模、排查或选择建议，但建议必须来自前文语义。

  4. 进入 IoC、生命周期、日志、HTTP 等新体系时，可以先讲普通 Go 写法和复杂工程中的痛点，再引出 Go-Spring 抽象。不要把 Go-Spring 写成替代 Go 语言习惯的框架。

  5. 禁止使用 依赖图、对象图、Bean 依赖图、静态对象图 以及类似以“图”来组织 IoC、装配或运行期对象关系的概念。

3. 表达 ​

  1. 默认读者熟悉 Go 工程开发。不要解释 struct、interface、error 处理、Go module、构建部署这类基础知识。

  2. 专有名词保持稳定写法，例如 Go-Spring、Properties、Path、Reader、Provider、Profile、Bean、IoC 容器、Starter、Map、Slice。配置 path、结构体标签、API 名称、类型名、配置项和命令行参数使用反引号标记。

  3. 语气自然、克制、连贯。可以使用 我们、咱们、大家，也可以少量使用设问和口语化转场，但不能写成聊天记录、营销稿或手册条目。

  4. 讲设计取舍时，先给约束条件，再推出结论。从通用工程问题转入框架能力时，要写清这是 Go-Spring 的能力、Go-Spring 的设计，还是 Go 语言或业务代码本身的约束。

  5. 加粗、编号列表和表格都可以使用，但必须服务于核心判断、规则顺序或短枚举，不要把复杂推理塞进表格。

4. 自检 ​

生成或改写后，只检查这些问题：

  1. 标题是否符合统一格式。
  2. 开头是否从上一篇或工程问题进入，而不是列目录。
  3. 正文是否围绕一个主题展开，而不是只有 API 清单。
  4. 示例是否有语义解释，而不是只贴代码。
  5. 小节标题是否具体。
  6. 是否讲清 Go-Spring 的能力、边界和使用建议。
  7. 是否避免改变 API、配置项、生命周期顺序、优先级、错误语义和设计边界。
  8. 是否避免 依赖图、对象图 等图式概念。

一句话要求：从真实工程复杂性出发，用自然但克制的技术文章语气，带读者看到 Go-Spring 的抽象、代码入口、适用边界、设计取舍和工程使用建议。
