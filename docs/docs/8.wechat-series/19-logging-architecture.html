Go-Spring 实战第 19 课 —— 日志架构：基于标签路由的 Log4j 2 架构 ​

一个程序要想正常运行，既需要配置体系提供运行参数，也需要运行模型组织标准化流程。不过，程序成功启动以后，我们还需要知道它在什么时间做了什么、是否出现异常，并为后续排查留下依据。这就是日志需要解决的问题。本篇咱们就来看一下 Go-Spring 是如何构建一套完整日志体系的。

对于简单程序来说，把日志内容写到控制台可能就够了。但在真实项目中，日志往往面临更多要求：业务日志需要写入文件，错误日志可能需要单独输出；线上环境通常采用便于采集的 JSON 格式，本地开发则更适合阅读普通文本。如果这些差异都由业务代码进行处理，日志调用就会与分类方式、输出目标和日志格式耦合在一起。

Go-Spring 的日志体系借鉴了 Log4j 2 的分层架构，也参考了 Go 标准库 slog 的结构化日志设计。这样一来，它既保留了分层组件灵活处理日志的能力，也延续了通过统一接口记录结构化字段的方式。在此基础上，Go-Spring 引入了 Tag（标签）来补充日志语义、实现日志路由，最终形成了一套基于标签路由的 Log4j 2 架构。

一个日志例子 ​

假设我们需要在订单创建完成之后记录一条业务日志，使用 Go-Spring 可以这样写：

go
var TagBizOrder = log.RegisterBizTag("order", "create")

func CreateOrder(ctx context.Context, orderID int64) {
	log.Info(ctx, TagBizOrder,
		log.Int("order_id", orderID),
		log.String("status", "created"),
		log.Msg("订单创建完成"),
	)
}

这段代码记录了一条 INFO 级别的订单创建日志。它与常见 Go 日志库的使用方式既有相同之处，也有不同之处。

相同之处在于，它使用 Int、String 和 Msg 等 Field 记录结构化内容，既便于程序处理，也避免了业务代码自行拼接日志文本；不同之处在于，Go-Spring 的日志 API 增加了一个 Tag 参数，也就是标签。

Tag 本质上是一个描述日志业务语义的字符串。例如，上面代码对应的标签是 _biz_order_create，它由 biz、order 和 create 三部分组成。其中，biz 表示业务日志，order 表示订单相关日志，create 则进一步表示订单创建日志。三者共同构成了从大类到小类的层级关系。

为什么需要 Tag 呢？常见的 Go 日志库通常提供一个全局日志句柄，或者为不同输出目标创建不同的日志句柄。使用全局句柄时，我们很难根据日志语义灵活控制输出位置；使用多个句柄时，同一个模块又可能需要持有多种句柄。也就是说，日志分类与输出策略耦合在了一起，业务代码不得不关心日志最终写到哪里。

Go-Spring 通过 Tag 将日志语义与输出目标分离开来。业务代码只需要使用固定的 Tag 描述日志，日志系统则可以根据 Tag 进行筛选和路由。例如，匹配 _biz 可以处理所有业务日志，匹配 _biz_order 可以处理订单日志，匹配 _biz_order_create 则可以进一步处理订单创建日志。至于这些日志最终输出到控制台、普通文件还是独立文件，则由日志配置决定。

这种根据层级进行日志路由的思路来自 Java 世界的 Log4j 2。不过，Go 没有 Java 的 Class 体系，因此 Go-Spring 使用 Tag 来承载层级关系。与代码所属的 Class 相比，Tag 描述的是更加具体的业务语义，而且不会受到代码位置的限制。

理解了 Tag 的作用之后，咱们再来看看这条日志最终会输出成什么样。

采用文本格式时，这条日志可以输出为便于阅读的文本：

text
[INFO][2026-06-06T10:30:00.000][order.go:18] _biz_order_create||order_id=10001||status=created||msg=订单创建完成

调整输出策略后，同一处业务代码又可以生成 JSON：

json
{"level":"info","time":"2026-06-06T10:30:00.000","fileLine":"order.go:18","tag":"_biz_order_create","order_id":10001,"status":"created","msg":"订单创建完成"}

两种输出格式虽然不同，但都来自同一处业务代码，差异只发生在日志系统内部。这种隔离能力，正是由 Go-Spring 的分层架构提供的。

分层架构 ​

Go-Spring 参考了 Log4j 2 的分层架构，把日志从调用到输出的过程拆成下面六个层次：

text
┌──────────────────────────────────────────────┐
│                 应用层 API                   │
│  Trace / Debug / Info / Warn / Error ...    │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│                  标签层 Tag                  │
│        描述日志语义，找到对应 Logger          │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│                 Logger 层                    │
│          级别过滤、同步或异步调度             │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│                Appender 层                   │
│          选择控制台、文件等输出目标            │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│                 Layout 层                    │
│          组织一条完整日志的整体格式            │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│                 Encoder 层                   │
│        编码基础类型、数组和嵌套对象字段         │
└──────────────────────────────────────────────┘

这六层并不是对日志流程的简单切段，而是对不同变化的拆分。一条日志从业务代码走到最终输出，需要依次回答几个问题：业务代码怎样描述日志、这条日志属于哪一类；是否需要处理、应该怎样调度；写到哪里、整条日志采用什么格式；每个字段又该怎样编码。这些问题发生变化的原因并不相同，因此不应该交给同一个组件共同承担。

其中，业务调用和日志语义通常相对稳定。一次订单创建始终是一条订单创建日志，不应该因为输出目标从控制台改为文件，或者处理方式从同步改为异步，业务代码就跟着修改。因此，Go-Spring 使用统一的应用层 API 描述日志内容，再用 Tag 表达业务语义，从而把稳定的调用点与易变的处理策略隔开。

日志进入处理系统以后，运行策略和输出目标仍然是两件事。日志级别决定一条日志是否处理，同步或异步决定怎样调度；控制台、文件或其他介质，则决定日志最终写到哪里。将两者分别交给 Logger 和 Appender 以后，同一种运行策略就可以连接多个输出目标，同一个输出目标也可以被不同的 Logger 复用。

确定输出目标以后，日志格式仍然可以分成两个层次：Layout 负责组织一条完整日志的结构，Encoder 负责把不同类型的 Field 编码成具体内容。这样一来，整体格式不必了解每一种字段类型；字段编码也不必关心时间、级别和 Tag 在整条日志中的位置。

按照这样的拆分，每一层只承接一种变化。业务代码保持稳定，日志系统则可以通过配置替换组件、调整策略并重新组合处理管线，这就是六层架构背后的设计思路。

了解了整体结构后，下面咱们再分别看各层的作用。

应用层 API ​

应用层 API 是业务代码使用日志系统的入口，解决的是“怎样记录一条日志”的问题。它会将业务代码中的一次日志调用转换成日志系统内部统一的 Event。

业务代码调用日志 API 时，需要提供 Context、Tag 和日志内容。应用层会先根据日志级别判断是否需要继续处理；如果需要，再收集日志时间、调用位置、Tag、业务 Field 和从 Context 中提取的公共信息，最终将它们整理成一个 Event。

Go-Spring 内置了覆盖 Trace、Debug、Info、Warn、Error、Panic 和 Fatal 的完整级别 API。每个级别既支持使用 Field 记录结构化内容，也提供以 f 结尾的格式化接口，例如 Info 和 Infof。其中，Trace 和 Debug 还采用延迟构造 Field 的方式，避免在对应级别未开启时产生不必要的开销。

应用层只要求业务代码描述清楚当前事件，不需要关心后续使用哪个 Logger、写到哪个 Appender，或者采用哪种 Layout。

标签层 ​

Tag 用来描述日志的语义分类，解决的是“这条日志属于哪一类”的问题。它关注的是日志表达的业务含义，而不是调用代码位于哪个 Go 包。

业务代码只持有稳定的 Tag，不直接持有具体 Logger。日志系统初始化后，每个 Tag 会根据精确规则或层级通配规则绑定到相应的 Logger；如果没有匹配到专门的 Logger，则交给 Root Logger。

Go-Spring 内置了 RegisterAppTag、RegisterBizTag 和 RegisterRPCTag，分别用于注册应用日志、业务日志和外部调用日志；我们也可以通过 RegisterTag 注册自定义分类。此外，TagAppDef 和 TagBizDef 可以作为应用日志和业务日志的默认标签。

Tag 将稳定的业务语义与可变化的 Logger 配置隔开。这样，日志级别、调度方式和输出目标发生变化时，业务调用仍然可以保持不变。

Logger 层 ​

Logger 是日志处理过程中的调度层，解决的是“是否处理、怎样处理”的问题。它负责级别过滤和事件调度，但不关心日志最终写入哪个文件，也不负责把字段编码成具体格式。

Logger 首先根据级别范围判断 Event 是否需要继续处理。通过级别过滤后，它会把 Event 交给一个或多个 Appender；同步 Logger 在当前 goroutine 中完成调用，异步 Logger 则先将 Event 放入缓冲区，再由后台 goroutine 继续处理。

Go-Spring 内置的 Logger 可以分成三类：

  • SyncLogger 和 AsyncLogger 是组合式 Logger，需要搭配一个或多个 Appender 使用，适合自由组合处理管线。
  • ConsoleLogger、FileLogger 和 RollingFileLogger 是集成式 Logger，内部已经组合了相应的 Appender。其中，RollingFileLogger 还支持文件滚动、级别分离以及同步或异步写入。
  • DiscardLogger 会直接丢弃 Event，适合明确关闭某类日志。

组合式 Logger 更灵活，集成式 Logger 配置更简单。无论使用哪一种，Logger 都只负责过滤和调度；真正的输出工作仍然由 Appender 完成。

Appender 层 ​

Appender 负责选择并管理日志的输出目标，解决的是“写到哪里”的问题。

Appender 收到 Event 后，会先调用 Layout 完成格式化，再将生成的字节写入对应目标。由于控制台、普通文件和滚动文件需要管理不同的资源与生命周期，因此 Go-Spring 分别提供了不同的 Appender 实现。

Go-Spring 内置了四种 Appender：

  • ConsoleAppender：将日志写入标准输出。
  • FileAppender：将日志持续写入指定文件。
  • RollingFileAppender：按时间滚动文件，并清理超过保留时间的历史文件。
  • DiscardAppender：接收 Event，但不产生实际输出。

一个 Logger 可以连接多个 Appender，因此同一条日志可以同时写入控制台和文件。Appender 也可以被不同的 Logger 复用，从而让输出目标与过滤、调度策略保持独立。

Layout 层 ​

Layout 负责组织一条完整日志的格式，解决的是“整条日志长什么样”的问题。

一个 Event 不仅包含业务 Field，还包含级别、时间、调用位置、Tag 和上下文信息。Layout 决定保留哪些内容、按照什么顺序组织，以及采用什么结构写入缓冲区。

Go-Spring 内置了两种 Layout：

  • TextLayout：生成便于开发人员直接阅读的单行文本。
  • JSONLayout：生成便于日志采集和分析系统解析的 JSON 对象。

两种 Layout 处理的是同一个 Event，只是最终的表现形式不同。Layout 负责整条日志的结构，其中每个 Field 的具体写法则交给 Encoder。

Encoder 层 ​

Encoder 负责结构化 Field 的具体编码，解决的是“每个字段怎样转换成最终字节”的问题。

前面示例中的 log.Int、log.String 和 log.Msg 都会生成 Field。每个 Field 都会保存字段名、字段类型和字段值。编码时，Field 会根据自身类型调用 Encoder 对应的方法，完成对布尔值、整数、浮点数、字符串、数组和嵌套对象的处理。

Go-Spring 内置了两种 Encoder：

  • JSONEncoder：按照 JSON 语法编码字段，并保留数字、布尔值、数组和对象的类型。
  • TextEncoder：将顶层字段编码成 key=value 形式；遇到数组和嵌套对象时，仍然使用 JSON 结构表达内部内容。

Encoder 只关心单个 Field 的类型和值，不关心它位于日志头部还是业务字段区域。这样一来，Layout 就可以专注于整体结构，而不必重复实现每一种数据类型的编码逻辑。

Go-Spring 日志架构 ​

Go-Spring 的日志架构将分类、路由、过滤、调度、输出和编码拆分开来。从业务侧看，我们只需要描述清楚当前事件；至于日志写到哪里、采用什么格式、以什么方式写入，则交给日志系统处理。这样，无论输出目标、日志格式还是写入方式如何变化，业务调用都可以保持稳定。
