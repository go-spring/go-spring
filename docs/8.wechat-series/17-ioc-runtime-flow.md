# Go-Spring 实战第 17 课 —— 容器运行流程：启动失败如何定位到具体阶段

前面几篇把 Bean 创建、注入方式、注册入口、条件和 Profile 分别拆开讲。真正排查启动问题时，单个 API 只是局部线索，还需要知道这些信息在 Go-Spring 容器里按什么顺序生效。

同样是“容器启动失败”，原因可能发生在注册参数、条件裁剪、依赖注入、初始化回调、Runner 执行、Server 启动或退出清理上。阶段判断错了，排查就会在无关代码里绕很久。

Go-Spring 容器从启动到关闭，可以按注册、解析、注入、运行和关闭五个阶段理解。这不是为了给生命周期起名字，而是为了在错误发生时先缩小问题范围，再回到对应阶段看注册语句、条件、依赖或回调。

## 注册阶段

注册阶段先回答一个问题：本次容器有哪些候选 Bean 定义。Go-Spring 的注册来源可以分成全局注册和当前应用注册。

下面的包级注册只是提供候选定义，还没有开始创建对象。

```go
func init() {
	gs.Provide(NewService)
	gs.Module(condition, moduleFunc)
}
```

这些注册通常发生在包的 `init()` 中，Go-Spring 会把它们保存到全局注册表。每次创建应用容器时，Go-Spring 会从全局注册表复制一份候选定义，再生成当前容器的独立视图。

当前应用也可以追加只属于本次启动的注册。这个入口常用于测试替换、临时集成或只在当前进程里生效的对象。

```go
func main() {
	gs.Configure(func(app gs.App) {
		app.Provide(NewTestDependency)
	}).Run()
}
```

`app.Provide()` 不会污染全局注册表，它只影响当前应用实例。这个边界对测试替换和局部集成很重要：全局注册提供通用候选，当前应用注册补充本次启动自己的差异。

## 解析阶段

解析阶段决定“哪些候选定义在本次启动中生效”。Go-Spring 会合并 `app.Provide()`、`gs.Provide()`、`gs.Module()`、`gs.Group()` 和 `Configuration` 产生的 Bean 定义，然后计算条件。

下面这段注册说明，条件裁剪发生在对象创建之前。

```go
func init() {
	gs.Provide(NewRedisClient).Condition(
		gs.OnProperty("redis.enabled").HavingValue("true"),
	)
}
```

如果 `redis.enabled` 不满足条件，`NewRedisClient` 对应的 Bean 定义会在解析阶段被裁掉，后续不会进入依赖注入，也不会执行初始化回调。

解析阶段还会做冲突检测。Go-Spring 不允许同类型、同名称的 Bean 被悄悄覆盖，因为覆盖会让依赖结果变得不确定。如果同一个类型和名称下出现多个候选，容器会在解析阶段启动失败。

因此，某个 Bean 没有被注入时，不要直接跳到字段注入代码里找原因。先确认它是否真的通过了解析阶段：条件是否满足，名称是否冲突，Module 是否成功展开，Profile 是否命中。

## 注入阶段

解析完成后，Go-Spring 会从 root bean 出发，按依赖关系递归创建对象并注入依赖。root bean 通常来自实现了 `gs.Runner` 或 `gs.Server` 的 Bean，也可以通过 `app.Root()` 显式指定。

下面的 `app.Root()` 指定的是依赖图从哪里展开，而不是强制创建所有已注册 Bean。

```go
func main() {
	bootstrap := &Bootstrap{}

	gs.Configure(func(app gs.App) {
		app.Root(bootstrap)
	}).Run()
}
```

从 root bean 可达的依赖会被创建，不可达的候选 Bean 即使注册成功，也不会因为注册本身被无条件实例化。注册提供候选，root bean 才决定本次依赖图的入口。

注入阶段会完成几件关键工作：建立类型和名称索引，从 root bean 递归解析依赖，创建 Bean 实例，执行字段或构造函数注入，处理延迟注入关系，执行初始化回调，并为后续销毁顺序建立依赖关系。

这一阶段暴露的问题通常更接近对象图本身：找不到依赖、同类型多个候选无法选择、循环依赖不被当前创建方式支持，或者初始化回调返回错误。和解析阶段不同，这时 Bean 已经被选中进入本次对象图，问题发生在创建和连接对象的过程中。

## 运行阶段

注入完成后，Go-Spring 应用才进入运行阶段。这里需要强调的是：Go-Spring 的 IoC 容器不是运行时 Bean 查找器。

依赖关系在启动期已经解析完成，业务代码运行时直接使用已经注入好的对象。Go-Spring 不提供运行时 `getBean()` 之类的动态查找入口，也不鼓励业务逻辑在请求过程中再回到容器里寻找对象。

这个设计带来几个结果：依赖错误会尽量在启动阶段暴露；运行时代码是普通 Go 对象调用；容器不需要为每次业务调用保留动态查找能力；对象关系也更容易从构造函数、字段和注册语句里读出来。

应用级运行流程还会继续执行 Runner、启动 Server，并等待 Server Ready。严格说，这些属于应用生命周期，不只是 IoC 容器本身。但排查时可以把它们放在“运行阶段”理解：对象图已经构建完成，失败更多来自启动任务、端口监听或长期服务本身。

## 关闭阶段

应用退出时，Go-Spring 会让容器进入关闭阶段。销毁顺序来自依赖关系，而不是注册顺序。

```go
func init() {
	gs.Provide(NewRepository)
	gs.Provide(NewService)
}
```

如果 `Service` 依赖 `Repository`，初始化时需要先准备好 `Repository`，再创建 `Service`。退出时顺序相反，Go-Spring 会先销毁 `Service`，再销毁 `Repository`。

这个逆序很关键。上层对象清理时，底层依赖仍然可用；比如服务先停止业务处理，再关闭数据库连接，就不会让清理逻辑面对已经释放的资源。

销毁阶段会执行 Bean 的 Destroy 回调，用来关闭连接、停止后台任务、刷写缓冲区等。销毁失败会被记录，但退出流程仍然要继续收束其他资源。

## 阶段定位

启动失败时，先定位阶段，再回头看对应 API，排查范围会小很多。

| 阶段 | 常见问题 |
|------|----------|
| 注册 | 注册参数不合法，启动后才追加全局注册 |
| 解析 | 条件不满足，Profile 未命中，Bean 冲突，Module 展开失败 |
| 注入 | 找不到依赖，多个候选无法选择，循环依赖，初始化回调失败 |
| 运行 | Runner 返回错误，Server 监听失败，Ready 一直没有触发 |
| 关闭 | Server 停止阻塞，Destroy 回调异常，后台任务没有响应退出 |

例如，报错是“同类型同名称 Bean 冲突”，入口在解析阶段；如果是找不到 `UserRepository`，入口在注入阶段；如果是端口监听失败，通常已经走到 Server 启动阶段。先判断阶段，再看该阶段负责的规则，问题会更容易落到具体代码。

## 容器运行流程

注册、解析、注入、运行、关闭连起来，就是 Go-Spring 容器的排错地图。注册阶段收集候选，解析阶段裁剪并检查冲突，注入阶段构建对象图，运行阶段使用已注入对象，关闭阶段按依赖逆序释放资源。

这条流程也解释了 Go-Spring IoC 的一些设计边界：为什么接口要显式导出，为什么按需创建，为什么不提供运行时 `getBean()`。这些取舍都围绕同一个目标展开：让依赖关系在启动期定型，让运行期保持直接、明确和可排查。
