# Go-Spring 实战第 17 课：容器运行流程：启动失败如何定位到注册、解析、注入、运行和关闭阶段

前面几篇把 Bean、注入、注册和条件分别拆开讲。真正排查启动问题时，单个 API 只是局部线索，还需要知道这些信息在 Go-Spring 容器启动过程中先后流向哪里。

同样是“容器启动失败”，原因可能发生在注册参数、条件裁剪、依赖注入、Runner 执行、Server 启动或退出清理上。阶段判断错了，排查就会在无关代码里绕很久。

Go-Spring 容器从启动到关闭可以按五个阶段理解，即注册、解析、注入、运行和关闭。这个顺序也是排查问题时最实用的地图。

## 注册阶段先收集全局定义和当前实例定义

Go-Spring 的注册来源可以分成全局注册和当前容器注册。

全局注册通常发生在包的 `init()` 中。

```go
func init() {
	gs.Provide(NewService)
	gs.Module(condition, moduleFunc)
}
```

这些注册信息会保存到全局注册表中。每次创建容器时，Go-Spring 都会从全局注册表复制一份候选定义，再生成独立容器实例。这样全局注册负责提供候选 Bean，具体容器仍然保持相对独立。

当前容器注册主要来自 `app.Provide()`。

```go
gs.Configure(func(app gs.App) {
	app.Provide(NewTestDependency)
})
```

它只作用于当前应用或当前测试容器。测试隔离正是建立在这个边界上。

## 解析阶段决定本次启动哪些 Bean 生效

容器启动后，Go-Spring 先合并所有 Bean 来源，包括 `app.Provide()` 注册的 Bean、`gs.Provide()` 注册的 Bean、`gs.Module()` 和 `gs.Group()` 生成的 Bean，以及 `Configuration` 模式导出的 Bean。

合并完成后，容器会执行条件裁剪。每个 Bean 的 Condition 都会被计算，满足条件的 Bean 保留，不满足条件的 Bean 剔除。

最后进入冲突检测。Go-Spring 不允许同类型、同名称的 Bean 覆盖，因为覆盖会让依赖结果变得不确定。一旦发现冲突，容器会直接启动失败。

这一阶段决定的是“本次启动到底有哪些 Bean 定义生效”。如果某个 Bean 没有被注入，首先要看它是否在解析阶段被条件裁剪，或者是否因为冲突导致启动中止。

## 注入阶段从 root bean 展开依赖图

解析完成后，Go-Spring 容器会从 root bean 出发，按依赖关系递归创建 Bean 并注入依赖。

核心流程按顺序展开。

1. 为所有 Bean 建立类型和名称索引。
2. 从 root bean 开始递归创建依赖。
3. 创建 Bean 实例后立即执行依赖注入。
4. 注入过程中检测循环依赖。
5. 记录延迟注入字段，后续统一处理。
6. 注入完成后执行 `init` 回调。
7. 对 destroy 依赖关系进行拓扑排序。
8. 清理临时元数据。

注入阶段完成后，本次启动需要的 Bean 都已经创建、注入并初始化。这里暴露的问题通常是找不到依赖、多个候选无法选择、循环依赖不被支持，或者初始化回调返回错误。

## 运行阶段容器不再参与业务对象查找

Go-Spring 使用的是启动期模型。依赖关系只在启动阶段解析，应用进入运行阶段后，容器不再参与业务调用，也不提供运行时 `getBean()` 之类的动态获取能力。

这个设计带来几个结果。

- 依赖错误在启动阶段暴露。
- 运行时代码直接调用已注入对象。
- 容器不需要保留大量运行期元数据。
- 运行时没有反射式 Bean 查找开销。

因此，应用逻辑通常直接使用构造函数参数、字段注入和显式工厂，而不是在运行期向容器要对象。

## 关闭阶段按依赖逆序执行销毁

应用收到退出信号后，Go-Spring 容器进入关闭阶段。Bean 的销毁顺序遵循依赖逆序，也就是谁依赖别人，谁先销毁。

销毁过程中会执行 Bean 的 `Destroy` 回调，用来关闭连接、停止后台任务、刷写缓冲区等。

这个顺序保证了上层对象清理时，底层依赖仍然可用。例如服务关闭时先停止业务处理，再关闭数据库连接，就不会出现清理逻辑还在使用一个已经释放的底层资源。

## 排错先定位失败发生在哪个阶段

启动失败时，先定位阶段，再回头看对应 API，排查范围会小很多。

- 注册阶段常见问题是启动后仍调用 `gs.Provide()`，或者注册参数不合法。
- 解析阶段常见问题是条件不满足、Bean 冲突、Module 绑定配置失败。
- 注入阶段常见问题是找不到依赖、存在多个候选、循环依赖、初始化回调失败。
- 运行阶段常见问题是 Runner 或 Server 返回错误。
- 关闭阶段常见问题是 Server 停止阻塞或 Destroy 回调异常。

例如报错是“同类型同名称 Bean 冲突”，排查入口在解析阶段；如果是“找不到 `UserRepository`”，排查入口在注入阶段；如果是“端口监听失败”，通常已经进入 Server 运行阶段。

这种阶段划分比只看到“容器启动失败”更有价值，因为它能先把问题压缩到对应生命周期片段里。

## 运行流程把 IoC 设计边界串起来

注册、解析、注入、运行、关闭这五个阶段连起来，就是 Go-Spring 容器的排错地图。启动失败、依赖缺失、条件未命中、生命周期顺序不对，都可以先定位到对应阶段。

这些运行阶段也解释了 Go-Spring IoC 的几个设计边界，包括接口为什么显式导出，为什么不提供 `getBean()`，为什么按需创建，以及它和 Wire 面向的约束有什么不同。
