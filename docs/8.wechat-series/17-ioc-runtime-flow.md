# Go-Spring 实战第 17 课：容器运行流程：注册、解析、注入、运行和关闭五个阶段

前面几篇把 Bean、注入、注册和条件拆开讲。真正排查问题时，单个 API 只是局部线索，还需要看到这些信息在容器启动过程中如何流动。

把这些能力连起来看，Go-Spring 容器从启动到关闭可以分成五个阶段。

- 注册阶段。
- 解析阶段。
- 注入阶段。
- 运行阶段。
- 关闭阶段。

理解这些阶段，有助于判断错误发生在哪里，也能解释为什么 Go-Spring 强调在启动期完成依赖解析。

## 注册阶段先收集全局和当前定义

Go-Spring 的注册分为全局注册和容器注册。

全局注册发生在包的 `init()` 中。

```go
func init() {
	gs.Provide(NewService)
	gs.Module(condition, moduleFunc)
}
```

这些注册信息保存到全局注册表中。每次创建容器时，都会从全局注册表复制一份注册信息，然后生成独立容器实例。这样全局注册负责提供候选定义，而具体容器仍然保持相对独立。

容器注册主要来自 `app.Provide()`。

```go
gs.Configure(func(app gs.App) {
	app.Provide(NewTestDependency)
})
```

它只作用于当前应用或当前测试容器。这正是测试隔离的基础。

## 解析阶段完成合并、裁剪和冲突检测

Go-Spring 容器启动后，先合并所有 Bean 来源。

- `app.Provide()` 注册的 Bean。
- `gs.Provide()` 注册的 Bean。
- `gs.Module()` 和 `gs.Group()` 生成的 Bean。
- `Configuration` 模式导出的 Bean。

合并后，容器会接着执行条件裁剪。每个 Bean 的 Condition 会被计算，满足条件的 Bean 保留，不满足的 Bean 剔除。

最后进行冲突检测。Go-Spring 不允许同类型、同名称的 Bean 覆盖。因为覆盖会让依赖结果变得不确定，一旦发现冲突，容器会直接启动失败。

这一阶段决定了“本次启动到底有哪些 Bean 生效”。如果某个 Bean 没有被注入，可以先看它是否在解析阶段被条件裁剪掉了。

## 注入阶段从 root beans 展开依赖图

解析完成后，Go-Spring 容器再从 root beans 出发，按依赖关系递归创建 Bean 并注入依赖。

核心流程包括下面几步。

1. 为所有 Bean 建立类型和名称索引。
2. 从 root bean 开始递归创建依赖。
3. 创建 Bean 实例后立即执行依赖注入。
4. 注入过程中检测循环依赖。
5. 记录延迟注入字段，后续统一处理。
6. 注入完成后执行 `init` 回调。
7. 对 destroy 依赖关系进行拓扑排序。
8. 清理临时元数据。

注入阶段完成后，所有需要的 Bean 都已经创建、注入并初始化。

## 运行阶段不再动态查找 Bean

Go-Spring 用的是启动期模型。依赖关系只在启动阶段解析，应用进入运行阶段后，容器不再参与业务调用，也不提供运行时 `getBean()` 之类的动态获取能力。

这样做会带来几个直接结果。

- 依赖错误在启动阶段暴露。
- 运行时代码直接调用已注入对象。
- 容器不需要保留大量运行期元数据。
- 运行时没有反射式 Bean 查找开销。

也就是说，应用逻辑通常直接使用构造函数参数、字段注入和显式工厂，而不是在运行期向容器要对象。

## 关闭阶段按依赖逆序销毁

应用收到退出信号后，Go-Spring 容器进入关闭阶段。Bean 的销毁顺序遵循依赖逆序，即谁依赖别人，谁先销毁。

销毁过程中会执行 Bean 的 `Destroy` 回调，用来关闭连接、停止后台任务、刷写缓冲区等。

这样安排顺序，是为了保证销毁期间依赖仍然可用，避免先把底层资源关掉，上层对象还在执行清理逻辑。

## 调试时先把失败阶段找出来

启动失败时，可以先判断失败阶段。

- 注册阶段通常是启动后仍调用 `gs.Provide()` 或注册参数不合法。
- 解析阶段常见是条件不满足、Bean 冲突、Module 绑定配置失败。
- 注入阶段常见是找不到依赖、存在多个候选、循环依赖、初始化回调失败。
- 运行阶段常见是 Runner 或 Server 返回错误。
- 关闭阶段常见是 Server 停止阻塞或 Destroy 回调异常。

举个简单判断方式——如果报错是“同类型同名称 Bean 冲突”，先看解析阶段；如果是“找不到 `UserRepository`”，先看注入阶段；如果是“端口监听失败”，通常已经进入 Server 运行阶段。先定位阶段，再回头看对应 API，排查范围会小很多。

这种阶段划分其实比直接看“容器启动失败”更有排查价值。

## 运行流程本身就是排错地图

注册、解析、注入、运行、关闭这五个阶段连起来，就是 Go-Spring 容器的排错地图。启动失败、依赖缺失、条件未命中、生命周期顺序不对，都可以先定位到对应阶段。

这些运行阶段背后，其实都指向 Go-Spring IoC 的几个核心取舍，即接口为什么显式导出，为什么不提供 `getBean()`，为什么按需创建，以及它和 Wire 面向的约束有什么不同。
