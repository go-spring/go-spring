# 配置系统的设计模型

我们先从一个几乎所有项目都会经历的场景说起：配置一开始只是几个 key，放在文件里，业务代码按名字取出来。这个阶段，把配置理解成 `map[string]string` 没有什么问题。

但应用稍微复杂一点，这个理解就会开始吃力。配置可能来自 YAML、TOML、JSON、环境变量和命令行参数；值也不再只是字符串，而是结构体、数组、Map、默认值、变量引用和校验规则。如果这时还把它当成一组散落的 key-value，后面的绑定、合并和刷新都会变得很难统一。

所以，Go-Spring 的第一步，是先把所有输入收拢到同一个抽象里：`Properties`。后续的绑定、校验、合并和刷新，都是在这个模型之上展开的。

## Properties 抽象

先看 `Properties`。无论原始配置是什么格式，Go-Spring 最终都会把它转换成扁平化的 key-value 结构。

这个设计的关键不在“扁平化”本身，而在统一访问接口。也就是说，上层的配置绑定不需要关心配置最初来自 YAML 文件、命令行参数还是环境变量；只要输入能转换成同一套配置路径和值，就可以进入同一条处理链路。

这里也要先记住一个细节：Go-Spring 的 key 匹配是大小写敏感的。框架不会做自动大小写转换，也不会做松散匹配，例如驼峰转下划线、省略分隔符自动匹配等。配置 key 是什么字符串，就必须按什么字符串使用。

## Path 语法

有了统一的 `Properties`，接着还需要一套规则表达嵌套结构。Go-Spring 使用 path 语法定位配置项：

- 使用点号 `.` 分隔嵌套层级，例如 `a.b.c` 表示 `a -> b -> c`。
- 使用方括号 `[index]` 表示数组索引，例如 `a.b[0].c` 表示 `a.b` 数组第一个元素的 `c` 字段。

比如这样一段 YAML：

```yaml
app:
  port: 8080
  database:
    - host: localhost
      port: 5432
    - host: repli.ca
      port: 5433
```

转换成 `Properties` 后是：

```properties
app.port = 8080
app.database[0].host = localhost
app.database[0].port = 5432
app.database[1].host = repli.ca
app.database[1].port = 5433
```

这样一来，每一个配置值都有了唯一的位置。对象层级、数组元素和叶子值，都可以用同一套路径规则表达。后面做绑定、合并或校验时，就都能回到这个位置上。

## 为什么不是简单 Map

从表面看，`Properties` 仍然是 key-value；但如果只停在这个表面，就会漏掉它真正的作用。我们真正得到的是一棵可以用 path 访问的配置树。点分路径表达对象层级，数组下标表达有序集合，叶子节点表达最终值。

这棵配置树会贯穿后续所有能力：

- 结构体绑定会根据 path 找到对应字段。
- slice、array 和 map 绑定会依赖数组下标与子路径。
- 校验表达式会在字段绑定后对具体值执行。
- 多来源合并会根据 path 判断覆盖、合并或整体替换。
- 变量引用会从同一套路径空间中查找其他配置项。

所以配置模型不是文档里的背景知识，而是我们理解整个配置系统的基础。后面每一步看起来都是新能力，实际都在复用这套路径空间。

## 这套模型的意义

配置模型看似底层，实际决定了后面所有能力的形状。只有先把嵌套对象、数组和叶子值统一到同一套 path 空间里，绑定、校验、优先级合并和变量引用才有共同语言。

有了这套共同语言，下一步就可以看配置绑定：这棵配置树怎样落到 Go 结构体和函数参数上。
