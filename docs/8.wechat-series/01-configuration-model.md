# 配置系统的设计模型

## 本篇要解决的问题

配置系统最容易被误解成一个简单的 `map[string]string`。如果只是读取几个开关，这样理解似乎够用；但一旦进入真实工程，配置会来自 YAML、TOML、JSON、环境变量、命令行参数，也会包含结构体、数组、Map、默认值、变量引用和校验规则。

Go-Spring 的配置系统不是把这些能力零散拼起来，而是先把所有输入统一抽象成 `Properties`，再在这个统一模型之上完成绑定、校验、合并和刷新。

## Properties 抽象

Go-Spring 配置系统的核心思路是：无论原始配置是什么格式，最终都会转换成扁平化的 key-value 结构，也就是 `Properties`。

这个设计的价值在于统一访问接口。上层的配置绑定、校验、优先级合并和变量引用不需要关心配置最初来自 YAML 文件、命令行参数还是环境变量。只要能转换成相同的配置路径和值，就可以走同一套处理流程。

需要注意的是，Go-Spring 的 key 匹配是大小写敏感的。框架不提供自动大小写转换，也不支持松散匹配，例如驼峰转下划线、省略分隔符自动匹配等。配置 key 是什么字符串，就必须按什么字符串使用。

## Path 语法

Go-Spring 使用 path 语法定位配置项：

- 使用点号 `.` 分隔嵌套层级，例如 `a.b.c` 表示 `a -> b -> c`。
- 使用方括号 `[index]` 表示数组索引，例如 `a.b[0].c` 表示 `a.b` 数组第一个元素的 `c` 字段。

一个 YAML 配置：

```yaml
app:
  port: 8080
  database:
    - host: localhost
      port: 5432
    - host: repli.ca
      port: 5433
```

转换成 `Properties` 后是：

```properties
app.port = 8080
app.database[0].host = localhost
app.database[0].port = 5432
app.database[1].host = repli.ca
app.database[1].port = 5433
```

这意味着每一个配置值都有唯一路径。嵌套对象、数组元素和叶子值都能通过同一套路径规则表达。

## 为什么不是简单 Map

`Properties` 表面上是 key-value，但它背后表达的是一棵配置树。点分路径表达对象层级，数组下标表达有序集合，叶子节点表达最终值。

这个配置树模型会贯穿后续所有能力：

- 结构体绑定会根据 path 找到对应字段。
- slice、array 和 map 绑定会依赖数组下标与子路径。
- 校验表达式会在字段绑定后对具体值执行。
- 多来源合并会根据 path 判断覆盖、合并或整体替换。
- 变量引用会从同一套路径空间中查找其他配置项。

所以配置模型不是文档里的背景知识，而是理解 Go-Spring 配置系统的基础。
