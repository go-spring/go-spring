# Go-Spring 实战第 30 课：能力地图：配置、IoC、运行时、日志、HTTP、Starter 和测试如何协作

系列走到最后了，我们需要把分散的能力重新合起来看。

前面分别拆开了配置、IoC、运行时、日志、HTTP、Starter 和测试。现在我们回到整体：Go-Spring 不是一组彼此独立的工具函数，而是围绕应用生命周期组织的一套工程框架能力。

这一篇不新增 API，只梳理这些能力之间的关系。

可以先按时间线理解这套能力：启动前，配置系统把外部差异整理成统一输入；启动时，IoC 容器根据配置和条件组织对象；运行中，运行时、日志和 HTTP Server 承担服务调度与观测；退出时，容器和 Server 按生命周期释放资源；开发过程中，测试体系负责验证这些装配和行为。这样看，后面的能力就不是并排清单，而是同一条应用链路上的不同阶段。

## 配置系统负责输入差异

先看 Go-Spring 配置系统。它负责把运行环境中的差异输入应用，这是一整套能力的入口。

它提供了这些能力：

- `Properties` 和 path 语法。
- 结构体与函数参数绑定。
- 类型转换和复杂类型绑定。
- 表达式校验。
- 多格式、多来源加载。
- 优先级和合并语义。
- Profile 多环境配置。
- 导入、变量引用和动态刷新。

Go-Spring 配置系统解决的是“应用如何从外部获得参数，并在启动或运行期安全使用这些参数”。

## IoC 容器负责组织对象

接着是 Go-Spring IoC 容器。它负责把应用对象组织成依赖图。

它提供了这些能力：

- Bean 注册。
- 字段注入和构造函数注入。
- 按类型、按名称、集合和配置驱动注入。
- 构造函数参数绑定。
- 生命周期回调。
- 接口导出。
- 条件注册和 Profile 条件。
- 按需创建和 root bean。
- 启动期依赖解析与关闭期逆序销毁。

IoC 容器解决的是“应用对象如何被创建、组合、启用和释放”。

## 运行时负责调度生命周期

然后是 Go-Spring 应用运行时。它把配置系统和 IoC 容器串成完整生命周期。

启动时：

- 打印 Banner。
- 加载配置。
- 初始化日志。
- 启动 IoC 容器。
- 执行 Runner。
- 启动 Server。

退出时：

- 监听退出信号。
- 取消 root context。
- 停止 Server。
- 关闭容器。
- 执行 Destroy。
- flush 日志。

所以，运行时解决的是“应用如何从进程入口进入服务可用状态，并在退出时有序释放资源”。

## 日志系统负责结构化观测

应用跑起来以后，还需要 Go-Spring 日志系统负责让运行状态可观测。

它提供了这些能力：

- 基于标签的语义路由。
- 标准和自定义日志级别。
- 格式化日志与结构化字段。
- 同步、异步、控制台、文件、滚动文件 Logger。
- Appender、Layout、Encoder 输出管线。
- 从 `context.Context` 提取链路字段。
- 配置驱动和动态刷新。
- 标准库 `log` 与 Zap 适配。

Go-Spring 日志系统解决的是“应用如何在不污染业务逻辑的前提下稳定输出可检索、可治理的运行信息”。

## HTTP Server 负责 Web 入口接入

Go-Spring 内置 HTTP Server 基于标准库 `net/http`，默认随应用启动。

它提供了这些能力：

- 默认 HTTP Server。
- 监听地址、超时和启用开关配置。
- `gs.HttpServeMux` 路由入口。
- 标准库路由接入。
- Gin、gorilla/mux、chi 等第三方路由接入。
- Ready 信号和优雅关闭。

HTTP Server 解决的是“Web 服务如何纳入 Go-Spring 应用生命周期”。

## Starter 负责组件复用

如果能力需要跨项目复用，Starter 会把一组注册逻辑、配置绑定、条件启用和生命周期管理封装成可复用包。

它基于：

- `gs.Provide()` 注册单实例。
- `gs.Module()` 进行条件化动态注册。
- `gs.Group()` 生成多实例。
- 统一配置前缀和 Bean 命名约定。
- 资源型组件的 Destroy 回调。

Starter 解决的是“基础设施和通用能力如何跨项目复用，而不是每个应用重复手写初始化代码”。

## 测试体系负责工程验证

最后，测试体系把 Go 原生 `go test` 和 Go-Spring 增强能力结合起来。

它提供了这些能力：

- 纯单元测试分层。
- `gs.RunTest` 容器测试。
- 测试配置覆盖。
- 测试 Bean 替换。
- 测试隔离模型。
- `assert` 与 `require`。
- `gs-mock` 接口、函数和方法 Mock。

Go-Spring 测试体系解决的是“应用能力如何被低成本验证，并在需要时覆盖容器装配行为”。

## 边界决定适用场景

go-spring 的核心边界可以先记成这几条：

- 不是简单工具集合，而是围绕应用生命周期组织能力。
- 不追求运行时动态获取 Bean，而是启动期完成依赖解析。
- 不用自动包扫描隐藏注册关系，而是显式注册和导入。
- 不把配置、IoC、日志、HTTP、测试割裂开，而是让它们在同一套生命周期里协作。

这个边界决定了它更适合需要配置驱动、条件装配、生命周期管理、可观测和组件化复用的服务端应用。反过来说，如果只是几个函数的轻量封装，用完整框架反而没有必要。

## 最后把能力串成一条生命周期链路

前面把每个能力拆开，是为了看清边界；最后把它们重新合起来，是为了看清整体。

我们可以把整套能力理解成一条应用生命周期链路：配置负责输入差异，IoC 负责组织对象，运行时负责调度生命周期，日志负责观测，HTTP 负责服务入口，Starter 负责封装复用，测试负责验证。

这条链路也是判断是否需要 Go-Spring 的标准：如果项目只是少量函数调用，直接写 Go 代码更轻；如果项目需要多环境配置、条件装配、组件复用、统一启动关闭和结构化观测，那么这些能力放在同一个生命周期模型里，维护成本会更低。

所以，它们并不是并排摆放的功能清单，而是围绕应用生命周期协作的一套工程能力。这也是 go-spring 的能力地图。
