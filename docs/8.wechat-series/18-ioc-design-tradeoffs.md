# IoC 设计取舍

技术框架最容易被问的不是“能不能”，而是“为什么这样设计”。

Go-Spring 的 IoC 容器有一些非常明确的边界：接口需要显式导出、Bean 按需创建、不提供运行时 `getBean()`、核心只支持单例模式、依赖注入发生在启动期。

这些不是功能缺失，而是围绕 Go 工程实践做出的取舍。我们最后把这些设计边界集中看一遍。

## 接口为什么需要显式导出

Go 的接口是隐式实现的。一个结构体可能实现多个接口，也可能无意中实现某个接口。如果容器自动把结构体导出为所有可匹配接口，依赖关系会变得不确定。

Go-Spring 要求显式声明：

```go
gs.Provide(NewServiceImpl).Export(gs.As[Service]())
```

这让接口注入关系在代码中可见，也避免了“恰好实现接口”导致错误注入。

## 为什么按需创建

Go-Spring 不会无条件实例化所有注册 Bean。容器会从 root bean 出发，只创建依赖图上真正需要的 Bean。

root bean 主要来自：

- 实现了 `gs.Runner` 的 Bean。
- 实现了 `gs.Server` 的 Bean。
- 通过 `app.Root()` 显式标记的 Bean。

按需创建可以避免未使用组件产生资源开销，也让应用入口更加明确。我们注册一个组件，并不意味着它一定要在每次启动中被创建。

## 循环依赖支持边界

Go-Spring 对循环依赖是有限支持：

- 字段注入互相依赖：可以支持，因为对象可以先创建，再填充字段。
- 字段注入与构造函数注入混合循环：有限支持，需要谨慎。
- 构造函数互相依赖：不支持，因为双方都要求对方在构造前已经完整存在。

如果出现复杂循环依赖，通常应该重新审视模块边界。延迟注入能解决一部分问题，但不应该成为常规设计手段。

## 销毁顺序

容器遵循依赖逆序销毁。A 依赖 B，则初始化时先 B 后 A，销毁时先 A 后 B。

这保证上层对象清理时，底层依赖还没有被释放。例如服务关闭时先停止业务处理，再关闭数据库连接。

## 为什么不提供 getBean

Go-Spring 不支持运行时动态获取 Bean。原因主要有五点：

- 大多数服务端应用的依赖关系在启动期已经确定。
- 运行时向容器查找对象会让依赖关系变得隐式。
- 容器需要保留更多元数据，增加复杂度。
- 错误可能从启动期延迟到运行期。
- Go 更适合显式依赖声明和直接函数调用。

如果确实需要运行期创建对象，建议注册一个工厂 Bean，由工厂负责创建和管理实例。

## 是否支持原型模式

Go-Spring 核心只支持单例模式，不直接提供 prototype 作用域。

在 Go 服务端应用中，大多数组件天然是无状态或可复用的单例。如果需要多实例，更自然的做法是使用工厂：

```go
type MyServiceFactory struct {
	Config Config `autowire:""`
}

func (f *MyServiceFactory) NewMyService(name string) *MyService {
	return &MyService{
		name:   name,
		config: f.Config,
	}
}

func init() {
	gs.Provide(new(MyServiceFactory))
}
```

容器管理工厂，工厂创建具体实例。这样既保留显式依赖，也避免让容器参与每一次业务对象创建。

## 运行时反射开销

Go-Spring 使用反射完成启动阶段的依赖解析、构造函数分析和字段注入。应用启动完成后，组件之间就是普通 Go 对象调用。

因此运行时没有反射式查找开销。容器也会清理部分临时元数据，让运行期尽量轻量。

## 为什么不用自动包扫描

Go 没有真正意义上的运行时包扫描。只有被 `import` 的包才会编译进二进制，未导入包不会自动加载。

即使做“自动扫描”，最终也需要通过 `import _ "xxx"` 触发包的 `init()`。Go-Spring 选择直接在 `init()` 中显式注册 Bean，避免额外扫描机制带来的不确定性。

如果需要减少手写注册代码，可以用代码生成在构建期生成 `Provide` 调用，但核心模型仍然是显式注册。

## 和 Wire 的差异

| 特性 | Go-Spring | Wire |
|------|-----------|------|
| 处理时机 | 运行期启动阶段 | 编译期代码生成 |
| 条件注册 | 支持 | 不支持 |
| 动态配置 | 支持 | 不支持 |
| 生命周期管理 | 支持 | 不支持 |
| 启动速度 | 足够快 | 更快 |

Wire 更偏向编译期确定一切。Go-Spring 更偏向启动期根据配置和条件组织系统结构。

两者不是简单优劣关系，而是面向不同工程约束。需要极致静态约束时，Wire 很合适；需要配置驱动、条件装配、生命周期和 Starter 生态时，Go-Spring 的模型更完整。

## IoC 板块至此收住

Go-Spring 的 IoC 容器不是为了追求“更动态”，而是把依赖关系、环境条件、生命周期和测试替换集中到启动期处理。显式注册、按需创建、接口显式导出和不提供 `getBean()`，都是围绕这个方向做出的取舍。
