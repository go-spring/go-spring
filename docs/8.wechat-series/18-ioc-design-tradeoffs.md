# Go-Spring 实战第 18 课 —— IoC 设计取舍：启动期装配为什么不提供运行时 getBean()

上一课把容器运行流程拆成注册、解析、注入、运行和关闭以后，Go-Spring IoC 的几个边界会变得更清楚：接口需要显式导出，Bean 按需创建，不提供运行时 `getBean()`，核心只支持单例，依赖注入集中发生在启动期。

这些边界看起来像功能限制，但它们背后指向同一个约束：Go 服务端应用的大多数依赖关系应该在启动期定型。只要这个约束成立，容器就能把错误提前暴露，把对象关系留在代码和注册语句里，并让运行期尽量轻。

所以讨论设计取舍时，重点不是 Go-Spring 少了哪些动态能力，而是它把哪些问题放在启动期解决，又把哪些运行期行为交还给普通 Go 代码。

## 接口显式导出

Go 的接口是隐式实现的。一个结构体可能实现多个接口，也可能无意中满足某个接口。如果容器自动把结构体导出为所有可匹配接口，依赖关系就会变得不确定。

接口注入关系需要在注册处明确表达。

```go
type UserService interface {
	Get(id int) (*User, error)
}

type UserServiceImpl struct{}

func NewUserServiceImpl() *UserServiceImpl {
	return &UserServiceImpl{}
}

func init() {
	gs.Provide(NewUserServiceImpl).Export(gs.As[UserService]())
}
```

`.Export(gs.As[UserService]())` 牺牲了一点自动化，但换来的是接口暴露关系可见。依赖方按 `UserService` 声明时，注册处能清楚说明这是有意暴露的接口，而不是结构体碰巧实现了某个方法集合。

如果构造函数本身返回接口类型，就不需要额外导出。

```go
func NewUserService() UserService {
	return &UserServiceImpl{}
}
```

两种写法的共同点是：接口关系由代码明确给出，而不是让容器在运行时猜测。Go-Spring 选择显式导出，是为了让依赖图在注册语句里可读。

## 按需创建

Go-Spring 不会无条件实例化所有注册 Bean。注册只是把 Bean 放进候选集合，容器会从 root bean 出发，只创建依赖图上真正需要的对象。

下面的 `app.Root()` 指定的是依赖图入口。

```go
func main() {
	bootstrap := &Bootstrap{}

	gs.Configure(func(app gs.App) {
		app.Root(bootstrap)
	}).Run()
}
```

root bean 主要来自三类入口：实现了 `gs.Runner` 的 Bean、实现了 `gs.Server` 的 Bean，以及通过 `app.Root()` 显式标记的 Bean。

按需创建可以避免未使用组件产生资源开销，也让应用入口更加明确。它同时带来一个边界：注册一个组件，只表示它有资格参与装配，不表示它一定会在每次启动中被创建。如果某个对象必须驱动启动流程，就应该让它成为 root bean，或者让它被 root bean 依赖到。

因此，排查“某个 Bean 为什么没有执行初始化”时，除了看条件是否成立，还要看它是否从 root bean 可达。不可达的候选不会因为注册成功就自动实例化。

## 循环依赖

循环依赖能否处理，不只取决于容器能力，也取决于对象创建语义。构造函数注入要求依赖在构造前已经可用，字段注入则允许对象先创建、再填充字段。

构造函数互相依赖无法被自然拆开。

```go
func NewA(b *B) *A {
	return &A{B: b}
}

func NewB(a *A) *B {
	return &B{A: a}
}
```

`NewA` 需要一个完整的 `*B`，`NewB` 又需要一个完整的 `*A`。双方都要求对方先存在，容器没有一个合法的创建起点。

字段注入的循环在某些场景下可以通过先创建对象、再填充字段来处理，但这不应该成为设计复杂依赖图的常规手段。复杂循环通常说明模块边界已经纠缠在一起，更稳的做法是抽出更小的接口、事件通道或协调对象，让依赖方向重新变清楚。

## 单例和工厂

Go-Spring 核心只支持单例模式，不直接提供 prototype 作用域。这是和服务端应用形态相关的选择：大多数基础设施组件、Repository、Client、Service 都是无状态或可复用对象，适合在启动期创建一次。

如果确实需要按请求、按租户或按业务参数创建新对象，更自然的做法是注册一个工厂 Bean。容器管理工厂，运行期实例由工厂创建。

```go
type MyServiceFactory struct {
	Config Config `autowire:""`
}

func (f *MyServiceFactory) NewMyService(name string) *MyService {
	return &MyService{
		name:   name,
		config: f.Config,
	}
}

func init() {
	gs.Provide(new(MyServiceFactory))
}
```

这样依赖仍然是显式的。`MyServiceFactory` 的配置和依赖由 Go-Spring 注入，具体 `MyService` 实例则由业务代码在运行期按参数创建。容器不需要参与每一次业务对象创建，也不会把运行期参数塞回启动期装配模型。

## 运行时 getBean

Go-Spring 不提供运行时动态获取 Bean 的 `getBean()`。这个取舍来自启动期装配模型。

如果业务代码可以在任意请求路径里向容器要对象，依赖关系就会从结构体字段、构造函数和注册语句里消失，变成运行期字符串或类型查找。错误也可能从启动阶段推迟到某一次请求才暴露。

相反，Go-Spring 要求依赖在启动期定型。对象进入运行阶段以后，就是普通 Go 对象之间的直接调用。容器使用反射完成启动阶段的依赖解析、构造函数分析和字段注入；启动完成后，业务调用不再经过反射式 Bean 查找。

如果确实需要运行期选择不同实现，通常有两种更清楚的方式：在启动期按条件注入一个策略对象，或者注入一个显式工厂，由工厂根据业务参数选择或创建实例。这样运行期变化仍然留在业务模型里，而不是藏进容器查找动作里。

## 显式注册

Go 没有真正意义上的运行时包扫描。只有被 `import` 的包才会编译进二进制，未导入包不会自动加载。

下面这个事实决定了 Go-Spring 不把“自动扫描”作为核心模型：即使引入扫描式写法，最后也需要通过 `import _ "xxx"` 触发包的 `init()`，否则对应注册代码根本不会进入程序。

因此，Go-Spring 选择在 `init()` 中显式注册 Bean。

```go
func init() {
	gs.Provide(NewUserService)
}
```

这让对象图来源和 Go 的编译模型保持一致。需要减少手写注册代码时，可以用代码生成在构建期生成 `Provide` 调用，但核心语义仍然是显式注册，而不是运行时发现一个未导入的包。

## 与 Wire 的边界

Go-Spring 和 Wire 面向的是不同约束。两者不是简单替代关系。

| 特性 | Go-Spring | Wire |
|------|-----------|------|
| 处理时机 | 启动阶段 | 编译期代码生成 |
| 条件注册 | 支持 | 不支持 |
| 配置驱动 | 支持 | 不支持 |
| 生命周期管理 | 支持 | 不支持 |
| 运行期成本 | 启动期反射，运行期普通调用 | 生成代码，运行期更静态 |

如果系统需要极致静态约束，Wire 很合适。如果系统需要配置驱动、条件装配、生命周期和 Starter 生态，Go-Spring 的启动期模型会覆盖更多运行语义。选择哪一个，取决于应用更需要编译期生成的静态对象图，还是启动期根据配置和环境裁剪对象图。

## 启动期装配

Go-Spring 的 IoC 容器不是为了追求更动态，而是把依赖关系、环境条件、生命周期和测试替换集中到启动期处理。接口显式导出、按需创建、核心单例、不提供 `getBean()`，都服务于这条边界。

边界明确以后，运行期代码就可以保持普通 Go 对象调用。Go-Spring 在启动期完成装配和校验，把运行期留给业务模型本身，这是理解这些设计取舍的主线。
