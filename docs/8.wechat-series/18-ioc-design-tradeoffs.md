# Go-Spring 实战第 18 课：IoC 设计取舍：启动期装配为什么不提供运行时 getBean

排查过容器运行流程以后，Go-Spring IoC 的一些边界会变得更明显。它要求接口显式导出，按需创建 Bean，不提供运行时 `getBean()`，核心只支持单例模式，依赖注入集中发生在启动期。

这些边界看起来像功能限制，但它们背后指向同一个约束——Go 服务端应用的大多数依赖关系应该在启动期定型。只要这个约束成立，容器就能把错误提前暴露，把对象关系留在代码和注册语句里，并让运行期尽量轻。

所以到了设计取舍这里，重点不是“Go-Spring 少了哪些动态能力”，而是这些取舍分别牺牲了什么，又换来了什么工程性质。

## 接口显式导出让注入关系留在注册处

Go 的接口是隐式实现的。一个结构体可能实现多个接口，也可能无意中满足某个接口。如果容器自动把结构体导出为所有可匹配接口，依赖关系就会变得不确定。

Go-Spring 要求在注册处显式声明接口导出。

```go
gs.Provide(NewServiceImpl).Export(gs.As[Service]())
```

这样做牺牲了一点自动化，但换来的是接口注入关系可见。依赖方按接口声明时，注册处能清楚看到这个接口暴露是有意为之，而不是结构体“碰巧”实现了某个方法集合。

## 按需创建要求应用入口明确

Go-Spring 不会无条件实例化所有注册 Bean。容器会从 root bean 出发，只创建依赖图上真正需要的 Bean。

root bean 主要来自三类入口。

- 实现了 `gs.Runner` 的 Bean。
- 实现了 `gs.Server` 的 Bean。
- 通过 `app.Root()` 显式标记的 Bean。

按需创建可以避免未使用组件产生资源开销，也让应用入口更加明确。它同时带来一个边界——注册一个组件，只是让它进入候选集合，并不意味着它一定会在每次启动中被创建。

## 循环依赖只支持能分两步创建的场景

Go-Spring 对循环依赖是有限支持，因为不同注入方式对创建顺序的要求不同。

- 字段注入互相依赖可以支持，因为对象可以先创建，再填充字段。
- 字段注入与构造函数注入混合循环是有限支持，需要谨慎。
- 构造函数互相依赖不支持，因为双方都要求对方在构造前已经完整存在。

这个边界来自对象创建语义，不是单纯的容器能力开关。延迟注入能解决一部分过渡问题，但复杂循环依赖通常说明模块边界已经纠缠在一起。

## 销毁顺序必须和依赖顺序相反

容器遵循依赖逆序销毁。A 依赖 B，则初始化时先 B 后 A，销毁时先 A 后 B。

这个顺序保证上层对象清理时，底层依赖仍然存在。例如服务关闭时先停止业务处理，再关闭数据库连接，就不会让清理逻辑面对已经释放的资源。

销毁顺序不是额外规则，而是依赖图在退出阶段的反向执行。

## 不提供 getBean 是为了让依赖在启动期定型

Go-Spring 不支持运行时动态获取 Bean。这个取舍来自几个约束。

- 大多数服务端应用的依赖关系在启动期已经确定。
- 运行时向容器查找对象会让依赖关系变得隐式。
- 容器需要保留更多元数据，增加复杂度。
- 错误可能从启动期延迟到运行期。
- Go 更适合显式依赖声明和直接函数调用。

如果确实需要运行期创建对象，可以注册一个工厂 Bean，由工厂负责创建和管理实例。这样依赖仍然是显式的，只是具体实例由工厂在运行期产生。

## 原型对象交给工厂 Bean 更符合运行期创建

Go-Spring 核心只支持单例模式，不直接提供 prototype 作用域。

在 Go 服务端应用中，大多数组件天然是无状态或可复用的单例。如果需要按请求、按租户或按业务参数创建新对象，更自然的做法是使用工厂 Bean。

```go
type MyServiceFactory struct {
	Config Config `autowire:""`
}

func (f *MyServiceFactory) NewMyService(name string) *MyService {
	return &MyService{
		name:   name,
		config: f.Config,
	}
}

func init() {
	gs.Provide(new(MyServiceFactory))
}
```

容器管理工厂，工厂创建具体实例。这样既保留了工厂对配置和依赖的显式声明，也避免让容器参与每一次业务对象创建。

## 反射只承担启动期解析成本

Go-Spring 使用反射完成启动阶段的依赖解析、构造函数分析和字段注入。应用启动完成后，组件之间就是普通 Go 对象调用。

因此，运行时没有反射式 Bean 查找开销。容器也会清理部分临时元数据，让运行期尽量保持轻量。

这个取舍和不提供 `getBean()` 是一致的。反射用于构建对象图，而不是进入每一次业务调用。

## 显式注册比运行时扫描更符合 Go 的编译模型

Go 没有真正意义上的运行时包扫描。只有被 `import` 的包才会编译进二进制，未导入包不会自动加载。

即使做“自动扫描”，最终也需要通过 `import _ "xxx"` 触发包的 `init()`。所以 Go-Spring 选择直接在 `init()` 中显式注册 Bean，避免额外扫描机制带来的不确定性。

如果需要减少手写注册代码，可以用代码生成在构建期生成 `Provide` 调用，但核心模型仍然是显式注册。

## Go-Spring 和 Wire 面向不同约束

Wire 更偏向编译期确定一切，Go-Spring 更偏向启动期根据配置和条件组织系统结构。两者处理的是不同工程约束，不是同一个维度上的简单替代。

| 特性 | Go-Spring | Wire |
|------|-----------|------|
| 处理时机 | 运行期启动阶段 | 编译期代码生成 |
| 条件注册 | 支持 | 不支持 |
| 动态配置 | 支持 | 不支持 |
| 生命周期管理 | 支持 | 不支持 |
| 启动速度 | 足够快 | 更快 |

如果系统需要极致静态约束，Wire 很合适。如果系统需要配置驱动、条件装配、生命周期和 Starter 生态，Go-Spring 的启动期模型会覆盖更多运行语义。

## 启动期装配边界决定 IoC 取舍

Go-Spring 的 IoC 容器不是为了追求更动态，而是把依赖关系、环境条件、生命周期和测试替换集中到启动期处理。显式注册、按需创建、接口显式导出和不提供 `getBean()`，都服务于这条边界。

IoC 解决的是对象装配。再往外一层，`gs.Run()` 会把配置加载、日志初始化、容器启动、Runner 执行和 Server 启动串成完整应用运行时。
