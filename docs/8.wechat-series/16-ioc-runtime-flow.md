# Go-Spring 实战第 16 课 —— 容器运行流程：注册、解析、注入、运行、关闭

前面几篇文章我们把 go-spring 依赖注入怎么使用讲明白了，首先是怎么使用依赖注入（字段注入和构造函数注入），然后是依赖注入的目标（单bean注入和集合注入），然后是 bean 注册的方式（结构体指针和构造函数以及函数指针），然后是 bean 元数据（name、生命周期、条件等），然后是 bean 注册方法（gs.provide、gs.module、gs.group,app.provide,configuration），然后是条件注册，这样我们就把依赖注入的方方面面都讲清楚了。

本篇咱们来详细聊聊 go-spring ioc 容器的运行原理。

## 运行原理

go-spring ioc 容器的运行分为注册阶段、解析阶段、注入阶段、运行阶段、关闭阶段五个阶段。其中

- 注册阶段，是指把所有候选的 Bean 定义注册到容器里，但是还没有开始创建对象。
- 解析阶段，是指根据注册的 Bean 定义，解析出 Bean 的依赖关系。
- 注入阶段，是指根据解析关系，注入依赖到 Bean 中，包括执行生命周期、初始化回调。
- 运行阶段，是指应用程序的运行，这个阶段 ioc 容器其实不参与，因为是启动期注入。
- 关闭阶段，是指在容器关闭时，调用关闭回调、释放资源等。

下面咱们详细看每个阶段的实现。

## 注册阶段

前面几篇其实都在讲注册阶段的事情。尤其相关的是注册方法篇，go-spring 提供了 gs.provide、gs.module、gs.group、app.provide、configuration 注册方法。每一种都有独特的场景。

咱们这里简述一下：

- gs.provide 用于将单个 bean 注册到全局注册表
- gs.module 用于注册多个 bean 到全局注册表
- gs.group 用于注册多个同类型的 bean 到全局注册表
- app.provide 用于将单个 bean 注册到当前 ioc 容器
- configuration 使用类似 java 配置类的方式注册子 bean

## 解析阶段

解析阶段要解决的问题是“注册的bean中哪些是有效的，哪些是无效的”，有效的被保留，无效的被删除。

这个阶段主要做的事情如下：

- 合并全局注册表里面的 bean 和当前容器里面的 bean
- 执行全局注册表里面的 module 回调，注册 bean 到当前容器
- 对所有的 bean 进行条件裁剪，根据条件判断是否保留
- 对所有的 bean 进行冲突检测，根据名称和类型判断是否冲突

最终，我们得到完整的一定存在的 bean 的列表，然后在下一个阶段开始创建对象。

## 注入阶段

注入阶段对每个 bean 做以下事情：

- 根据 bean 的定义，创建 bean 实例，
- 对 bean 的字段进行注入和配置绑定
- 执行 bean 的 init 回调

在执行过程中，还要收集每个 bean 注册的销毁函数，记录下来，以便关闭阶段使用。

**Root Bean 与 按需创建**

root bean 是一种特殊的 bean，它们是依赖注入的源头。

在 go 里面，很多 bean 可能引入了但是不会被使用，即使他们通过了解析阶段。但如果没有其他的 bean 来依赖它，那么这个 bean 就没有创建的必要。这里需要提一下 go-spring app 的运行模型。

go-spring app 抽象出 runner 和 server 两种运行组件，一种是启动阶段执行一些初始化的任务，一种是启动服务器，监听端口、接受网络请求等。理论上其他类型的 bean 都应当从这两种 bean 出发。runner 可能依赖 db bean，server 可能依赖 http server bean，可能依赖 grpc server bean。

这也就是按需创建 bean。对于 go 来说，很可能不小心就引入一些不需要的 bean，但是显式排除的成本很高。按需创建的方案可以明显减少这种排除的工作。

## 运行阶段

go-spring 在运行阶段其实什么事情都不做，因为所有的注入都在启动期完成了。go-spring 不支持运行期注入。

这样模型简单，也不会有跟注入相关的 bug 流入运行时。也减少排查的成本，所有的错误都在启动期暴露。容器不需要保留运行时查找 bean 和注入 bean 的能力，不需要面对并发等问题。

运行期，server 会持续运行，但是那和容器已经没关系了，完全是 bean 的自身行为。

## 关闭阶段

应用退出时，go-spring 容器会进入关闭阶段。

在这个阶段，ioc 容器会根据依赖关系，反向执行 bean 的销毁函数。即先销毁依赖的 bean，再销毁依赖它的 bean。

这个逆序很重要。上层对象清理时，底层依赖仍然可用；比如服务先停止业务处理，再关闭数据库连接，就不会让清理逻辑面对已经释放的资源。
