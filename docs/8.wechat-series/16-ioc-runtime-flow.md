# Go-Spring 实战第 16 课 —— 容器运行流程：注册、解析、注入、运行、关闭

前面几篇文章我们已经把 Go-Spring 依赖注入的使用方式全部讲完了：从字段注入和构造函数注入，到单 Bean 注入和集合注入，从结构体指针到构造函数和函数指针，从Bean 注册到Bean 元数据和条件注册。

本篇咱们讲一下 Go-Spring 容器的运行流程。这样我们就可以理解 Go-Spring 容器的内部工作原理，以及它在运行期的行为。同时在遇到问题时，也知道按照什么样的流程来排查。

## 运行原理

Go-Spring 的 IoC 容器可以拆成五个阶段：

- 注册阶段：收集候选 BeanDefinition。
- 解析阶段：根据配置、条件、配置类和重复检查，决定本次容器刷新保留哪些 Bean。
- 注入阶段：从 Root Bean 出发，创建对象、注入依赖、绑定配置、执行 Init，并收集 Destroy。
- 运行阶段：需要的对象已经创建完成，业务代码使用普通 Go 引用，容器不再做运行期注入。
- 关闭阶段：容器执行已收集的 Destroy 回调，释放 Bean 自己持有的资源。

Go-Spring 只有启动期装配。应用运行起来以后，业务代码拿到的是普通对象和接口引用。运行时不能注册或者获取 Bean。

## 注册阶段

注册阶段就是把各种注册入口提交的 Bean 定义收集起来。

Go-Spring 提供了以下几种注册方式：

- `gs.Provide()`：将 Bean 注册到全局注册表。
- `gs.Module()`：注册一组 Bean，并且可以通过配置动态决定是否注册这组 Bean。
- `gs.Group()`：`gs.Module()` 的特化版本，专门注册一组类型相同的 Bean。
- `app.Provide()`：将 Bean 注册到当前 App 的容器。
- `.Configuration()`：把一个 Bean 标记成配置类，后续扫描它的方法生成子 Bean。

此时容器只关心“有哪些候选 Bean”，还不会判断它们最终是否生效，也不会创建对象。为什么呢？

因为很多 Bean 的注册发生在 go `init()` 阶段，这时候配置还没有加载，当前应用到底会不会使用这个 Bean 也还不知道。如果注册时就创建对象，导入一个包就可能打开连接、读取配置、启动后台逻辑，副作用会很难控制了。所以我们只是先手机对象的定义，而不是对象本身。

这时候我们收集的信息也就是 bean 的元数据有以下这些内容：

- Bean 的类型和名称。
- 构造函数或已有实例。
- 导出的接口类型。
- 条件、初始化函数、销毁函数。
- 显式依赖和配置类信息。

## 解析阶段

在解析阶段开始之前，应用的配置已经加载完成。所以解析阶段可以基于已经加载完成的配置做判断。条件判断、模块激活和后续配置绑定都会基于它。

解析阶段仍然不创建对象。解析阶段主要做五件事。

第一，合并 BeanDefinition。

`gs.Provide()` 将 bean 注册到全局注册表，`app.Provide()` 将 bean 注册到当前容器。解析阶段开始后，会首先把两部分合并成一组候选定义。

第二，执行满足条件的 Module。

`gs.Module()` 本质上是一个在解析阶段执行的回调。用户在回调里面根据传入的配置信息动态注册需要的 bean。module 在执行之前先进行条件判断，只有条件成立才会调用模块函数。这种机制很适合 starter，控制一组完成某项功能的 bean。通过配置控制 starter 的行为是最简单的方式（AI:可以考虑修改这句话）。

第三，扫描 Configuration。

接下来容器会找到带有 configuration 标记的 Bean，然后扫描它的方法，根据规则注册子 Bean。这些子 bean 依赖于配置类 Bean 的存在，所以子 bean 会自动添加依赖父 bean 存在性的条件，也就是 `OnBeanID(配置类 Bean)`。

第四，执行 Bean 条件裁剪。

ioc 容器会检查每个 bean 的条件，只有条件成立才会保留下来。如果条件不成立，则这个 bean 会被标记为删除。这里的条件可以是根据配置，也可以是根据 bean 的存在性。

第五，做重复检查。

在 go-spring 里面，类型和名称一起构成了 bean 的唯一标识。每个 bean 都应该在容器中是唯一的。因此条件判断过后，保留下来的 bean 也应当都是唯一的。我们在这里检查一下，杜绝违反这个规则的情况出现。

## 注入阶段

注入阶段才会真正的创建对象。但是 go-spring 不会创建所有已经解析过的 bean，而是从 Root Bean 出发，沿着依赖关系递归创建。

为什么要这样呢？因为在 go 里面存在空白导入，这种空白导入量太大的时候，很难说通过配置来一个一个的控制它们是否启用。那么有没有一种自动的方式能决定 bean 是否值得被创建呢？因此这里我们采用类似 gc 根的概念，重要的一定从根出发，否则就是不重要的。也就不值得被创建。

在 go-spring 里面，runner 和 server 是默认的 Root Bean。其他类型都不是。理论上其他的 bean 都应该从这两种 bean 出发。当然，也存在特殊情况，那是在和旧框架集成的时候，这里不展开了。其他篇会详细讲解。

每个 bean 在处理时都经历以下步骤：

1. 循环检测（dependson 和 构造函数）
2. 创建对象（执行构造函数，包括参数解析）
   1. 构造函数返回失败会阻止启动
3. 注入依赖和配置
   1. 如果遇到 `gs.Dync[T]` 标签，登记到动态配置对象列表里。
   2. 发现 lazy 字段，先记录下来。而当前 bean 也不要使用 lazy 字段，因为没有值。
4. 执行 Init 回调
5. 标记为 wired

在所有 bean 都注入完成之后，还会做两件事：

1. 执行所有的 lazy 字段的注入。
2. 对所有的 destroy 回调进行排序。

## 运行阶段

go-spring 不提供运行时注册和获取 bean 的操作。一方面我们通常不太需要运行时注册 bean 的功能，一方面如果我们需要在运行时获取 bean，那么我们可以在启动期就保留这个 bean 的引用。并且，只关心启动期注入也避免并发的考虑，实现起来也很简单。

每个请求到来之后，需要执行依赖注入的情况应该是很少的。而且不是 ioc 容器的功能，而是 http server 框架的能力。（这里可以补充这个观点）

我们说 go-spring 容器没有运行时依赖注入。但是它有动态刷新配置的能力。声明为 `gs.Dync[T]` 的字段，可以在运行期根据配置进行刷新，配置更新后出发刷新操作。

使用启动期模型，可以大大减少人们对运行时的性能焦虑。go-spring 在运行时完全不依赖反射。

## 关闭阶段

ioc 容器关闭时程序退出的一个关键环节。ioc 容器会在这个阶段执行所有的 Destroy 回调。

destroy 回调的顺序在注入阶段末尾确定。通常来说，被依赖的 Bean 会先执行，依赖它的 Bean 后执行。这个顺序意味着：普通 Bean 的 Destroy 更适合释放自己持有的资源，例如关闭连接池、关闭文件、释放客户端。

另外，destroy 回调执行失败了也不会中断后续 Destroy，只是记录日志。
