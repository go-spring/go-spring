# Go-Spring 实战第 16 课 —— 容器运行流程：注册、解析、注入、运行、关闭

前面几篇文章，我们已经把 Go-Spring 依赖注入的使用方式基本讲完了：从字段注入和构造函数注入，到单 Bean 注入和集合注入；从结构体指针到构造函数和函数指针；从 Bean 注册到 Bean 元数据和条件注册。

本篇我们继续往里走，梳理一下 Go-Spring 容器的运行流程。理解这条流程之后，就能把前面讲过的使用方式和容器内部动作对应起来，也能在遇到问题时按阶段排查。

## 运行原理

Go-Spring 的 IoC 容器可以拆成五个阶段：

- 注册阶段：收集候选 BeanDefinition。
- 解析阶段：根据配置、条件、配置类和重复检查，决定本次容器刷新保留哪些 Bean。
- 注入阶段：从 Root Bean 出发，创建对象、注入依赖、绑定配置、执行 Init，并收集 Destroy。
- 运行阶段：需要的对象已经创建完成，业务代码使用普通 Go 引用，容器基本不再参与依赖注入。
- 关闭阶段：容器执行已收集的 Destroy 回调，释放 Bean 自己持有的资源。

Go-Spring 只在启动期完成装配。应用运行起来以后，业务代码拿到的是普通对象和接口引用；容器不支持运行时注册或获取 Bean。

## 注册阶段

注册阶段的职责，是把各种注册入口提交的 Bean 定义收集起来。

Go-Spring 提供了以下几种注册方式：

- `gs.Provide()`：将 Bean 注册到全局注册表。
- `gs.Module()`：注册一组 Bean，并且可以通过配置动态决定是否注册这组 Bean。
- `gs.Group()`：`gs.Module()` 的特化版本，专门注册一组类型相同的 Bean。
- `app.Provide()`：将 Bean 注册到当前 App 的容器。
- `.Configuration()`：把一个 Bean 标记成配置类，后续扫描它的方法生成子 Bean。

此时容器只关心“有哪些候选 Bean”，还不会判断它们最终是否生效，也不会创建对象。这样设计的原因是：

很多 Bean 的注册发生在 Go 的 `init()` 阶段。此时配置还没有加载，当前应用到底会不会使用这个 Bean 也还不知道。如果注册时就创建对象，导入一个包就可能打开连接、读取配置、启动后台逻辑，副作用会很难控制。所以这里先收集对象的定义，而不是对象本身。

这一阶段收集到的信息，也就是 Bean 的元数据，主要包括：

- Bean 的类型和名称。
- 构造函数或已有实例。
- 导出的接口类型。
- 条件、初始化函数、销毁函数。
- 显式依赖和配置类信息。

## 解析阶段

进入解析阶段时，配置视图已经准备好。因此容器可以基于配置做判断，条件判断、模块激活和后续配置绑定都会以它为依据。

解析阶段仍然不创建对象，主要做五件事。

第一，合并 BeanDefinition。

`gs.Provide()` 将 Bean 注册到全局注册表，`app.Provide()` 将 Bean 注册到当前容器。解析阶段开始后，会先把这两部分合并成一组候选定义。

第二，执行满足条件的 Module。

`gs.Module()` 本质上是一个在解析阶段执行的回调。用户可以在回调里面根据传入的配置信息，动态注册当前应用需要的 Bean。Module 在执行之前会先进行条件判断，只有条件成立才会调用模块函数。这种机制很适合 starter：一个 starter 通常要注册一组完成某项功能的 Bean，而配置正好可以决定这组 Bean 是否进入当前应用。

第三，扫描 Configuration。

接下来，容器会找到带有 Configuration 标记的 Bean，然后扫描它的方法，并根据规则注册子 Bean。这些子 Bean 依赖于配置类 Bean 的存在，因此会自动添加一个依赖父 Bean 存在性的条件，也就是 `OnBeanID(配置类 Bean)`。

第四，执行 Bean 条件裁剪。

IoC 容器会检查每个 Bean 的条件，只有条件成立才会保留下来。如果条件不成立，这个 Bean 就会被标记为删除。这里的条件可以来自配置，也可以来自 Bean 的存在性。

第五，做重复检查。

在 Go-Spring 里面，类型和名称一起构成 Bean 的唯一标识。每个 Bean 都应该在容器中保持唯一。因此条件判断之后，保留下来的 Bean 也应当都是唯一的。这里会再做一次检查，避免出现违反这个规则的情况。

## 注入阶段

到了注入阶段，容器才会真正创建对象。不过，Go-Spring 不会创建所有已经解析过的 Bean，而是从 Root Bean 出发，沿着依赖关系递归创建。

为什么要这样做呢？因为 Go 里面存在空白导入。当空白导入数量较多时，很难完全依靠配置逐个控制它们是否启用。因此这里采用了类似 GC Roots 的概念：重要的 Bean 一定能从根出发访问到；如果访问不到，就说明它对当前应用并不重要，也就不值得被创建。

在 Go-Spring 的默认运行模型里，runner 和 server 可以理解为默认的 Root Bean。更准确地说，App 持有 runner 数组、server 数组，以及用户显式指定的其他 Root Bean；容器会从这些入口开始创建对象。理论上，其他 Bean 都应该从 runner 或 server 出发被访问到。当然，也存在一些特殊情况，比如和旧框架集成的时候，这里先不展开，后续再详细讲解。

每个 Bean 在处理时都会经历以下步骤：

1. 循环检测（dependson 和构造函数）
2. 创建对象（执行构造函数，包括参数解析）
   1. 构造函数返回失败会阻止启动
3. 注入依赖和配置
   1. 如果遇到 `gs.Dync[T]` 标签，登记到动态配置对象列表里。
   2. 发现 lazy 字段，先记录下来；当前 Bean 的 Init 阶段不要依赖这个字段，因为它还没有完成注入。
4. 执行 Init 回调
5. 标记为 wired

在所有 Bean 都注入完成之后，还会做两件事：

1. 执行所有 lazy 字段的注入。
2. 对所有的 Destroy 回调进行排序。

## 运行阶段

Go-Spring 不提供运行时注册和获取 Bean 的操作。一方面，业务通常不太需要运行时注册 Bean；另一方面，如果确实需要在运行时使用某个 Bean，也可以在启动期就保留它的引用。容器只关心启动期注入，也可以避免在运行期处理并发装配问题。

每个请求到来之后，如果还需要做某种请求级对象绑定，那通常属于 Web 框架或业务框架的能力，不是 IoC 容器的核心职责。

前面说 Go-Spring 容器没有运行时依赖注入，但这并不代表它完全不参与运行期能力。声明为 `gs.Dync[T]` 的字段，可以在运行期根据配置进行刷新，并在配置更新后触发刷新操作。

使用启动期模型，可以大大减少对运行时性能的担忧。正常业务调用路径使用的是已经注入好的对象引用，不需要再经过容器反射查找。

## 关闭阶段

IoC 容器关闭是程序退出的一个关键环节，容器会在这个阶段执行所有 Destroy 回调。

Destroy 回调会在注入阶段收集并排序。只有真正被创建出来的 Bean，才需要执行 Destroy；一个 Bean 即使通过了解析阶段，如果没有从 Root Bean 出发被创建，也不会执行对应的 Destroy 回调。

通常来说，被依赖的 Bean 会先执行 Destroy，依赖它的 Bean 后执行 Destroy。这个顺序意味着：普通 Bean 的 Destroy 更适合释放自己持有的资源，例如关闭连接池、关闭文件、释放客户端。

另外，Destroy 回调执行失败也不会中断后续 Destroy，只会记录日志。
