# Go-Spring 实战第 16 课 —— 容器运行流程：注册、解析、注入、运行、关闭

前面几篇文章我们已经把 Go-Spring 依赖注入的使用方式全部讲完了：从字段注入和构造函数注入，到单 Bean 注入和集合注入，从结构体指针到构造函数和函数指针，从Bean 注册到Bean 元数据和条件注册。

本篇咱们讲一下 Go-Spring 容器的运行流程。这样我们就可以理解 Go-Spring 容器的内部工作原理，以及它在运行期的行为。同时在遇到问题时，也知道按照什么样的流程来排查。

## 运行原理

Go-Spring 的 IoC 容器可以拆成五个阶段：

- 注册阶段：收集候选 BeanDefinition。
- 解析阶段：根据配置、条件、配置类和重复检查，决定本次容器刷新保留哪些 Bean。
- 注入阶段：从 Root Bean 出发，创建对象、注入依赖、绑定配置、执行 Init，并收集 Destroy。
- 运行阶段：对象图已经创建完成，业务代码使用普通 Go 引用，容器不再做运行期注入。
- 关闭阶段：容器执行已收集的 Destroy 回调，释放 Bean 自己持有的资源。

可以理解成下面这条链路：

```text
  -> 解析候选定义
     -> 合并候选 Bean
     -> 执行 Module
     -> 扫描 Configuration
     -> 执行 Condition
     -> 检查重复 Bean
  -> 创建对象图
     -> 从 Root Bean 递归创建依赖
     -> 注入字段和绑定配置
     -> 执行 Init
     -> 收集 Destroy
```

这里有一个关键点：Go-Spring 不是运行期服务定位器，而是启动期装配器。应用运行起来以后，业务代码拿到的是普通对象和接口引用，容器不会在每次业务调用时参与依赖查找。

## 注册阶段

注册阶段处理的是 BeanDefinition，也就是 Bean 的候选定义。

常见注册入口有几类：

- `gs.Provide()`：注册包级全局 Bean。
- `gs.Module()`：注册一组条件化注册逻辑。
- `gs.Group()`：根据配置 Map 批量注册命名 Bean。
- `app.Provide()`：注册当前 App 专属 Bean。
- `.Configuration()`：把一个 Bean 标记成配置类，后续扫描它的方法生成子 Bean。

注册阶段只记录元数据，不创建对象。

这背后有一个很现实的考虑：很多 Bean 注册发生在 `init()` 里，此时最终配置还没有加载，当前应用到底会不会使用这个 Bean 也还不知道。如果注册时就创建对象，导入一个包就可能打开连接、读取配置、启动后台逻辑，副作用会很难控制。

也就是说，下面这些信息都会被记录下来：

- Bean 的类型和名称。
- 构造函数或已有实例。
- 导出的接口类型。
- 条件、初始化函数、销毁函数。
- 显式依赖和配置类信息。

但构造函数不会在这里调用，字段不会在这里注入，`Init` 也不会在这里执行。

`gs.Module()` 也是一样。注册时只是把模块函数登记下来，真正是否执行，要等解析阶段拿到本次配置以后再判断。

配置类也不会在注册阶段立刻展开。`.Configuration()` 只是给 BeanDefinition 打上配置类标记，后续解析阶段才会扫描方法。默认规则是扫描 `New.*` 方法，生成的子 Bean 会带上配置类 Bean 本身的条件。

所以注册阶段的核心结论是：这里收集的是“候选定义”，不是“可用对象”。

## 解析阶段

解析阶段要解决的问题是：注册表里这么多候选 Bean，本次容器刷新到底保留哪些。

解析阶段会基于已经加载完成的配置做判断。这里的配置是容器看到的最终配置视图，条件判断、模块激活和后续配置绑定都会基于它。

单独的解析阶段让容器不用一边创建对象，一边再决定这个对象是否应该存在。先解析，可以把候选集合收窄成一个相对确定的对象图输入；后面的注入阶段只需要面对“应该被创建的 Bean”，错误边界也更清楚。

解析阶段主要做五件事。

第一，合并 BeanDefinition。

包级 `gs.Provide()` 进入全局列表，`app.Provide()` 进入当前容器列表。容器刷新时，会把两部分合并成一组候选定义。

第二，执行满足条件的 Module。

`gs.Module()` 不是注册时执行，而是在解析阶段执行。容器会先判断模块条件，条件成立才调用模块函数。`gs.Group()` 也是通过这个机制实现的：先判断配置 key 是否存在，再把配置 Map 展开成多个命名 Bean。

第三，扫描 Configuration。

容器会找到带有 configuration 元数据的 Bean，扫描它的方法，默认把 `New.*` 方法注册成子 Bean。方法 Bean 会自动带上 `OnBeanID(配置类 Bean)` 条件，所以配置类本身被裁剪掉时，它的方法 Bean 也会失效。

第四，执行 Bean 条件裁剪。

每个 Bean 可以挂多个 `Condition`。解析阶段会逐个执行这些条件，只要有一个条件不成立，这个 Bean 就会被标记为删除。

`OnProperty` 看配置，`OnBean` 和 `OnMissingBean` 看当前解析上下文里的有效 Bean。这里不是简单扫描原始注册表，而是会递归解析匹配 Bean 自己的条件。因此，条件看到的是“本次容器刷新中可成立的候选 Bean”。

第五，做重复检查。

Go-Spring 会用 Bean 的名称和类型组成 BeanID，并把 `.Export()` 导出的接口类型也纳入检查。重复检查的粒度是“名称 + 类型”。如果两个 Bean 在同一个名称和同一个类型上冲突，解析阶段会直接失败。

解析阶段结束以后，容器得到的是一组状态为 resolved 的 BeanDefinition。注意，它们仍然只是定义；对象还没有真正创建。

## 注入阶段

注入阶段才真正把 BeanDefinition 变成可用对象。

Go-Spring 不会创建所有 resolved Bean，而是从 Root Bean 出发，沿着依赖关系递归创建。Root Bean 是对象图入口，不是“创建所有 Bean”的开关。

App 层会把应用自身作为一个 Root Bean，也允许用户通过 `app.Root()` 显式追加 Root。本文不展开 App 启动链路，只需要知道：容器拿到 Root 列表以后，只创建从这些 Root 可达的 Bean。

这就是按需创建。它解决的是 Go 项目里很常见的问题：很多包会在 `init()` 里注册通用能力，但一个具体应用未必真的用到它们。如果容器把所有候选 Bean 都创建出来，应用就要不断写排除条件；按需创建则把问题反过来：只有进入 Root 依赖图的 Bean 才会创建。

注入阶段会先建立名称索引和类型索引。类型索引不仅包含 Bean 自身类型，也包含 `.Export()` 导出的接口类型。这样，“按名称找”和“按类型找”都走确定的查找规则，单 Bean、集合 Bean、接口导出也能统一处理。

创建单个 Bean 时，核心顺序是：

```text
处理 DependsOn()
  -> 进入创建栈
  -> 检测循环依赖
  -> 调用构造函数或使用已有实例
  -> 注入 autowire / inject 字段
  -> 绑定 value 字段
  -> 执行 Init 回调
  -> 标记为 wired
```

这里有几个容易混淆的点。

`DependsOn()` 是显式初始化顺序依赖。它不要求当前 Bean 的字段真的引用对方，只表达“对方必须先完成创建”。

`autowire` 和 `inject` 都是依赖注入标签，单 Bean、slice、map 都走同一套查找逻辑。单 Bean 注入如果按类型匹配到多个候选，又没有指定名称，会报错。

`value` 是配置绑定标签。普通字段绑定后就是普通值；如果字段类型是 `gs.Dync[T]`，容器会把它登记到动态配置对象列表里，后续动态刷新只影响这些动态字段。

构造函数 Bean 也是在这个阶段调用。构造函数可以返回单个 Bean，也可以返回 `(Bean, error)`；如果返回 error，容器刷新失败。

生命周期回调顺序也要记住：`Init` 在对象创建和字段注入之后执行；`Destroy` 不在启动阶段执行，只会被收集起来留给关闭阶段。

如果注入阶段失败，常见原因包括：

- 找不到必需 Bean。
- 单 Bean 注入匹配到多个候选 Bean。
- 配置项缺失或类型转换失败。
- 构造函数返回 error。
- 出现非 lazy 的循环依赖。
- `Init` 回调返回 error。

这些错误都会在容器刷新期暴露，而不是拖到业务请求进来以后才失败。

## 运行阶段

从 IoC 容器视角看，运行阶段很简单：容器已经完成对象创建和依赖注入，业务代码使用的是启动期已经注入好的普通 Go 引用。

Go-Spring 不支持运行期自动注入。不会因为某个请求进来，容器再动态查找一个 Bean 注入到对象里；也不会在每次方法调用时通过容器做服务定位。

这个模型的好处是边界清楚。和依赖相关的问题会在容器刷新期暴露，运行期代码面对的是已经装配完成的对象图。

运行期容器仍然可能保留少量状态：

- Destroy 回调，用于关闭阶段释放资源。
- 动态配置对象列表，用于 `gs.Dync[T]` 字段刷新。

如果应用没有任何动态字段，启动完成后 Go-Spring 会清理配置缓存，减少运行期持有的数据。这个细节也说明了 Go-Spring 的基本取向：能在启动期确定的东西，不留到运行期反复处理。

应用层在这个阶段还会继续做自己的生命周期编排，后面的应用启动链路文章会专门展开。

## 关闭阶段

关闭阶段只看 IoC 容器，就是执行收集好的 Destroy 回调。

Destroy 回调来自注册阶段的元数据，例如 `.Destroy(fn)` 或 `.DestroyMethod("Close")`。如果 Destroy 返回 error，框架会记录日志，但不会中断后续 Destroy。

Destroy 回调不是在解析阶段收集的，而是在注入阶段收集的。原因很直接：只有真正被创建并进入对象图的 Bean，才需要关闭。一个 Bean 即使通过了解析阶段，如果没有被 Root 依赖图触达、从未创建，就不会有对应的 Destroy 执行。

Go-Spring 会结合注入路径和 `DependsOn()` 关系对 Destroy 做拓扑排序。实际执行时，被依赖的 Bean 的 Destroy 会先执行，依赖它的 Bean 的 Destroy 后执行。

这个顺序意味着：普通 Bean 的 Destroy 更适合释放自己持有的资源，例如关闭连接池、关闭文件、释放客户端；不要把复杂的跨 Bean 协作收尾放进 Destroy 里。需要和其他运行组件协作的停止动作，通常应该放在 App 层的停止流程或业务自己的退出逻辑中。

因此，本篇里的关闭阶段不是 App 层的完整优雅退出。IoC 容器本身负责的是其中和 Bean 生命周期相关的一步：对已经创建过的 Bean 执行资源释放回调。
