# Go-Spring 实战第 10 课：依赖注入方式：创建边界决定字段注入还是构造函数注入

Go-Spring 容器负责连接对象以后，业务代码最先要决定的是依赖写在哪里。这个决定看起来只是语法选择，实际影响的是对象什么时候进入可用状态。

如果依赖只是在对象创建后补齐，字段注入就很轻量。如果对象一创建出来就必须拿到完整依赖，构造函数注入会把边界表达得更清楚。Go-Spring 同时支持这两种方式，是为了适配不同代码形态，而不是让所有对象都统一成一种写法。

## 字段注入适合对象先创建再补齐依赖

字段注入通过 `autowire` 或 `inject` 标签声明依赖。它适合控制器、配置承载对象和少量简单组件，因为这类对象通常可以先创建，再由容器补齐外部依赖。

```go
type UserController struct {
	Service UserService `autowire:""`
}
```

Go-Spring 容器创建 `UserController` 后，会根据字段类型和标签内容查找匹配的 Bean，并把结果写入字段。

这种写法的优点是直接，不需要额外构造函数。不过它也明确带来一个边界——字段值是在对象创建之后才进入的。因此，构造函数内部不能依赖这些字段，初始化逻辑也应该放到依赖注入完成之后的生命周期阶段。

如果一个对象只是承载少量依赖，字段注入能减少样板代码；如果对象创建时就要用依赖初始化路由、预计算策略或校验参数，字段注入就会把关键动作推迟到创建之后。

## 构造函数注入把可用状态放到创建完成时

构造函数注入通过函数参数声明依赖。下面这个例子里，`UserController` 只有在拿到 `UserService` 后才会被创建出来。

```go
type UserController struct {
	service UserService
}

func NewUserController(service UserService) *UserController {
	return &UserController{service: service}
}

func init() {
	gs.Provide(NewUserController)
}
```

Go-Spring 容器会分析构造函数参数，并在创建 Bean 时注入匹配依赖。

这种方式符合普通 Go 构造函数的写法。依赖在对象创建时已经完整传入，所以对象一返回就处在可用状态，字段也可以保持未导出。测试时同样直接，单元测试可以手动调用 `NewUserController(mockService)`，不需要为了填充字段而启动容器。

## 选择注入方式要看依赖何时必须可用

大多数业务对象更适合构造函数注入。原因是依赖关系集中出现在函数签名里，对象创建完成后依赖已经完整，纯单元测试也更容易手写。

字段注入仍然有合适位置。依赖关系非常简单、写构造函数收益不高，或者需要把配置字段和依赖字段放在同一个结构体上声明时，字段注入会更轻。少量循环依赖场景也可能需要先创建实例，再由容器补齐字段。

所以判断点不是“哪种注入方式更高级”，而是对象的创建边界。依赖复杂、创建时就要完整可用的对象，优先使用构造函数；简单承载型对象，字段注入也能保持代码轻量。

## Arg 让函数参数保留普通 Go 签名

Go 语言不能给函数参数添加 tag。当构造函数参数需要按名称、按配置或按固定值绑定时，Go-Spring 会在注册阶段通过 `Arg` 补充这些语义。

下面的仓储构造函数仍然是普通 Go 函数。真正的选择信息放在注册语句里，`TagArg("slave")` 表示第一个参数注入名为 `slave` 的 Bean。

```go
func NewRepository(ds *DataSource) *Repository {
	return &Repository{ds: ds}
}

func init() {
	gs.Provide(NewRepository, gs.TagArg("slave"))
}
```

如果参数能按类型唯一匹配，`TagArg` 可以省略。只有同类型多实例、配置绑定或固定值进入参数时，才需要显式补充参数绑定信息。这样一来，构造函数签名仍然保持 Go 的自然写法，容器语义集中在注册阶段。

## 清楚的依赖边界比统一写法更重要

字段注入和构造函数注入都能表达依赖，区别在于依赖进入对象的时间。Go-Spring 没有把其中一种包装成唯一正确的风格，而是让对象按照自己的创建边界选择声明方式。

声明方式确定以后，容器还要继续回答一个更具体的问题。这个注入点到底应该拿到单个 Bean、多个 Bean、配置指定的 Bean，还是一个延迟补齐的依赖。
