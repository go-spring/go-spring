# 配置优先级与合并语义

配置来源变多以后，系统立刻会遇到一个更现实的问题：同一个 key 出现在多个地方时，到底谁说了算。

基础配置、环境配置、环境变量、命令行参数和代码默认值经常同时存在。如果多个来源出现同名配置，就必须有确定规则；否则同一份代码在不同启动方式下可能表现完全不同。

Go-Spring 的规则可以概括为两层：

- 来源有明确优先级。
- 不同数据类型有不同合并语义。

我们先看优先级，再看叶子值、Map 和 Slice 的不同处理方式。

## 配置优先级

从高到低，Go-Spring 的配置优先级是：

| 优先级 | 来源 |
|--------|------|
| 1 | 命令行参数 |
| 2 | 环境变量 |
| 3 | Profile 特定配置 |
| 4 | 基础配置文件 |
| 5 | 应用内置默认配置 |
| 6 | 结构体标签默认值 |

这个顺序体现的是工程直觉：越接近运行环境、越临时的配置，优先级越高。也就是说，命令行参数适合一次性覆盖，环境变量适合容器部署，配置文件适合版本化维护，代码默认值则作为兜底。

## 叶子值覆盖

叶子值采用覆盖语义。高优先级来源中存在同一个 key 时，直接覆盖低优先级值。

```properties
# 基础配置
app.port = 8080

# 环境覆盖
app.port = 9000
```

最终结果是：

```properties
app.port = 9000
```

这是最常见、也最容易理解的规则。因为一个具体值只能有一个最终结果，所以优先级高的配置覆盖优先级低的配置。

## Map 合并

Map 对象采用合并语义。不同来源可以各自提供一部分 key，最终合并成完整对象。

```properties
# 基础配置
server.port=8080

# 环境覆盖
server.host=localhost
```

合并结果：

```properties
server.port=8080
server.host=localhost
```

如果同一个 key 同时出现，仍然由高优先级覆盖低优先级。

这种规则适合环境覆盖：生产环境只需要写差异项，不需要把整个对象重复一遍。这样我们可以把通用结构留在基础配置里，把环境差异放到 Profile 或环境变量中。

## Slice 整体替换

Slice 和 Map 不同，采用整体覆盖语义。只要高优先级来源定义了某个数组，低优先级的整个数组都会被忽略。

```properties
# 低优先级
my.list[0]=a
my.list[1]=b

# 高优先级
my.list[0]=c
```

最终结果是：

```properties
my.list[0]=c
```

而不是 `[c, b]`。

这是一个明确的设计取舍。因为数组的局部覆盖很容易制造歧义：追加、替换、删除某个位置的语义都不够清晰。整体替换虽然简单，但行为确定。

## 典型使用方式

我们可以把基础配置放通用值：

```yaml
server:
  port: 8080
  timeout: 5s
features:
  enabled:
    - metrics
    - tracing
```

生产 Profile 只覆盖差异：

```yaml
server:
  timeout: 3s
features:
  enabled:
    - metrics
```

合并后，`server.port` 保留基础配置，`server.timeout` 被覆盖，`features.enabled` 整体替换为生产配置中的数组。

## 使用建议

Map 适合表达可局部覆盖的结构，例如 `server`、`database`、`clients`。Slice 适合表达完整列表，例如启用插件列表、过滤器链路、白名单。

如果希望环境只补充少量项，就不适合依赖数组局部覆盖。这时应该考虑把结构改成 Map，或者让高优先级配置完整声明整个列表。

## 确定性比技巧更重要

配置叠加最怕“看起来能用，实际说不清”。Go-Spring 通过明确的来源优先级和数据类型合并规则，让同一组输入得到稳定结果。
