# Go-Spring 实战第 6 课 —— 配置优先级与合并语义：多来源同名 key 如何决定最终值

上一篇我们讲了配置的来源。其中，Reader 负责解析内容格式，Provider 负责接入外部来源，环境变量和命令行参数也都可以转换成配置 key。这样，不同来源的配置最终都可以进入同一套 `Properties` 模型。

但是配置的来源一多，就会出现一个新的问题，即同一个配置可能同时出现在基础配置、Profile 配置、环境变量、命令行参数和代码默认值里。那么这种情况下，同名 key 的最终值该如何决定呢？

Go-Spring 从两个方面解决这个问题。一方面是确定来源的优先级，用来决定同名的 key 谁会覆盖谁。另一方面是数据形态，用来决定叶子值、Map 和 Slice 分别怎样合并。

## 来源优先级

Go-Spring 将配置来源的优先级从高到低约定如下。

| 优先级 | 来源 | 原因 |
|--------|------|------|
| 1 | 命令行参数 | 表达一次启动的明确意图 |
| 2 | 环境变量 | 通常来自部署环境的覆盖配置 |
| 3 | Profile 特定配置 | 根据环境差异决定不同的配置 |
| 4 | 基础配置文件 | 用户在开箱之后的自定义能力 |
| 5 | 应用内置默认配置 | 避免使用配置文件，同时具备可覆盖性 |
| 6 | 结构体标签默认值 | 字段级的默认值，用来兜底 |

这个顺序的核心判断是，越接近本次运行环境、越临时的输入，优先级应当越高。

## 叶子值

叶子值是指不能再被拆分的最终值，比如端口、超时时间、开关等。对于这类配置，Go-Spring 使用覆盖语义。也就是说，只要高优先级来源里存在同一个 key，低优先级的值就不会再生效。

比如在下面这个例子里，基础配置和环境覆盖都提供了 `app.port`。

```properties
# 基础配置
app.port = 8080

# 环境覆盖
app.port = 9000
```

它们在合并之后只会保留高优先级来源中的值，即 `app.port = 9000`。

这种规则是合理的，因为一个具体的配置项最终只能有一个值，继续保留多个来源也没有意义。

## Map

Map 或对象配置表达的是一组子 key，它们不是一个单独的叶子值。对于这类结构，Go-Spring 会在不同的子 key 之间做合并。

比如下面的两个来源分别提供了 `server` 下面的不同字段。

```properties
# 基础配置
server.port=8080

# 环境覆盖
server.host=localhost
```

它们在合并后结果会同时保留这两个子 key。

```properties
server.port=8080
server.host=localhost
```

也就是说，Go-Spring 合并的是 `server` 下面的子路径集合。

但如果同一个子 key 同时出现，那么对叶子值的优先级规则仍然生效。比如基础配置和环境覆盖都写了 `server.port`，那么最终还是高优先级的 `server.port` 生效。Map 的合并语义只针对不同子 key。

通常来说，这种语义非常适合用来表达环境差异。我们可以在基础配置里放通用字段，然后在 Profile 或环境变量里只写需要变化的部分，而不需要把整个对象重复一遍。

## Slice

Slice 与 Map、叶子值的合并语义都不同。因为数组的下标虽然也能写进 path，但其实数组更接近一个有顺序的整体。所以只要高优先级来源定义了某个数组，那么 Go-Spring 就会用这个数组的整体来替换低优先级来源中的整个数组。

比如下面的高优先级来源只声明了 `my.list[0]`。

```properties
# 低优先级
my.list[0]=a
my.list[1]=b

# 高优先级
my.list[0]=c
```

但它们在合并之后的结果如下，并不是 `[c, b]`。

```properties
my.list[0]=c
```

因为高优先级来源实际上已经定义了 `my.list` 整个数组，所以低优先级来源里的整个数组都会被忽略。

这是一个有意的设计取舍。因为数组的局部合并非常容易产生歧义。因为 `my.list[0]=c` 既可以被理解成只替换第一个元素，也可以被理解成当前环境只启用 `c` 这一个元素。两种解释都说得通，但最终行为完全不同。Go-Spring 选择整体替换，就是为了让数组覆盖保持确定。 // todo (这里的解释并不好，可能在更复杂的配置中才合理，这里的两个解释我看出有啥区别。)

## 环境差异

有了 Map 合并和 Slice 整体替换以后，环境配置就可以只表达差异，但前提是要选对数据形态。

基础配置可以放通用值。

```yaml
server:
  port: 8080
  timeout: 5s
features:
  enabled:
    - metrics
    - tracing
```

生产 Profile 只写变化的部分。

```yaml
server:
  timeout: 3s
features:
  enabled:
    - metrics
```

这组配置合并后，`server.port` 会继续使用基础配置中的 `8080`，`server.timeout` 会被生产 Profile 覆盖成 `3s`。但 `features.enabled` 是数组，所以会整体替换，最终只剩下 `metrics`。

因此，Map 适合表达可以局部覆盖的结构，比如 `server`、`database`、`clients`。Slice 适合表达完整列表，比如启用插件列表、过滤器链路、白名单。如果一个列表经常需要按环境增删某个元素，数组下标就不太适合表达这种差异。这时候可以把结构改成 Map，用命名 key 表达每个条目，或者让高优先级配置完整声明整个列表。

## 配置优先级与合并语义

配置优先级解决的是“同一个 key 来自多个来源时谁生效”的问题，合并语义解决的是“同一棵配置树在不同数据形态下怎样组合”的问题。Go-Spring 把这两件事拆开处理，让命令行参数、环境变量、Profile、基础配置和默认值都能回到同一个 path 空间里，并得到确定的最终结果。
