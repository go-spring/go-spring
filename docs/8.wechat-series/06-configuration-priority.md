# Go-Spring 实战第 6 课：配置优先级与合并语义：多来源同名 key 如何决定最终值

Go-Spring 支持多种配置来源以后，系统立刻会遇到一个更现实的问题，即同一个 key 出现在多个地方时，到底谁说了算呢？

基础配置、环境配置、环境变量、命令行参数和代码默认值经常同时存在。如果多个来源出现同名配置，就需要一套确定规则；不然同一份代码在不同启动方式下可能表现完全不同。

Go-Spring 的规则要分成两层看。第一层是来源优先级，决定同名 key 谁覆盖谁；第二层是数据形态，决定 Map 和 Slice 这类结构如何合并。

先看优先级，再看叶子值、Map 和 Slice 的不同处理方式。只有这两层规则都确定，多来源配置才不会在不同环境里产生隐式差异。

## 来源优先级先决定同名 key 的胜负

从高到低，Go-Spring 的配置优先级如下。

| 优先级 | 来源 |
|--------|------|
| 1 | 命令行参数 |
| 2 | 环境变量 |
| 3 | Profile 特定配置 |
| 4 | 基础配置文件 |
| 5 | 应用内置默认配置 |
| 6 | 结构体标签默认值 |

这个顺序符合工程直觉——越接近运行环境、越临时的配置，优先级越高。因为它们更能代表“本次启动”的意图。命令行参数适合一次性覆盖，环境变量适合容器部署，配置文件适合版本化维护，代码默认值则作为兜底。

## 叶子值只能用高优先级直接覆盖

叶子值用的是覆盖语义。高优先级来源中存在同一个 key 时，直接覆盖低优先级值。

下面这个例子里，两个来源都给出了同一个叶子 key，所以最终结果只能保留高优先级来源的值。

```properties
# 基础配置
app.port = 8080

# 环境覆盖
app.port = 9000
```

最后得到的结果如下。

```properties
app.port = 9000
```

这是最常见、也最容易理解的规则。因为一个具体值只能有一个最终结果，所以优先级高的配置覆盖优先级低的配置。

## Map 用子 key 合并保留低优先级补充项

Map 对象用的是合并语义。不同来源可以各自提供一部分 key，最终合并成完整对象。

下面的 `server.port` 和 `server.host` 是同一对象下的不同子 key，所以 Go-Spring 会把它们合并到同一个 `server` 配置对象里。

```properties
# 基础配置
server.port=8080

# 环境覆盖
server.host=localhost
```

合并结果如下。

```properties
server.port=8080
server.host=localhost
```

如果同一个 key 同时出现，仍然由高优先级覆盖低优先级。也就是说，Map 是在不同子 key 之间合并，而不是放弃优先级规则。

这种规则适合环境覆盖——生产环境只需要写差异项，不需要把整个对象重复一遍。这样我们可以把通用结构留在基础配置里，把环境差异放到 Profile 或环境变量中。

## Slice 整体替换避免下标合并歧义

Slice 和 Map 不同，用的是整体覆盖语义。只要高优先级来源定义了某个数组，低优先级的整个数组都会被忽略。

下面的高优先级来源只声明了 `my.list[0]`。Go-Spring 会把这个数组视为一个整体覆盖，而不是只覆盖第一个元素。

```properties
# 低优先级
my.list[0]=a
my.list[1]=b

# 高优先级
my.list[0]=c
```

最后得到的结果如下。

```properties
my.list[0]=c
```

而不是 `[c, b]`。

这里就是一个明确的设计取舍了。因为数组的局部覆盖很容易制造歧义，即追加、替换、删除某个位置的语义都不够清晰。整体替换虽然简单，但行为确定。

如果这里做成局部合并，`my.list[0]=c` 到底表示“只替换第一个元素，保留后面的 b”，还是“生产环境只启用 c 这一个元素”？这两种解释都说得通，但最终行为完全不同。Go-Spring 选择整体替换，就是为了避免这种隐式规则。

## 环境覆盖只写差异才能体现合并价值

有了 Map 合并和 Slice 整体替换以后，环境配置就可以只表达差异。基础配置放通用值。

```yaml
server:
  port: 8080
  timeout: 5s
features:
  enabled:
    - metrics
    - tracing
```

生产 Profile 只覆盖差异。

```yaml
server:
  timeout: 3s
features:
  enabled:
    - metrics
```

合并后，`server.port` 保留基础配置，`server.timeout` 被覆盖，`features.enabled` 整体替换为生产配置中的数组。

## 数据形态决定该用 Map 还是 Slice

Map 适合表达可局部覆盖的结构，例如 `server`、`database`、`clients`。Slice 适合表达完整列表，例如启用插件列表、过滤器链路、白名单。

如果希望环境只补充少量项，数组局部覆盖就不太合适。这时候可以考虑把结构改成 Map，或者让高优先级配置完整声明整个列表。因为一旦列表语义需要“增删改某个元素”，Map 的命名 key 会比数组下标更容易说清楚。

## 确定性是多来源配置的底线

配置叠加最怕“看起来能用，实际说不清”。Go-Spring 通过明确的来源优先级和数据类型合并规则，让同一组输入得到稳定结果。

有了确定的覆盖规则，Profile 才能真正发挥作用——基础配置负责共性，环境配置只表达差异，多个 Profile 也可以按清晰顺序组合。
