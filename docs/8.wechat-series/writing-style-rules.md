# Go-Spring 微信文章写作规约

本规约用于约束 AI 编写或改写 `docs/8.wechat-series` 中 `00-series-plan.md` 之后的微信正文文章。当前风格基准以已经定版的 `01` 到 `14` 为准。

要求文章以技术准确性为底线，同时保持文章连贯、表达自然。所有标题优化、转场润色、示例调整和语气修改，都不能改变 `01` 到 `14` 已经定版的 API、配置项、生命周期顺序、优先级、错误语义和设计边界。

## 1. 硬规则

1. 主标题使用统一格式：

```text
Go-Spring 实战第 N 课 —— 大主题：具体能力或问题
```

2. 开头从上一篇或真实工程问题进入。可以简短说明本篇范围，但不要写成目录式预告，避免 `本文将介绍...`、`本篇要解决的问题有...`、`首先来看目录...` 这类句式。

3. 每篇只围绕一个明确主题展开。主题可以包含多个分支或多个 API 入口，但这些分支必须共同回答同一个问题；不要写成零基础教程、API 手册、宣传稿或多个主题混在一起的大章。

4. 正文不能只有 API 清单而没有语义主线。机制型文章要讲清规则和边界；概念型文章要讲清普通 Go 写法的约束、Go-Spring 接管的部分和不接管的部分；API 入口型文章要讲清入口解决什么问题、对应什么语义、适合什么场景。

5. 小节标题要具体，可以是对象名、能力名、阶段名或问题名，例如 `Properties`、`value 标签`、`Reader`、`Slice`、`Profile 维度`、`启动期装配`、`哪种方式更好`。不要使用 `总结`、`小结`、`最佳实践`、`注意事项` 这类空泛标题。

6. 示例必须服务于当前小节。可以用 `看个例子`、`示例如下` 承接，但示例前后至少要有一处说明它证明什么；示例后要解释 Go-Spring 语义、适用场景或边界，不要逐行复述代码。

7. 规则型主题可以先给总规则，再按来源、数据形态或阶段拆开解释。机制讲清后，可以补充建模、排查或选择建议，但指导必须来自前文语义。

8. 结尾可以用本篇主题名、具体能力名、选择建议或短收束段落收住主题，说明这块能力在 Go-Spring 整体模型里的位置。如果最后一个具体小节已经自然收住主题，可以不加单独的结尾段。不要写单独的下一篇预告。

9.  进入 IoC、生命周期、日志、HTTP 等新体系时，可以先讲普通 Go 写法和复杂工程中的痛点，再引出 Go-Spring 抽象。不要把 Go-Spring 写成替代 Go 语言习惯的框架。

10. 正文禁止使用 `依赖图`、`对象图`、`Bean 依赖图`、`静态对象图` 以及类似以“图”来组织 IoC、装配或运行期对象关系的概念。

## 2. 写法参考

读者默认已经熟悉 Go 工程开发。不要解释 `struct`、`interface`、`error` 处理、Go module、构建部署这类基础知识。

术语写法保持 `01` 到 `14` 的定版口径。`Go-Spring`、`Properties`、`Path`、`Reader`、`Provider`、`Profile`、`Bean`、`IoC 容器`、`Starter`、`Map`、`Slice` 等专有名词保持稳定写法；配置 path、结构体标签、API 名称、类型名、配置项和命令行参数使用反引号标记。

语气保持自然、克制、连贯。可以使用 `我们`、`咱们`、`大家`，也可以少量使用设问和口语化转场，但不能写成聊天记录、营销稿或手册条目。段落之间有因果、转折、递进或条件关系时，要把关系写出来。

讲设计取舍时，先给约束条件，再推出结论。从通用工程问题转入框架能力时，要写清楚这是 Go-Spring 的能力、Go-Spring 的设计，还是 Go 语言或业务代码本身的约束。

加粗、编号列表和表格都可以使用，但要服务于核心判断、规则顺序或短枚举，不要把复杂推理塞进表格。

## 3. 最终自检

生成或改写后，只检查这些问题：

1. 标题是否符合统一格式。
2. 开头是否从上一篇或工程问题进入，而不是列目录。
3. 正文是否围绕一个主题展开，而不是只有 API 清单。
4. 示例是否有语义解释，而不是只贴代码。
5. 小节标题是否具体。
6. 是否讲清 Go-Spring 的能力、边界和使用建议。
7. 涉及 `01` 到 `14` 已定版语义时，是否与原文一致。
8. 是否避免了 `依赖图`、`对象图` 等图式概念。

一句话要求：从真实工程复杂性出发，用自然但克制的技术文章语气，带读者看到 Go-Spring 的抽象、代码入口、适用边界、设计取舍和工程使用建议。
