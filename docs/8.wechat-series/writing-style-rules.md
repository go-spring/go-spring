# Go-Spring 微信文章写作规约

本规约用于约束 AI 编写或改写 `docs/8.wechat-series` 中 `00-series-plan.md` 之后的微信正文文章。

执行目标：保持已有 `01` 到 `30` 的文章风格，让内容适合微信公众号阅读，同时不改变 Go-Spring 的技术含义。

硬性优先级：技术准确性 > 文章连贯性 > 口语化表达。

连接词、语气助词、标题优化和转场润色都不能改变 API、配置项、生命周期顺序、错误语义和设计边界。

## 1. 基本定位

面向已经熟悉 Go 工程开发、希望理解 Go-Spring 能力边界和设计取舍的读者。

不要解释 Go 基础语法和通用工程常识，例如 `struct`、`interface`、`error` 处理、Go module、构建部署。

不要写成：

- 零基础教程
- API 手册
- 宣传稿
- 多个能力混在一起的大章

每篇文章只围绕一个明确主题展开，并按这个顺序组织内容：

1. 真实工程问题
2. Go-Spring 的抽象或能力
3. 代码或配置示例
4. 示例背后的语义
5. 适用场景和边界

## 2. 标题规则

主标题推荐格式：

```text
Go-Spring 实战第 N 课：大主题：具体问题
```

标题必须同时说明：

- 讲哪个能力
- 解决什么问题

读者即使没有读过前一篇，也要能从标题判断这篇文章是否适合自己。

好的标题：

- `配置优先级与合并语义：多来源同名 key 如何决定最终值`
- `日志输出管线：Appender、Layout、Encoder 各负责什么`

避免这类标题：

- `一文搞懂 Go-Spring 配置`
- `不要随便加 key`
- `高级用法`

小节标题必须有明确对象和判断。不要使用 `概述`、`基本用法`、`注意事项`、`最佳实践`、`总结`、`小结`。收束段也要把结论直接写进标题。

## 3. 开头和正文推进

开头禁止写提纲。不要使用：

- `本文将介绍...`
- `本篇要解决的问题有...`
- `首先来看目录...`

开头通常用两到四段完成：

1. 承接上一篇或引出真实工程场景。
2. 说明问题为什么会变复杂。
3. 说明如果没有统一抽象会出现什么成本。
4. 自然引出 Go-Spring 在这里提供的能力。

正文不要按 API 清单平铺。每个小节只围绕一个判断展开，顺序通常是：

1. 给出判断。
2. 解释原因。
3. 放代码或配置示例。
4. 解释示例语义。
5. 补充边界，或者转入下一节。

如果小节标题可以直接替换成某个 API 或配置项名称，而正文判断不变，说明内容在平铺 API，需要重写。

必须使用确切主语。从通用工程问题转入框架能力时，要写清楚这是 Go-Spring 的能力、Go-Spring 的设计，还是 Go 语言或业务代码本身的约束。不要连续使用 `它`、`这套机制`、`框架` 这类代词。

## 4. 段落连接和口语化

相邻句子或段落之间，如果存在因果、转折、递进或条件关系，必须把关系写出来。

可用连接方式：

- 因果：`所以`、`因此`、`这样一来`
- 转折：`但`、`不过`、`反过来说`
- 递进：`接着`、`然后`、`再往前一步`
- 条件：`如果`、`否则`、`一旦`
- 收束：`也就是说`、`换句话说`

连接词只用于表达真实逻辑关系，不要堆叠。两个句子之间通常只保留一处连接词；如果需要连续堆多个连接词，先重排逻辑。

可以少量使用 `呢`、`其实`、`也`、`就`、`再` 这类语气助词，但只能让转场更自然，不能把文章写成聊天记录。

优先使用更自然的完成态或过程表达：

- `Go-Spring 提供两种方式` -> `Go-Spring 提供了两种方式`
- `配置进入系统后转换成 Properties` -> `配置进入系统后，会先转换成 Properties`
- `错误暴露在启动阶段` -> `错误最好在启动阶段就暴露出来`

改写后自检：如果某句话需要读者自己补逻辑关系或时间顺序，就继续补过渡。

## 5. 解释方式和标点

正文里少用冒号解释。冒号会让句子变成“标题 + 补丁”结构，连续出现时会像信息清单。

解释关系优先级：

1. 先拆成两句话自然展开。
2. 不能拆句时，用 `即`。
3. 需要解释语义时，用 `也就是` 或 `换句话说`。
4. 需要补充说明时，用破折号。

推荐写法：

```text
这时候问题已经变了——不是能不能绑定，而是绑定后的值能不能用。
```

避免写法：

```text
问题是：绑定后的值能不能用。
```

冒号只保留在标题、表格前短提示、代码前提示和列表引出里。如果冒号后面是连续正文，改成拆句或破折号。

## 6. 代码和配置示例

示例必须服务于当前小节的判断。不要为了展示 API 而堆代码。

如果删除代码后，小节判断仍然完整成立，说明代码没有承担解释任务，应删除、合并或替换。

代码上方不要只写 `示例如下`。第一次出现某个 API 或配置项时，代码前要交代：

1. 当前读者正遇到什么问题。
2. 这个例子演示 Go-Spring 的哪一层能力。
3. 读代码时应该先看哪个关键点。

如果上下文已经自然覆盖这三点，不要机械重复。

示例代码下方不要复述代码。要解释选择理由和边界，例如：

- 为什么这里用 `Runner` 而不是 `Server`
- 为什么这里用 `Map` 而不是 `Slice`
- 为什么这个配置适合作为默认值而不是环境覆盖

代码块标题或首行注释尽量呼应小节标题的判断，让读者知道代码在证明什么。

## 7. 设计取舍和语气

必须讲清设计取舍，但不要说教。

讲设计取舍时，先给约束条件，再推出结论。不要直接宣布“应该”或“不应该”。

可以使用 `不是 X，而是 Y`，但只能用于澄清边界。例如：

```text
Go-Spring 的 IoC 容器不是为了追求更动态，而是把依赖关系、环境条件和生命周期集中到启动期处理。
```

避免纠正读者的语气：

```text
这样就不会把它们误解成几套彼此割裂的 API。
```

改成场景化推导：

```text
把这些能力放回同一条生命周期链路里看，配置、IoC、运行时和测试之间的关系会更清楚。
```

不要替读者下判断，也不要替读者感受。避免：

- `你可能会觉得困惑...`
- `大家很容易忽略的是...`
- `说白了就是...`
- `其实没那么复杂...`

结论要有，但必须是推导出来的结论，不是宣布出来的结论。

## 8. 结尾和转场

结尾不要写空泛总结，也不要写单独的 `下一篇预告`。

避免：

- `综上所述，本文详细介绍了 X，希望对大家有所帮助。`
- `下一篇预告：我们将深入讲解 Y 模块的实现。`

最后一个小节仍然使用判断型标题。标题要亮出核心论断，而不是描述“本节讲什么”。

结尾必须先收束本篇主题，说明这块能力在 Go-Spring 整体模型里的位置和价值。

如需衔接下一篇，只用自然过渡句。过渡句要来自逻辑缺口，而不是来自连载预告。

## 9. 最终自检

生成或改写后，只检查这些问题：

1. 标题是否让无上下文读者知道本文讲什么问题。
2. 开头是否从工程场景进入，而不是列提纲。
3. 正文是否按问题、Go-Spring 抽象、示例、边界推进。
4. 相邻段落之间是否补足因果、转折、递进或条件关系。
5. 代码示例上方是否有铺垫，下方是否解释选择理由。
6. 是否在关键位置补出了 `Go-Spring` 或具体能力名称。
7. 是否减少了不必要的冒号解释。
8. 是否避免了说教式表达。
9. 是否没有写成零基础教程、API 手册或营销稿。

一句话要求：从真实工程复杂性出发，带读者一步步看到 Go-Spring 的抽象、代码入口、适用边界和设计取舍。
