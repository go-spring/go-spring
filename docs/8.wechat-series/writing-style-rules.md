# Go-Spring 微信文章写作风格约束

本文档用于约束 AI 编写或改写 `docs/8.wechat-series` 中 `00-series-plan.md` 之后的微信正文文章。

它不是 Go-Spring 的技术规范，而是写作执行规则。目标是让新文章保持已有 `01` 到 `30` 的风格，同时更适合微信公众号阅读。

## 1. 写作目标

文章面向已经熟悉 Go 工程开发、希望理解 Go-Spring 能力边界和设计取舍的读者。

不要写成零基础教程、API 手册或宣传稿。每篇文章只围绕一个明确主题展开，按下面顺序推进：

1. 先说真实工程问题。
2. 再说 Go-Spring 用什么抽象承接。
3. 接着给出代码或配置示例。
4. 然后解释示例背后的语义。
5. 最后收束适用场景和边界。

准确性优先。口语化、转场和标题优化都不能改变 API、配置项、生命周期顺序、错误语义和设计边界。

## 2. 标题规则

主标题推荐使用下面格式。

```text
Go-Spring 实战第 N 课：大主题：具体问题
```

标题必须让没有上下文的读者也能看懂。它要说明“讲哪个能力”和“解决什么问题”，不要只写作者自己知道含义的短句。

好的标题：

- `配置优先级与合并语义：多来源同名 key 如何决定最终值`
- `日志输出管线：Appender、Layout、Encoder 各负责什么`

需要避免的标题：

- `一文搞懂 Go-Spring 配置`
- `不要随便加 key`
- `高级用法`

小节标题也要有明确对象和判断。不要只写 `概述`、`基本用法`、`注意事项`、`最佳实践`、`总结`。

## 3. 开头和正文推进

开头不要写提纲。不要使用 `本文将介绍...`、`本篇要解决的问题有...`、`首先来看目录...`。

开头通常用两到四段完成：

1. 承接上一篇或引出真实工程场景。
2. 说明这个问题为什么会变复杂。
3. 说明如果没有统一抽象会出现什么成本。
4. 自然引出 Go-Spring 在这里提供的能力。

正文不要按 API 清单平铺。每个小节只围绕一个判断展开，通常先给判断，再解释原因，然后放示例，最后补边界或转入下一节。

写作时要在关键位置补出 `Go-Spring` 这个主语。尤其是从通用工程问题转入框架能力时，要写清楚这是 Go-Spring 的能力、Go-Spring 的设计，还是 Go 语言或业务代码本身的约束。不要连续使用 `它`、`这套机制`、`框架` 这类代词。

## 4. 段落连接和口语化

微信公众号文章要让读者跟得上思考路径。相邻句子或段落之间，如果存在因果、转折、递进或条件关系，要把关系写出来。

可以自然使用这些连接方式：

- 因果：`所以`、`因此`、`这样一来`
- 转折：`但`、`不过`、`反过来说`
- 递进：`接着`、`然后`、`再往前一步`
- 条件：`如果`、`否则`、`一旦`
- 收束：`也就是说`、`换句话说`

这些词不是为了堆口语，而是为了减少读者自己补上下文的成本。

可以适当使用少量语气助词，例如 `呢`、`其实`、`也`、`就`、`再`。但语气助词只能让转场更自然，不能把文章写成聊天记录。

同类表达里，优先选择更自然的完成态。

- `Go-Spring 提供两种方式` 可以改成 `Go-Spring 提供了两种方式`。
- `配置进入系统后转换成 Properties` 可以改成 `配置进入系统后，会先转换成 Properties`。
- `错误暴露在启动阶段` 可以改成 `错误最好在启动阶段就暴露出来`。

## 5. 解释方式和标点

正文里少用冒号做解释。连续使用冒号会让文章像提纲或说明书。

解释关系优先用下面几种方式：

- 用 `即` 衔接定义。
- 用 `也就是` 或 `换句话说` 衔接解释。
- 用破折号补充说明。
- 拆成两句话自然展开。

可以写：

```text
这时候问题已经变了——不是能不能绑定，而是绑定后的值能不能用。
```

不要把解释句都写成：

```text
问题是：绑定后的值能不能用。
```

冒号仍然可以用于标题、表格、代码前提示和列表引出，但不要密集使用。

## 6. 示例代码写法

代码示例必须服务于当前小节的判断。不要为了展示 API 而堆代码。

代码上方不能只写 `示例如下`。第一次出现某个 API 或配置项时，代码前至少交代三件事：

1. 当前读者正遇到什么问题。
2. 这个例子演示 Go-Spring 的哪一层能力。
3. 读代码时应该先看哪个关键点。

例如：

```text
下面这个例子先不展开完整业务逻辑，只看构造函数参数是怎样交给 Go-Spring 解析的。读的时候重点看 Provide 后面的 Arg，它补上的就是函数参数本身写不出来的绑定信息。
```

代码下方不要只复述代码。要解释代码背后的选择，例如为什么这里用 `Runner` 而不是 `Server`，为什么这里用 `Map` 而不是 `Slice`，为什么这个配置适合作为默认值而不是环境覆盖。

## 7. 设计取舍和语气

文章要讲清设计取舍，但不要说教。

可以写 `不是 X，而是 Y`，但必须用于澄清边界。例如：

```text
Go-Spring 的 IoC 容器不是为了追求更动态，而是把依赖关系、环境条件和生命周期集中到启动期处理。
```

不要写成纠正读者的语气。比如：

```text
这样就不会把它们误解成几套彼此割裂的 API。
```

可以改成：

```text
把这些能力放回同一条生命周期链路里看，配置、IoC、运行时和测试之间的关系会更清楚。
```

原则是少训诫，多铺陈；少直接下结论，多把判断放进具体场景里自然推出。

## 8. 结尾和转场

结尾不要写空泛总结，也不要写单独的 `下一篇预告`。

最后一个小节仍然应该是判断型标题，例如：

- `覆盖规则最重要的是确定性`
- `运行流程本身就是排错地图`
- `Logger 的职责只是调度事件`

结尾先收束本篇主题，说明这块能力在 Go-Spring 整体模型里的位置和价值。需要衔接下一篇时，用自然过渡句带出即可。

## 9. 最终自检

生成或改写文章后，只检查这些问题：

1. 标题是否让无上下文读者知道本文讲什么问题。
2. 开头是否从工程场景进入，而不是列提纲。
3. 正文是否按问题、Go-Spring 抽象、示例、边界推进。
4. 相邻段落之间是否补足了因果、转折、递进或条件关系。
5. 代码示例上方是否有足够铺垫，下方是否解释了选择理由。
6. 是否在关键位置补出了 `Go-Spring` 或具体能力名称。
7. 是否减少了不必要的冒号解释。
8. 是否避免了说教式表达。
9. 是否没有写成零基础教程、API 手册或营销稿。

一句话概括：这套文章要从真实工程复杂性出发，带着读者一步步看到 Go-Spring 的抽象、代码入口、适用边界和设计取舍。
