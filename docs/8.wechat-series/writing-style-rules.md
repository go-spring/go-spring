# Go-Spring 微信文章写作规约

本规约用于约束 AI 编写或改写 `docs/8.wechat-series` 中 `00-series-plan.md` 之后的微信正文文章。

当前风格基准以已经定版的 `01` 到 `06` 为准。后续文章应延续这几篇的写法：从工程问题进入，围绕一个 Go-Spring 能力或一组明确规则逐层展开，用代码和配置解释语义，必要时补充建模、选择和排查建议，最后把能力放回整体模型中收束。

硬性优先级：技术准确性 > 文章连贯性 > 口语化表达。

连接词、语气调整、标题优化和转场润色都不能改变 API、配置项、生命周期顺序、错误语义和设计边界。

## 1. 基本定位

读者默认已经熟悉 Go 工程开发，希望理解 Go-Spring 的能力边界、使用方式和设计取舍。

不要解释 Go 基础语法和通用工程常识，例如 `struct`、`interface`、`error` 处理、Go module、构建部署。

不要写成：

- 零基础教程
- API 手册
- 宣传稿
- 多个主题混在一起的大章

每篇文章只围绕一个明确主题展开。这个主题可以包含多个自然分支，比如“配置来源”可以包含 Reader、Provider、环境变量和命令行参数，“配置优先级”可以包含来源顺序、叶子值、Map 和 Slice，但这些分支必须共同服务于同一个问题。

文章通常按这个方向推进：

1. 从上一篇或真实工程场景进入。
2. 说明问题为什么会变复杂。
3. 引出 Go-Spring 在这里提供的抽象或能力。
4. 用代码、配置、表格或列表展示使用方式。
5. 解释示例背后的语义、边界和适用位置。
6. 如果主题涉及规则选择，补充建模、排查或选择建议。
7. 在结尾收束这个能力在 Go-Spring 整体模型中的作用。

## 2. 标题规则

主标题使用 01-06 的格式：

```text
Go-Spring 实战第 N 课 —— 大主题：具体能力或问题
```

标题里的“大主题”通常是本篇所在模块，例如配置模型、配置绑定、配置校验、配置来源。冒号后面写清这篇具体讲什么。

推荐标题：

- `Go-Spring 实战第 1 课 —— 统一配置模型：Properties 与 Path`
- `Go-Spring 实战第 2 课 —— 配置绑定：Properties 映射到 Go 类型`
- `Go-Spring 实战第 5 课 —— 配置来源：Reader、Provider、环境变量与命令行参数`

避免这类标题：

- `一文搞懂 Go-Spring 配置`
- `不要随便加 key`
- `高级用法`
- `Go-Spring 配置详解`

小节标题可以是对象名、能力名或问题名，不强制写成完整判断句。`Properties`、`Path 语法`、`value 标签`、`Bind 函数`、`Reader`、`Provider` 这类标题是可以接受的，但正文必须在标题下面讲清它解决什么问题、处在什么层次、和前后能力是什么关系。

指导性小节也要使用具体标题，例如 `配置建模`、`最终值排查`、`环境差异`。不要使用空泛的 `总结`、`小结`、`最佳实践`、`注意事项`。

收束小节也可以使用主题名作为标题，例如 `配置绑定`、`复杂类型绑定`、`配置校验`、`配置来源`、`配置优先级与合并语义`。

## 3. 开头和正文推进

开头禁止写提纲。不要使用：

- `本文将介绍...`
- `本篇要解决的问题有...`
- `首先来看目录...`

开头通常用两到四段完成。

第一段承接上一篇或从一个常见工程场景说起。例如 `上一篇讲了...`、`前面几篇我们...`、`我们先从一个几乎所有项目都会遇到的场景说起`。

第二段说明单纯做法为什么不够。这里要写具体复杂性，比如配置来源变多、值不再只是字符串、校验不能只靠类型转换、输入格式和配置来源混在一起会导致入口割裂。

第三段自然引出 Go-Spring 的抽象或能力。引出时不要宣传，要把能力写成对前面问题的回应。

正文不要按 API 清单平铺。每个小节围绕一个对象或能力展开，顺序通常是：

1. 说明这个对象或能力负责什么。
2. 说明为什么需要它。
3. 给出代码、配置、表格或列表。
4. 解释示例进入 Go-Spring 后的语义。
5. 补充边界，或者转入下一节。

如果一个小节只是把函数签名、参数和返回值逐条列出来，而没有说明它在启动链路、配置模型或绑定语义中的位置，就需要重写。

如果主题本身是一组规则或语义，例如配置优先级、合并规则、生命周期顺序，可以先给出总规则，再按来源、数据形态或阶段拆开解释。每条规则都要说明适用对象、示例结果和边界。

机制讲清以后，可以追加指导性小节，但指导必须来自前文语义。比如讲完 Map 和 Slice 的合并规则后，可以继续讲配置建模和最终值排查；不要脱离机制直接写泛泛的经验建议。

必须使用确切主语。从通用工程问题转入框架能力时，要写清楚这是 Go-Spring 的能力、Go-Spring 的设计，还是 Go 语言或业务代码本身的约束。不要连续使用 `它`、`这套机制`、`框架` 这类代词。

## 4. 段落连接和语气

01-06 的整体语气是技术文章式的自然表达，不是聊天记录，也不是手册条目。

可以使用 `我们` 来建立连续叙述，例如 `我们先从...说起`、`前面几篇我们...`、`我们可以把...`。但不要频繁使用 `你` 来替读者下判断。

相邻句子或段落之间，如果存在因果、转折、递进或条件关系，要把关系写出来。

常用连接方式：

- 因果：`所以`、`因此`、`这样一来`
- 转折：`但`、`不过`、`反过来说`
- 递进：`接着`、`然后`、`进一步`
- 条件：`如果`、`否则`、`一旦`
- 收束：`也就是说`、`换句话说`

连接词只用于表达真实逻辑关系，不要堆叠。两个句子之间通常只保留一处连接词；如果需要连续堆多个连接词，先重排逻辑。

可以少量使用设问来处理真实转场，例如 `那么 Go-Spring 是如何解决这个问题的呢？`。设问后必须马上给出答案，不要用设问制造悬念。

指导性段落可以比机制说明略自然一些，例如 `就可以`、`只需要`、`会更清楚`。这些表达用于让建议更顺，不要写成闲聊、调侃或情绪化判断。

避免过度口语化和纠正读者的语气：

- `你可能会觉得困惑...`
- `大家很容易忽略的是...`
- `说白了就是...`
- `其实没那么复杂...`

## 5. 解释方式和标点

解释要自然嵌入正文，不要写成“标题 + 补丁”的信息清单。

正文中可以使用冒号，但要克制。冒号适合用于标题、列表引出、表格前提示、代码前提示、规则总结和分类说明，以及少量概念解释。连续多段都用冒号引出解释时，会显得像手册。

解释关系优先级：

1. 能拆成两句话时，优先拆句。
2. 需要解释语义时，用 `也就是说` 或 `换句话说`。
3. 需要强调转折或边界时，用破折号。
4. 引出表格、规则列表或短促定义时，可以使用冒号。

推荐写法：

```text
这时候问题已经变了。如果说配置绑定回答的是“能不能转过去”的问题，那么配置校验回答的就是“转过去以后能不能用”的问题。
```

也可以使用 01-06 中常见的直接解释：

```text
Reader 关心的是“这段内容是什么语法”，Provider 关心的是“这段内容从哪里来”。
```

避免连续写成：

```text
问题是：绑定后的值能不能用。
原因是：业务代码需要合法值。
结论是：必须配置校验。
```

## 6. 代码、配置、表格和列表

示例必须服务于当前小节的说明。不要为了展示 API 而堆代码。

代码和配置上方要交代当前示例在证明什么。写法可以很短，例如：

- `下面这段 YAML 同时包含对象和数组。`
- `对应的结构体如下。`
- `下面这个例子演示了一个 JSON Provider，从环境变量里读取一段 JSON。`

如果上下文已经自然覆盖，不需要机械补齐“问题、能力、关键点”三件事。

示例下方不要逐行复述代码。要解释进入 Go-Spring 之后发生了什么、哪些路径会被读取、错误会在哪个阶段暴露、这个写法适合什么场景。

适合使用示例的内容：

- 配置文件如何展开成 `Properties`。
- `value` 标签如何映射到字段。
- `expr` 标签如何校验绑定后的 Go 值。
- Reader 或 Provider 如何接入配置加载流程。
- 环境变量或命令行参数如何转换成配置 key。

表格适合呈现短枚举、优先级、规则分类和简短含义，例如格式与后缀、类型与示例、来源与适用场景、配置来源与含义。表格可以带一列短解释，但不要承载复杂推理，复杂推理要写成正文。

列表适合表达规则、步骤或同类能力。列表下方最好有一两句话收束，让读者知道这些条目共同说明了什么。

块引用只用于强调重要边界，例如连续下标要求、启动期失败语义这类容易影响使用结果的约束。不要把普通提醒都写成块引用。

## 7. 设计取舍和边界

必须讲清设计取舍，但不要说教。

讲设计取舍时，先给约束条件，再推出结论。不要直接宣布“应该”或“不应该”。

推荐写法：

```text
命令行参数不适合承载很长、很复杂的配置。它更像一次启动的明确指令。如果配置需要长期维护、多人协作或者表达环境差异，那么放回配置文件、Profile 或配置中心会更清楚。
```

可以使用 `不是 X，而是 Y`，但只能用于澄清边界。例如：

```text
配置校验不是为了替代业务初始化，而是先把局部、确定、字段内的错误拦下来。
```

边界要落在具体能力上，而不是落在读者感受上。写 `expr 标签解决的是字段级问题`，不要写 `不要误以为 expr 什么都能做`。

当一个能力有前置条件、注册时机或失败时机时，要明确写出来。例如转换器和校验函数应该在 `init` 阶段注册，绑定和校验错误会在启动阶段暴露。

可以使用反事实设问解释设计取舍。例如先问“如果数组采用局部合并，会发生什么情况”，再说明下标无法表达稳定身份，最后回到 Go-Spring 选择整体替换是为了保证结果确定。反事实设问只能用于解释为什么不采用另一种设计，不能用来制造悬念。

## 8. 结尾和转场

结尾不要写空泛总结，也不要写单独的 `下一篇预告`。

避免：

- `综上所述，本文详细介绍了 X，希望对大家有所帮助。`
- `下一篇预告：我们将深入讲解 Y 模块的实现。`

最后一个小节使用本篇主题名作为标题即可。结尾段一般用一段完成，先收束本篇主题，再说明这块能力在 Go-Spring 整体模型中的位置。

推荐收束方式：

```text
配置绑定是 `Properties` 进入业务代码的关键一步。它把配置树上的 path、默认值和类型转换连接起来，让业务代码拿到的是结构化的 Go 对象，而不是散落的字符串配置。
```

如需衔接下一篇，只用自然过渡句。过渡句要来自当前主题留下的逻辑缺口，而不是来自连载预告。

## 9. 最终自检

生成或改写后，只检查这些问题：

1. 主标题是否符合 `Go-Spring 实战第 N 课 —— 大主题：具体能力或问题`。
2. 开头是否从上一篇或工程场景进入，而不是列提纲。
3. 正文是否围绕一个主题展开，而不是平铺 API 清单。
4. 小节标题是否简洁清楚，正文是否补足了对象、语义和边界。
5. 相邻段落之间是否补足因果、转折、递进或条件关系。
6. 代码和配置示例上方是否有自然铺垫，下方是否解释了 Go-Spring 语义。
7. 如果主题涉及规则选择，是否补充了建模、排查或选择建议。
8. 是否在关键位置补出了 `Go-Spring` 或具体能力名称。
9. 是否减少了手册式冒号解释和逐条参数复述。
10. 是否避免了说教式表达、营销表达和零基础教程式解释。
11. 结尾是否用主题名收束，并说明该能力在整体模型中的作用。

一句话要求：从真实工程复杂性出发，用自然但克制的技术文章语气，带读者看到 Go-Spring 的抽象、代码入口、适用边界、设计取舍和工程使用建议。
