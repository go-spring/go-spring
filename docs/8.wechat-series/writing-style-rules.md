# Go-Spring 微信文章写作规约

本规约用于约束 AI 编写或改写 `docs/8.wechat-series` 中 `00-series-plan.md` 之后的微信正文文章。当前风格基准以已经定版的 `01` 到 `10` 为准。

`01` 到 `08` 是配置体系样稿，重点展示机制、规则、示例和工程边界的写法；`09` 到 `10` 是 IoC 体系开篇样稿，重点展示概念定位、普通 Go 写法对照、设计取舍和使用建议的写法。后续文章要同时继承这两类写法。

硬性优先级：技术准确性 > 文章连贯性 > 口语化表达。

所有标题优化、转场润色、示例调整和语气修改，都不能改变 API、配置项、生命周期顺序、优先级、错误语义和设计边界。

## 1. 硬规则

1. 主标题使用统一格式：

```text
Go-Spring 实战第 N 课 —— 大主题：具体能力或问题
```

2. 开头从上一篇或真实工程问题进入，不写提纲。系列切换或新主题开篇时，可以用一两句话说明本篇范围，但不能写成目录式预告。不要使用 `本文将介绍...`、`本篇要解决的问题有...`、`首先来看目录...`。

3. 每篇只围绕一个明确主题展开。主题可以包含多个分支，但这些分支必须共同回答同一个问题。不要写成零基础教程、API 手册、宣传稿或多个主题混在一起的大章。

4. 正文按文章类型选择推进方式，但都不能按 API 清单平铺。

   - 机制型文章按“问题 -> Go-Spring 抽象或规则 -> 示例 -> 语义 -> 边界或指导”推进。
   - 概念定位或设计取舍型文章按“普通 Go 写法 -> 复杂度或约束 -> 编程原则 -> Go-Spring 抽象 -> 示例 -> 定位或边界”推进。
   - 用法对比型文章按“方式定义 -> 最小示例 -> 匹配语义或错误语义 -> 使用建议”推进。

5. 小节标题可以是对象名、能力名、阶段名或问题名，例如 `Properties`、`value 标签`、`Reader`、`Slice`、`Profile 维度`、`启动期装配`、`哪种方式更好`、`配置建模`、`最终值排查`。标题要具体，不要使用 `总结`、`小结`、`最佳实践`、`注意事项` 这类空泛标题。复杂主题可以少量使用三级标题承接扩展点，例如 `扩展 Reader`、`spring.app.imports`、`原始绑定`、`PropertiesRefresher`，但不要堆叠层级。

6. 示例必须服务于当前小节。示例上方要说明它证明什么，示例下方要解释 Go-Spring 语义、适用场景或边界，不要逐行复述代码。

7. 如果主题是一组规则，例如配置优先级、合并语义、生命周期顺序，可以先给总规则，再按来源、数据形态或阶段拆开解释。机制讲清后，可以补充 `配置建模`、`最终值排查` 这类指导性小节，但指导必须来自前文语义。

8. 结尾通常使用本篇主题名作为小节标题，收束本篇主题，并说明这块能力在 Go-Spring 整体模型里的位置。如果文章已经在最后一个具体能力小节里自然收束，可以不另起总结标题，也可以用 `---` 后接一小段定位性收束。系列阶段结束时，可以在结尾承接下一个体系，但必须服务于当前能力在整体模型中的位置，不要写单独的下一篇预告。

9. 涉及 `01` 到 `10` 已经定版的语义时，以这些正文为事实来源。不要改写大小写匹配、slice 下标连续性、Reader 与 Provider 职责边界、`expr` 与必填字段边界、来源优先级、Map 合并、Slice 整体替换、Profile 显式激活与声明顺序、变量引用解析时机、动态刷新边界、IoC 职责分离、Bean 定义、启动期装配、字段注入、构造函数注入、`TagArg` 和构造函数返回非 nil `error` 的错误语义等已经确定的技术语义。

10. 进入 IoC、生命周期、日志、HTTP 等新体系时，可以先讲普通 Go 写法和复杂工程中的痛点，再引出 Go-Spring 抽象。不要把 Go-Spring 写成替代 Go 语言习惯的框架；要强调对象、构造函数、包组织和业务代码仍然保持普通 Go 写法，只是把重复装配、生命周期和横切能力交给 Go-Spring 统一处理。

## 2. 写法参考

读者默认已经熟悉 Go 工程开发。不要解释 `struct`、`interface`、`error` 处理、Go module、构建部署这类基础知识。即使是 IoC 这类概念开篇，也只解释对象创建、依赖声明和装配边界，不讲 Go 语言入门知识。

术语写法保持 `01` 到 `10` 的定版口径。`Go-Spring`、`Properties`、`Path`、`Reader`、`Provider`、`Profile`、`Bean`、`IoC 容器`、`Starter`、`Map`、`Slice` 等专有名词保持稳定写法；配置 path、结构体标签、API 名称、类型名、配置项和命令行参数使用反引号标记，例如 `value`、`expr`、`autowire`、`inject`、`gs.Dync[T]`、`gs.Provide`、`gs.Run`、`gs.TagArg`。

语气保持自然、克制、连贯。可以使用 `我们`、`咱们`、`大家` 建立连续叙述，也可以少量使用设问、冒号和口语化转场，但不能写成聊天记录、营销稿或手册条目。第一人称表达只用于开篇或体系切换时的轻量衔接，正文仍以机制和判断为主。

相邻段落之间如果有因果、转折、递进或条件关系，要把关系写出来。常用表达包括 `所以`、`因此`、`但`、`不过`、`如果`、`也就是说`、`换句话说`。

从通用工程问题转入框架能力时，要写清楚这是 Go-Spring 的能力、Go-Spring 的设计，还是 Go 语言或业务代码本身的约束。不要连续使用 `它`、`这套机制`、`框架` 这类代词。

讲设计取舍时，先给约束条件，再推出结论。可以使用反事实设问解释为什么不采用另一种设计，但设问后必须马上回答，并回到 Go-Spring 的确定性语义。

加粗只用于强调本篇核心判断、关键原则或定义，不要把普通结论都加粗。编号列表适合呈现装配步骤、问题枚举或规则顺序，但列表后要回到正文解释它们共同说明的语义。

表格适合呈现短枚举、优先级、规则分类和简短含义。表格可以带一列短解释，但复杂推理要写成正文。

## 3. 最终自检

生成或改写后，只检查这些问题：

1. 标题是否符合统一格式。
2. 开头是否从上一篇或工程问题进入，而不是列提纲。
3. 正文是否围绕一个主题展开，而不是平铺 API 清单。
4. 示例前是否有铺垫，示例后是否解释了 Go-Spring 语义。
5. 小节标题是否具体，是否避免了空泛标题。
6. 如果主题涉及规则选择，是否补充了建模、排查或选择建议。
7. 如果主题涉及概念定位，是否讲清普通 Go 写法的约束、Go-Spring 接管的边界和不接管的部分。
8. 是否在关键位置写清楚 `Go-Spring` 或具体能力名称。
9. 是否避免了说教式表达、营销表达和零基础教程式解释。
10. 结尾是否用主题名、具体能力名或短收束段落说明该能力在整体模型中的作用。
11. 涉及 `01` 到 `10` 已定版语义时，是否与原文保持一致。

一句话要求：从真实工程复杂性出发，用自然但克制的技术文章语气，带读者看到 Go-Spring 的抽象、代码入口、适用边界、设计取舍和工程使用建议。
