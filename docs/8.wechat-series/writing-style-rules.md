# Go-Spring 微信文章写作风格约束

本文档用于约束 AI 在编写或改写 Go-Spring 微信系列文档时的写作风格。

这些规则不是 Go-Spring 项目本身的技术规则，而是文章如何展开、标题如何设计、段落如何起承转合，以及哪些写法需要避免。目标是让新文章看起来和 `01` 到 `30` 已有文章保持同一套表达习惯。

## 总体定位

这套文章不是零基础教程，也不是 API 手册，而是面向已经熟悉 Go 工程开发、希望理解 Go-Spring 能力边界和设计取舍的读者。

写作时要保持工程化表达。文章应该从真实工程问题切入，再讲 Go-Spring 的抽象、典型代码和适用边界。不要为了完整而解释大量基础概念，也不要为了宣传而夸大能力。

一篇文章只围绕一个明确主题展开。主题可以是一个能力、一条链路、一个设计取舍或一组相邻 API，但不能把多个独立能力硬塞成一篇“大而全”的文章。

## 标题规则

标题必须同时说明主题和具体问题。

推荐格式如下。

```text
Go-Spring 实战第 N 课：大主题：具体问题
```

标题里的“大主题”说明本文属于哪个能力域，“具体问题”说明读者看完后能理解什么。例如：

- `配置优先级与合并语义：多来源同名 key 如何决定最终值`
- `IoC 设计取舍：为什么不提供运行时 getBean`
- `日志输出管线：Appender、Layout、Encoder 各负责什么`

标题不要写成营销话术，也不要写成泛泛的概念名。

禁止使用下面这类标题。

- `一文搞懂 Go-Spring 配置`
- `Go-Spring 配置最佳实践`
- `从入门到精通 IoC`
- `你必须知道的几个高级用法`

## 开头规则

开头不要列提纲，要先把工程问题说清楚。

通常用两到四段完成开头。

第一段承接前文或引出真实工程场景。

第二段说明问题为什么会变复杂。

第三段指出如果没有统一抽象会发生什么。

最后自然引出本文要讲的 Go-Spring 能力。

可以使用下面这些句式。

- `上一篇我们已经看过...现在可以把视角往前挪一步。`
- `真实应用里，问题不会停在...`
- `如果这些能力都散落在业务代码里，边界会很难统一。`
- `所以，Go-Spring 把这件事收进...`

不要用提纲式开头。

禁止使用下面这些写法。

- `本文将介绍...`
- `本篇要解决的问题有三个...`
- `首先我们来看目录...`
- `废话不多说，直接开始...`
- `大家好，今天给大家带来...`

## 内容展开顺序

正文不要按 API 清单平铺，要按工程理解顺序展开。

推荐顺序是：

1. 先说明问题从哪里来。
2. 再说明 Go-Spring 用什么抽象承接这个问题。
3. 然后给出最小可读的代码或配置示例。
4. 接着解释示例背后的语义。
5. 最后补上适用边界和常见取舍。

每个能力都要尽量回答这几个问题。

- 它解决什么工程问题。
- 它为什么放在这个层次。
- 它和相邻能力的边界是什么。
- 什么场景适合用它。
- 什么场景不应该滥用它。

不要只写“怎么用”，也不要只写“为什么”。这套文章的基本节奏是：问题、抽象、代码、边界。

## 小节标题规则

小节标题要带判断，不要只写名词。

好的小节标题应该像一句可以独立成立的工程判断。例如：

- `Runner 不是长期任务的放置点`
- `动态刷新不等于资源重建`
- `字段稳定性比字段数量更重要`
- `注册入口要按组织边界来选`
- `配置错误越早暴露越好`

不要使用过于泛化的小节标题。

禁止使用下面这类标题。

- `概述`
- `基本用法`
- `注意事项`
- `高级用法`
- `最佳实践`
- `总结`

如果一个小节标题只能说明“这里有一组内容”，而不能说明“这里要给出什么判断”，就需要重写。

## 段落推进规则

每个小节只围绕一个判断展开。

常见结构如下。

1. 第一段给判断。
2. 第二段说明为什么。
3. 中间放代码、配置或表格。
4. 示例之后解释关键语义。
5. 末尾补适用场景、边界或转入下一节。

段落不要过长。一个段落通常只表达一个意思。复杂判断可以拆成短段，而不是写成长篇解释。

可以使用克制的转折和澄清句式。

- `也就是说，...`
- `换句话说，...`
- `反过来说，...`
- `这时候问题已经不是...而是...`
- `边界清楚后，...`
- `可以先按这条顺序看...`

这些句式要服务于解释，不要为了形成风格而机械堆叠。

## 代码和配置示例规则

示例必须服务于当前段落的判断。

代码前要说明示例解决什么问题。代码后要解释它表达了什么语义、适合什么场景，或者有什么边界。

不要让代码块孤立出现。不要为了展示 API 而堆很多不必要的代码。

示例优先保持已有 guides 或已有文章中的 API、配置项和术语，不要为了文章感强行改写技术含义。

如果示例只是说明核心顺序，可以用简化代码；如果示例涉及真实使用方式，就要保持 API 和配置名称准确。

## 表格和列表规则

列表用于归纳，不用于替代解释。

适合使用列表的场景包括：

- 展示能力分类。
- 展示流程阶段。
- 展示选择顺序。
- 展示几个并列边界。

列表之后通常要再补一段解释，把这些项重新收束到当前主题上。

表格适合做对比，但不要把文章写成参数表。表格中的内容要能帮助读者做工程判断，而不是罗列所有 API。

## 设计取舍写法

文章要多讲设计取舍，少写宣传。

可以使用“不是 X，而是 Y”的结构，但必须用于澄清边界。例如：

- `Go-Spring 的 IoC 容器不是为了追求更动态，而是把依赖关系集中到启动期处理。`
- `动态配置适合开关、阈值、超时时间等轻量值，不直接替代资源生命周期管理。`

不要把取舍写成情绪化评价。不要说某个设计“完美”“优雅”“强大”，而要说明它牺牲了什么、换来了什么。

如果涉及两种方案对比，要避免简单下结论。应该说明它们面向不同工程约束。

## 转场规则

文章内部转场要自然承接。

上一节结束时，可以顺手带出下一节的问题。例如：

- `绑定成功只说明配置能落到类型上，不代表这个值一定可用。`
- `注册之后还要看本次是否生效。`
- `Logger 选好以后，真正写出日志前还要经过 Appender、Layout 和 Encoder。`

不要突然从一个 API 跳到另一个 API。每次切换都要让读者知道视角为什么移动。

跨篇转场要轻，不要写成单独的“下一篇预告”。结尾可以自然带一句下一篇会继续处理什么问题。

## 结尾规则

结尾不要写空泛总结。

最后一个小节仍然要是判断型标题，例如：

- `配置校验守住启动前置条件`
- `覆盖规则最重要的是确定性`
- `运行流程本身就是排错地图`
- `Logger 的职责只是调度事件`

结尾要先完成本篇主题收束，再自然带到下一篇。收束时要说明这块能力在 Go-Spring 整体模型中的位置和价值。

禁止使用下面这类结尾。

- `本文介绍了...希望对你有帮助。`
- `以上就是本篇全部内容。`
- `下一篇我们继续讲...敬请期待。`
- `如果你觉得不错，欢迎点赞转发。`

## 语言风格

语言要克制、直接、工程化。

常用表达包括：

- `通常`
- `更适合`
- `可以先`
- `这时候`
- `也就是说`
- `反过来说`
- `边界清楚后`
- `最终要服务于`

少用绝对表达。不要写“必须永远”“一定最好”“唯一正确”。如果确实是强约束，要说明约束来自哪里。

不要使用营销化、口号化或情绪化表达。

禁止使用下面这类表达。

- `非常强大`
- `极致优雅`
- `轻松拿捏`
- `保姆级`
- `闭眼用`
- `史上最全`
- `灵魂拷问`

## 禁止项

写作时必须避免下面这些问题。

- 不要写成零基础教程。
- 不要大量解释 Go 基础语法、HTTP 基础概念、依赖注入基础概念。
- 不要写成 API 参考手册。
- 不要把多个独立能力合并成一篇大章。
- 不要用标题党。
- 不要用公众号套话开头或结尾。
- 不要为了显得完整而展开无关背景。
- 不要只展示代码，不解释工程语义。
- 不要只讲优点，不讲边界。
- 不要把“下一篇预告”写成单独章节。

## 写作自检清单

生成文章后，按下面清单检查。

1. 标题是否同时包含主题和具体问题。
2. 开头是否从工程问题进入，而不是列提纲。
3. 每个小节标题是否带有明确判断。
4. 正文是否按问题、抽象、代码、边界推进。
5. 代码示例前后是否都有解释。
6. 是否避免了零基础科普和 API 清单式写法。
7. 是否讲清楚适用场景和不适合的场景。
8. 结尾是否先收束本篇主题，再自然过渡。
9. 是否没有使用营销化、口号化、夸张表达。
10. 读完后是否能看出这篇文章在 Go-Spring 整体能力链路中的位置。

一句话概括：这套文章的风格不是教程腔，也不是宣传稿，而是从真实工程复杂性出发，把一个能力的抽象、代码入口、适用边界和设计取舍讲清楚。
