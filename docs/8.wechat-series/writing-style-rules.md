# Go-Spring 微信文章写作风格约束

本文档用于约束 AI 在编写或改写 Go-Spring 微信系列文档时的写作风格。

这些规则不是 Go-Spring 项目本身的技术规则，而是文章如何展开、标题如何设计、段落如何起承转合，以及哪些写法需要避免。目标是让新文章看起来和 `01` 到 `30` 已有文章保持同一套表达习惯。

适用范围主要是 `00-series-plan.md` 之后的正文文章，即 `01` 到 `30` 以及后续新增的微信文章。`00-series-plan.md` 是规划文档，可以不完全套用正文写作节奏。

## 总体定位

这套文章不是零基础教程，也不是 API 手册，而是面向已经熟悉 Go 工程开发、希望理解 Go-Spring 能力边界和设计取舍的读者。

写作时要保持工程化表达。文章应该从真实工程问题切入，再讲 Go-Spring 的抽象、典型代码和适用边界。不要为了完整而解释大量基础概念，也不要为了宣传而夸大能力。

一篇文章只围绕一个明确主题展开。主题可以是一个能力、一条链路、一个设计取舍或一组相邻 API，但不能把多个独立能力硬塞成一篇“大而全”的文章。

文章最终会发布到微信公众号，所以除了准确，还要注意可读性和亲民性。这里的亲民不是把技术讲浅，而是让读者能跟上作者的思考路径：为什么要讲这件事，前后两段是什么关系，示例代码为什么出现在这里，看完以后能把它放回 Go-Spring 的整体能力里。

## 标题规则

标题必须同时说明主题和具体问题。

推荐格式如下。

```text
Go-Spring 实战第 N 课：大主题：具体问题
```

标题里的“大主题”说明本文属于哪个能力域，“具体问题”说明读者看完后能理解什么。例如：

- `配置优先级与合并语义：多来源同名 key 如何决定最终值`
- `IoC 设计取舍：为什么不提供运行时 getBean`
- `日志输出管线：Appender、Layout、Encoder 各负责什么`

标题不要写成营销话术，也不要写成泛泛的概念名。

标题必须让读者在没有上下文的情况下也能看懂。不要使用只有作者自己知道含义的短句，也不要把问题压缩成不合常理的口号。例如“不要随便加 key”这种标题不够清楚，因为读者不知道是什么 key、加到哪里、为什么不能随便加。可以改成更具体的表达，例如“配置 key 要先回到统一路径模型里”。

小节标题也要遵守这个原则。标题可以有判断，但判断必须有明确对象和原因，不能只写一个看起来有态度、实际不解释对象的短句。

禁止使用下面这类标题。

- `一文搞懂 Go-Spring 配置`
- `Go-Spring 配置最佳实践`
- `从入门到精通 IoC`
- `你必须知道的几个高级用法`

## 开头规则

开头不要列提纲，要先把工程问题说清楚。

通常用两到四段完成开头。

第一段承接前文或引出真实工程场景。

第二段说明问题为什么会变复杂。

第三段指出如果没有统一抽象会发生什么。

最后自然引出本文要讲的 Go-Spring 能力。

可以使用下面这些句式。

- `上一篇我们已经看过...现在可以把视角往前挪一步。`
- `真实应用里，问题不会停在...`
- `如果这些能力都散落在业务代码里，边界会很难统一。`
- `所以，Go-Spring 把这件事收进...`

不要用提纲式开头。

禁止使用下面这些写法。

- `本文将介绍...`
- `本篇要解决的问题有三个...`
- `首先我们来看目录...`
- `废话不多说，直接开始...`
- `大家好，今天给大家带来...`

## 口语化连接规则

文章可以比普通技术文档更口语化一些，尤其要加强因果、转折和递进关系。读者在微信公众号里阅读时，不会像读 API 文档那样反复回看上下文，所以文章要主动把思考路径写出来。

适合补充的连接词包括：

- 因果：`所以`、`因此`、`因为`、`这样一来`、`也正因为如此`
- 转折：`但`、`不过`、`反过来说`、`问题是`、`真正麻烦的是`
- 递进：`接着`、`然后`、`再往前一步`、`再往外一层`、`到了真实项目里`
- 条件：`如果`、`否则`、`一旦`、`只有...才...`
- 收束：`也就是说`、`换句话说`、`最终还是回到...`

这些词不是为了让文章变得松散，而是为了让读者知道作者为什么从上一句话走到下一句话。尤其是两个判断之间，如果缺少因果或转折，就要补一句过渡。

可以适当使用少量语气助词，拉近和读者的距离。例如 `呢`、`其实`、`也`、`就`、`再`。但语气助词只能用于自然过渡，不能把文章写成聊天记录。每段里如果已经有明确连接关系，就不要再硬加语气词。

同类表达里，优先选择更自然的完成态或口语化说法。例如 `Go-Spring 提供了...` 通常比 `Go-Spring 提供...` 更顺；`已经进入...` 通常比 `进入...` 更有过程感；`可以先看...` 通常比 `看...` 更像是在带读者往下走。

## 解释标点规则

正文里不要过度依赖冒号来做解释。连续使用冒号会让文章更像提纲或说明书，读起来不够自然。

遇到解释关系时，优先考虑下面几种写法。

- 用 `即` 衔接定义，例如 `这里的 Provider，即从某个来源读取配置的扩展点。`
- 用 `也就是` 或 `换句话说` 衔接解释，例如 `它解决的不是格式问题，也就是配置从哪里来的问题。`
- 用破折号补充说明，例如 `这时候问题已经变了——不是能不能绑定，而是绑定后的值能不能用。`
- 拆成两句话，让解释自然展开。

冒号仍然可以用于标题、表格说明、代码前提示和列表引出，但不要把每个解释句都写成 `A：B`。

## 内容展开顺序

正文不要按 API 清单平铺，要按工程理解顺序展开。

推荐顺序是：

1. 先说明问题从哪里来。
2. 再说明 Go-Spring 用什么抽象承接这个问题。
3. 然后给出最小可读的代码或配置示例。
4. 接着解释示例背后的语义。
5. 最后补上适用边界和常见取舍。

每个能力都要尽量回答这几个问题。

- 它解决什么工程问题。
- 它为什么放在这个层次。
- 它和相邻能力的边界是什么。
- 什么场景适合用它。
- 什么场景不应该滥用它。

不要只写“怎么用”，也不要只写“为什么”。这套文章的基本节奏是：问题、抽象、代码、边界。

写作时还要有意识地补充 Go-Spring 这个主语。尤其是一段从通用工程问题转入框架能力时，要写清楚这是 Go-Spring 的能力、Go-Spring 的设计，还是 Go 语言或业务代码本身的约束。不要连续使用“它”“这套机制”“框架”这类代词，导致读者忘了当前讲的是 Go-Spring 的哪一层能力。

不过，Go-Spring 这个名字也不要每句都重复。通常在小节开头、能力归属变化处、结论收束处补一次即可。

## 小节标题规则

小节标题要带判断，不要只写名词。

好的小节标题应该像一句可以独立成立的工程判断。例如：

- `Runner 不是长期任务的放置点`
- `动态刷新不等于资源重建`
- `字段稳定性比字段数量更重要`
- `注册入口要按组织边界来选`
- `配置错误越早暴露越好`

不要使用过于泛化的小节标题。

禁止使用下面这类标题。

- `概述`
- `基本用法`
- `注意事项`
- `高级用法`
- `最佳实践`
- `总结`

如果一个小节标题只能说明“这里有一组内容”，而不能说明“这里要给出什么判断”，就需要重写。

## 段落推进规则

每个小节只围绕一个判断展开。

常见结构如下。

1. 第一段给判断。
2. 第二段说明为什么。
3. 中间放代码、配置或表格。
4. 示例之后解释关键语义。
5. 末尾补适用场景、边界或转入下一节。

段落不要过长。一个段落通常只表达一个意思。复杂判断可以拆成短段，而不是写成长篇解释。

可以使用克制的转折和澄清句式。

- `也就是说，...`
- `换句话说，...`
- `反过来说，...`
- `这时候问题已经不是...而是...`
- `边界清楚后，...`
- `可以先按这条顺序看...`

这些句式要服务于解释，不要为了形成风格而机械堆叠。

前后句之间如果只是并排摆放，读者就需要自己补关系。修改文章时，要重点检查相邻两段之间是否缺少连接。常见问题包括：

- 上一段刚讲完问题，下一段直接贴 API，中间缺少“Go-Spring 如何接住这个问题”。
- 上一段讲完一种方案，下一段直接讲另一种方案，中间缺少“为什么要换视角”。
- 上一段说完适用场景，下一段直接说边界，中间缺少“如果用错会怎样”。

这些地方不一定需要长篇解释，很多时候补一句 `如果继续沿着这个思路看...`、`但到了真实项目里...`、`否则这个能力就会被误用到...` 就够了。

## 代码和配置示例规则

示例必须服务于当前段落的判断。

代码前要说明示例解决什么问题。代码后要解释它表达了什么语义、适合什么场景，或者有什么边界。

不要让代码块孤立出现。不要为了展示 API 而堆很多不必要的代码。

示例优先保持已有 guides 或已有文章中的 API、配置项和术语，不要为了文章感强行改写技术含义。

如果示例只是说明核心顺序，可以用简化代码；如果示例涉及真实使用方式，就要保持 API 和配置名称准确。

代码上方的介绍不能太短。尤其是第一次出现某个 API 或配置项时，代码前至少要交代三件事：

1. 当前读者正遇到什么问题。
2. 这个例子准备演示 Go-Spring 的哪一层能力。
3. 读代码时应该先看哪个关键点。

例如，不要只写 `示例如下。`。更好的写法是：`下面这个例子先不展开完整业务逻辑，只看构造函数参数是怎样交给 Go-Spring 解析的。读的时候重点看 Provide 后面的 Arg，它补上的就是函数参数本身写不出来的绑定信息。`

代码下方也不要只复述代码。要解释代码背后的选择，例如为什么这里用 `Runner` 而不是 `Server`，为什么这里用 `Map` 而不是 `Slice`，为什么这个配置适合作为默认值而不是环境覆盖。

## 表格和列表规则

列表用于归纳，不用于替代解释。

适合使用列表的场景包括：

- 展示能力分类。
- 展示流程阶段。
- 展示选择顺序。
- 展示几个并列边界。

列表之后通常要再补一段解释，把这些项重新收束到当前主题上。

表格适合做对比，但不要把文章写成参数表。表格中的内容要能帮助读者做工程判断，而不是罗列所有 API。

## 设计取舍写法

文章要多讲设计取舍，少写宣传。

可以使用“不是 X，而是 Y”的结构，但必须用于澄清边界。例如：

- `Go-Spring 的 IoC 容器不是为了追求更动态，而是把依赖关系集中到启动期处理。`
- `动态配置适合开关、阈值、超时时间等轻量值，不直接替代资源生命周期管理。`

不要把取舍写成情绪化评价。不要说某个设计“完美”“优雅”“强大”，而要说明它牺牲了什么、换来了什么。

如果涉及两种方案对比，要避免简单下结论。应该说明它们面向不同工程约束。

讲取舍时要避免说教。文章应该引导读者跟着问题往前走，而不是直接替读者下判断。像 `就不会觉得它们是另一套规则`、`就不会把它们误解成几套彼此割裂的 API`、`我们还要回到一个更朴素...` 这类句子，如果语气像是在纠正读者，就要改得更平缓。

更好的写法是先给上下文，再给结论。例如：

- 不要写：`这样就不会把它们误解成几套彼此割裂的 API。`
- 可以写：`把这些能力放回同一条生命周期链路里看，配置、IoC、运行时和测试之间的关系会更清楚。`

- 不要写：`我们还要回到一个更朴素的问题。`
- 可以写：`这时候可以先回到应用启动本身，看它到底需要哪些前置条件。`

原则是少训诫，多铺陈；少直接否定读者，多把判断放进具体场景里自然推出。

## 转场规则

文章内部转场要自然承接。

上一节结束时，可以顺手带出下一节的问题。例如：

- `绑定成功只说明配置能落到类型上，不代表这个值一定可用。`
- `注册之后还要看本次是否生效。`
- `Logger 选好以后，真正写出日志前还要经过 Appender、Layout 和 Encoder。`

不要突然从一个 API 跳到另一个 API。每次切换都要让读者知道视角为什么移动。

跨篇转场要轻，不要写成单独的“下一篇预告”。结尾可以自然带一句下一篇会继续处理什么问题。

## 结尾规则

结尾不要写空泛总结。

最后一个小节仍然要是判断型标题，例如：

- `配置校验守住启动前置条件`
- `覆盖规则最重要的是确定性`
- `运行流程本身就是排错地图`
- `Logger 的职责只是调度事件`

结尾要先完成本篇主题收束，再自然带到下一篇。收束时要说明这块能力在 Go-Spring 整体模型中的位置和价值。

禁止使用下面这类结尾。

- `本文介绍了...希望对你有帮助。`
- `以上就是本篇全部内容。`
- `下一篇我们继续讲...敬请期待。`
- `如果你觉得不错，欢迎点赞转发。`

## 语言风格

语言要克制、直接、工程化。

常用表达包括：

- `通常`
- `更适合`
- `可以先`
- `这时候`
- `也就是说`
- `反过来说`
- `边界清楚后`
- `最终要服务于`

少用绝对表达。不要写“必须永远”“一定最好”“唯一正确”。如果确实是强约束，要说明约束来自哪里。

不要使用营销化、口号化或情绪化表达。

禁止使用下面这类表达。

- `非常强大`
- `极致优雅`
- `轻松拿捏`
- `保姆级`
- `闭眼用`
- `史上最全`
- `灵魂拷问`

口语化润色时，要优先补顺句子的过程感，而不是把句子变得随意。可以把偏硬的书面表达改得更自然，例如：

- `Go-Spring 提供两种方式` 可以改成 `Go-Spring 提供了两种方式`。
- `配置进入系统后转换成 Properties` 可以改成 `配置进入系统后，会先转换成 Properties`。
- `Reader 负责解析配置格式` 可以保留；如果前后句较硬，可以改成 `这里可以先把 Reader 理解成解析配置格式的入口`。
- `错误暴露在启动阶段` 可以改成 `错误最好在启动阶段就暴露出来`。

但不要为了口语化牺牲准确性。API 名称、配置项、生命周期顺序、错误语义和边界判断不能被改软。

## 公众号发布评分标准

评估 `01` 之后的正文文章时，可以按 100 分制打分。评分不是为了追求文采，而是为了判断文章能不能发布到微信公众号，以及是否符合这一系列的知识分享目标。

建议维度如下。

| 维度 | 分值 | 评分重点 |
|------|------|----------|
| 技术准确性 | 25 | API、配置项、生命周期、边界和示例是否准确，没有为了表达顺滑改变技术含义 |
| 可读性 | 20 | 开头是否能带入问题，段落是否顺，代码前后是否解释充分，读者是否能一路读下去 |
| 亲民性 | 15 | 语气是否自然，有没有适当的因果、转折、递进和少量语气助词，是否避免了生硬说教 |
| 体系连贯性 | 15 | 是否能承接上一篇、收束本篇、自然带到下一篇，是否能放回 Go-Spring 整体能力链路 |
| Go-Spring 特点 | 15 | 是否讲清 Go-Spring 的身份、能力边界和设计取舍，而不是写成普通 Go 技术文章 |
| 示例可读性 | 10 | 示例是否必要、简洁、准确，代码上方是否有足够铺垫，代码下方是否解释选择理由 |

评分解释可以按下面标准使用。

- `90-100`：可以发布，只需要少量字句润色。
- `80-89`：主体可用，但需要补过渡、示例说明或局部语气调整。
- `70-79`：技术内容基本可用，但公众号阅读体验不足，需要重写部分段落。
- `70 以下`：结构、准确性或表达风格存在明显问题，不建议直接发布。

每篇文章评分时，除了给总分，还要列出扣分最多的两个维度。这样后续修改时能优先处理真正影响发布质量的问题。

## 禁止项

写作时必须避免下面这些问题。

- 不要写成零基础教程。
- 不要大量解释 Go 基础语法、HTTP 基础概念、依赖注入基础概念。
- 不要写成 API 参考手册。
- 不要把多个独立能力合并成一篇大章。
- 不要用标题党。
- 不要用公众号套话开头或结尾。
- 不要为了显得完整而展开无关背景。
- 不要只展示代码，不解释工程语义。
- 不要只讲优点，不讲边界。
- 不要把“下一篇预告”写成单独章节。
- 不要让标题只有态度没有对象。
- 不要在解释句里密集使用冒号。
- 不要让相邻段落缺少因果、转折或递进关系。
- 不要用说教式句子替代场景化引导。
- 不要连续使用代词而不补出 Go-Spring 或具体能力名称。

## 写作自检清单

生成文章后，按下面清单检查。

1. 标题是否同时包含主题和具体问题。
2. 开头是否从工程问题进入，而不是列提纲。
3. 每个小节标题是否带有明确判断。
4. 正文是否按问题、抽象、代码、边界推进。
5. 相邻段落之间是否有必要的因果、转折或递进。
6. 代码示例前是否有足够铺垫，代码示例后是否解释了选择理由。
7. 是否避免了零基础科普和 API 清单式写法。
8. 是否讲清楚适用场景和不适合的场景。
9. 是否在关键位置补出了 Go-Spring 或具体能力名称。
10. 是否避免了说教式表达，改成场景化引导。
11. 是否减少了不必要的冒号解释，改用 `即`、破折号或自然拆句。
12. 结尾是否先收束本篇主题，再自然过渡。
13. 是否没有使用营销化、口号化、夸张表达。
14. 读完后是否能看出这篇文章在 Go-Spring 整体能力链路中的位置。

一句话概括：这套文章的风格不是教程腔，也不是宣传稿，而是从真实工程复杂性出发，把一个能力的抽象、代码入口、适用边界和设计取舍讲清楚。
