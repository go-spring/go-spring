# 2.guides 写作风格推理过程

本文记录 `go-spring/docs/2.guides` 目录下 guide 文档写作风格的推理过程。
规则结论见 [writing-style-conclusions.md](writing-style-conclusions.md)。

分析范围排除 `08-http-gen.md`，仅覆盖以下文档：

- `01-configuration.md`
- `02-ioc-container.md`
- `03-app-start-stop.md`
- `04-logging.md`
- `05-http-server.md`
- `06-components.md`
- `07-testing.md`

## 分析过程

### 共同写作风格抽象

这些文档整体采用的是**面向 Go 开发者的解释型教程风格**，不是单纯的 API Reference。
共同特点可以抽象为以下几类。

#### 1. 先建立心智模型，再展开 API 用法

文档通常不会直接罗列函数签名，而是先解释这个能力解决什么问题、在 Go-Spring 中处于什么位置，
再进入具体 API、配置项或代码示例。

典型结构是：

1. 说明能力定位。
2. 解释核心概念或设计背景。
3. 给出最小可用示例。
4. 展开规则、配置、扩展点和边界行为。

这种写法让读者先理解“为什么这样设计”，再学习“怎么使用”。

#### 2. 内容推进遵循“概念 -> 示例 -> 规则 -> 建议”

大多数章节都遵循类似的局部结构：

- 先用自然语言解释概念。
- 再给 Go 代码、配置文件或流程示意。
- 然后说明语法规则、匹配规则、优先级、生命周期或限制。
- 最后补充推荐做法、适用场景或注意事项。

这种结构在配置、IoC、启动、日志、HTTP Server、测试文档中都很常见。

#### 3. 强调工程判断，而不是只描述功能存在

文档经常告诉读者应该如何选择：

- 哪种方式是推荐入口。
- 哪种场景适合使用某个 API。
- 哪些能力只是可用，但不应滥用。
- 哪些行为是设计决策，而不是偶然实现。

这类表述常以“推荐”“建议”“适用于”“需要注意”“不建议”“优先使用”等词出现。

#### 4. 示例偏完整场景，而不是孤立代码片段

代码示例通常会包含较完整的上下文，例如：

- `package main`
- `import`
- `init`
- `main`
- 结构体定义
- 构造函数
- 配置片段
- 调用方式

这使示例更接近真实项目中的使用方式，降低读者从文档迁移到代码的成本。

#### 5. 使用 Markdown 结构化降低理解成本

常见表达方式包括：

- 标题层级拆分主题。
- 列表表达能力分类、流程步骤和推荐实践。
- 表格表达配置项、类型、级别、优先级。
- 代码块表达 Go、YAML、properties、shell、文本输出。
- 引用块表达提示、注意事项或补充说明。
- 加粗强调关键设计、推荐项或限制。

整体上，文档倾向于把规则显性化，而不是藏在长段落里。

#### 6. 语气偏解释型和陪伴型

文档大量使用“我们”“你”“此时”“因此”等表达，让文本更像教程讲解。
这种语气降低了框架概念的进入门槛，尤其适合从 Java Spring 或普通 Go 项目迁移过来的读者。

需要注意的是，这种语气在不同文档中的强度并不一致。

#### 7. 经常和 Go / Spring / 主流生态建立对照

文档会主动说明 Go-Spring 与以下对象的关系：

- Java Spring / Spring Boot
- Go 原生机制
- 标准库
- Wire 等编译期 DI 工具
- Log4j2、zap、zerolog 等日志生态
- Go 原生 `go test`

这种对照帮助读者把新概念放进已有经验中理解。

### 可变写作风格抽象

这些文档虽然有共同取向，但没有强行套用同一个固定模板。
其中一部分差异是合理的：不同主题的复杂度不同，文档深度、原理展开和章节数量也应该随之变化。
真正需要关注的不是“每篇是否一样长”，而是“是否在当前主题下解释到足够清楚”。

差异主要体现在以下方面。

#### 1. 入口结构不固定

有的文档有明确的“快速开始”章节，有的没有。

- 有“快速开始”：`02-ioc-container.md`、`04-logging.md`、`05-http-server.md`
- 无独立“快速开始”：`01-configuration.md`、`03-app-start-stop.md`、`06-components.md`、`07-testing.md`

没有“快速开始”的文档通常直接从概念或场景开始推进。
这说明“快速开始”不是当前文档集的通用固定模板，而是和文档类型有关：
面向接入和使用的文档更常提供快速开始，偏概念、机制或策略说明的文档更常先建立心智模型。

#### 2. 篇幅和信息密度按主题复杂度变化

文档深度不在同一层级，但这本身不是问题。
复杂领域需要更多原理解释，简单能力则只需要讲清楚使用方式和少量边界条件。

具体表现为：

- `02-ioc-container.md` 和 `04-logging.md` 是体系化长文，覆盖概念、用法、原理、FAQ、扩展点。
- `01-configuration.md` 是中长篇，覆盖配置系统的主要能力，但语气更教程化。
- `03-app-start-stop.md` 是机制型文档，围绕启动、关闭、Runner、Server、Context 等生命周期展开。
- `05-http-server.md` 和 `06-components.md` 更像短 guide，聚焦常用能力。
- `07-testing.md` 是实践型文档，围绕测试分类、断言和 Mock 展开。

这种差异说明文档没有机械追求统一篇幅，而是根据主题本身决定解释深度。
这是合理的写法，也是后续规则部分的判断基础。

#### 3. 口语化程度不一致

`01-configuration.md` 最口语化，会出现较多面向读者的直接表达，例如“你只需要”“是不是一目了然”“框架帮你处理好了”。

`03-app-start-stop.md`、`05-http-server.md`、`06-components.md`、`07-testing.md` 相对克制，更像工程说明文。

`02-ioc-container.md` 介于两者之间，前半部分教程化较强，后半部分更偏体系化说明。

`04-logging.md` 则更像产品级技术设计文档，强调架构分层、配置模型、性能和生产实践。

#### 4. 视觉标记和提示块不统一

部分文档使用 emoji、对错符号、引用提示或分隔线：

- `01-configuration.md` 使用提示块和 emoji。
- `02-ioc-container.md` 使用对错符号标记对比示例。
- `04-logging.md` 使用大量分隔线、表格和结构图。
- `07-testing.md` 使用提示块说明测试依赖注入的限制。

但 `03-app-start-stop.md`、`05-http-server.md`、`06-components.md` 基本没有这些视觉标记。

这说明现有 guide 的视觉表达还没有完全收敛。
这类差异对内容理解影响不大，但需要形成统一约定：提示类视觉元素可以保留少量标准化用法，但不应滥用，也不应替代文字说明。

#### 5. FAQ 不是通用结构

只有 `02-ioc-container.md` 有明显的“常见问题”章节。
其他文档即使包含边界解释，也通常散落在相关章节内，而不是集中成 FAQ。
这说明 FAQ 不是通用必选章节，而是主要出现在复杂主题中。
短文档或简单能力通常把边界解释放在对应章节内；如果有少量高频问题，也可以设置简短 FAQ 小节。

#### 6. 原理展开程度按主题需要变化

复杂主题会深入解释内部机制：

- `02-ioc-container.md` 有容器运行流程、核心设计、循环依赖、销毁顺序等内容。
- `04-logging.md` 有日志分层架构、插件注入、生命周期、刷新和适配。
- `03-app-start-stop.md` 有完整启动流程和关闭流程。

而 `05-http-server.md`、`06-components.md` 更偏“如何使用”，原理解释较浅。
这种处理符合主题复杂度：复杂机制需要原理支撑，简单能力只要让读者知道如何接入和配置即可。

#### 7. 标题命名方式存在差异

标题有几种不同风格：

- 名词型：`Bean 定义`、`配置绑定`、`日志级别`
- 动作型：`设置默认配置`、`替换依赖`、`实现 Server`
- 场景型：`为什么需要 IoC 容器？`、`选择哪种注入方式`
- 机制型：`路由机制`、`核心机制`、`运行流程`

这些标题都可读，但尚未形成统一命名约定。
这说明标题命名有收敛空间，尤其是同类章节可以减少命名漂移。

### 逐篇遵守情况

| 文档 | 共同风格遵守度 | 主要特点 | 可变风格表现 |
|------|----------------|----------|----------------|
| `01-configuration.md` | 高 | 教程感强，概念、示例、规则、优先级讲解完整 | 口语化最强，没有独立快速开始，提示块风格与其他文档不完全一致 |
| `02-ioc-container.md` | 高 | 最完整的教程型长文，包含背景、快速开始、用法、原理、FAQ | 领域复杂，因此展开到原理、设计取舍和 FAQ；对错符号不是通用结构 |
| `03-app-start-stop.md` | 中高 | 机制说明清晰，围绕生命周期和运行流程展开 | 没有快速开始，语气更克制，视觉提示较少 |
| `04-logging.md` | 高 | 架构分层、配置表格、生产实践和扩展点最充分 | 日志系统复杂且面向生产实践，因此更像设计文档；分隔线和大量表格不是其他文档的通用模式 |
| `05-http-server.md` | 中 | 短而直接，定位、快速开始、配置项、路由集成完整 | 主题相对收敛，因此以使用方式和集成为主，原理展开较少是合理的 |
| `06-components.md` | 中 | 聚焦 Starter 机制，结构简洁，示例直接 | 主题较集中，因此用短 guide 说明机制和约定即可 |
| `07-testing.md` | 中高 | 实践导向明显，区分单元测试、容器测试、断言和 Mock | 没有快速开始，更多是测试策略说明，不是框架核心机制说明 |

## 大型项目实践补充观察

仅从现有 `2.guides` 文档中抽象风格还不够。
如果 Go-Spring 文档要长期服务多个版本、复杂功能和生产使用场景，还需要吸收大型项目文档中的治理规则。

这些规则不是现有文档中已经稳定体现出来的风格，而是后续应补齐的能力：

- **版本与状态标记**：功能从哪个版本开始支持、是否实验性、是否已弃用、是否计划移除，需要在文档中明确说明。
- **前置条件说明**：读者在进入正文前应知道需要先了解哪些概念、完成哪些安装或配置。
- **故障排查章节**：复杂能力和生产常用能力应集中列出常见错误、原因和处理方式。
- **交叉引用规范**：相关概念、相关 API、相关 Guide 应在首次出现或章节末尾建立链接。
- **渐进式展开约定**：文档应先讲基础路径，再讲高级能力，最后讲生产实践，避免初级读者一开始被高级概念淹没。
- **可观察性说明**：生产级文档应说明如何验证功能正常工作、如何开启相关日志、如何观察关键状态。
- **统一术语表引用**：核心术语首次出现时，应链接到统一定义或所属概念文档。

这些补充观察已沉淀到 [writing-style-conclusions.md](writing-style-conclusions.md) 的规则中。
