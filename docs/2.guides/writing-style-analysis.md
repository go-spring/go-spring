# 2.guides 写作风格分析

本文分析 `go-spring/docs/2.guides` 目录下 guide 文档的写作风格。

分析范围排除 `08-http-gen.md`，仅覆盖以下文档：

- `01-configuration.md`
- `02-ioc-container.md`
- `03-app-start-stop.md`
- `04-logging.md`
- `05-http-server.md`
- `06-components.md`
- `07-testing.md`

本文分为两部分：

- 第一部分保留对现有 guide 的观察与分析过程。
- 第二部分沉淀已经认可、后续可以执行的写作规则和结论。

## 第一部分：分析过程

### 共同写作风格抽象

这些文档整体采用的是**面向 Go 开发者的解释型教程风格**，不是单纯的 API Reference。
共同特点可以抽象为以下几类。

#### 1. 先建立心智模型，再展开 API 用法

文档通常不会直接罗列函数签名，而是先解释这个能力解决什么问题、在 Go-Spring 中处于什么位置，
再进入具体 API、配置项或代码示例。

典型结构是：

1. 说明能力定位。
2. 解释核心概念或设计背景。
3. 给出最小可用示例。
4. 展开规则、配置、扩展点和边界行为。

这种写法让读者先理解“为什么这样设计”，再学习“怎么使用”。

#### 2. 内容推进遵循“概念 -> 示例 -> 规则 -> 建议”

大多数章节都遵循类似的局部结构：

- 先用自然语言解释概念。
- 再给 Go 代码、配置文件或流程示意。
- 然后说明语法规则、匹配规则、优先级、生命周期或限制。
- 最后补充推荐做法、适用场景或注意事项。

这种结构在配置、IoC、启动、日志、HTTP Server、测试文档中都很常见。

#### 3. 强调工程判断，而不是只描述功能存在

文档经常告诉读者应该如何选择：

- 哪种方式是推荐入口。
- 哪种场景适合使用某个 API。
- 哪些能力只是可用，但不应滥用。
- 哪些行为是设计决策，而不是偶然实现。

这类表述常以“推荐”“建议”“适用于”“需要注意”“不建议”“优先使用”等词出现。

#### 4. 示例偏完整场景，而不是孤立代码片段

代码示例通常会包含较完整的上下文，例如：

- `package main`
- `import`
- `init`
- `main`
- 结构体定义
- 构造函数
- 配置片段
- 调用方式

这使示例更接近真实项目中的使用方式，降低读者从文档迁移到代码的成本。

#### 5. 使用 Markdown 结构化降低理解成本

常见表达方式包括：

- 标题层级拆分主题。
- 列表表达能力分类、流程步骤和推荐实践。
- 表格表达配置项、类型、级别、优先级。
- 代码块表达 Go、YAML、properties、shell、文本输出。
- 引用块表达提示、注意事项或补充说明。
- 加粗强调关键设计、推荐项或限制。

整体上，文档倾向于把规则显性化，而不是藏在长段落里。

#### 6. 语气偏解释型和陪伴型

文档大量使用“我们”“你”“此时”“因此”等表达，让文本更像教程讲解。
这种语气降低了框架概念的进入门槛，尤其适合从 Java Spring 或普通 Go 项目迁移过来的读者。

需要注意的是，这种语气在不同文档中的强度并不一致。

#### 7. 经常和 Go / Spring / 主流生态建立对照

文档会主动说明 Go-Spring 与以下对象的关系：

- Java Spring / Spring Boot
- Go 原生机制
- 标准库
- Wire 等编译期 DI 工具
- Log4j2、zap、zerolog 等日志生态
- Go 原生 `go test`

这种对照帮助读者把新概念放进已有经验中理解。

### 可变写作风格抽象

这些文档虽然有共同取向，但没有强行套用同一个固定模板。
其中一部分差异是合理的：不同主题的复杂度不同，文档深度、原理展开和章节数量也应该随之变化。
真正需要关注的不是“每篇是否一样长”，而是“是否在当前主题下解释到足够清楚”。

差异主要体现在以下方面。

#### 1. 入口结构不固定

有的文档有明确的“快速开始”章节，有的没有。

- 有“快速开始”：`02-ioc-container.md`、`04-logging.md`、`05-http-server.md`
- 无独立“快速开始”：`01-configuration.md`、`03-app-start-stop.md`、`06-components.md`、`07-testing.md`

没有“快速开始”的文档通常直接从概念或场景开始推进。
这说明“快速开始”不是当前文档集的通用固定模板，而是和文档类型有关：
面向接入和使用的文档更常提供快速开始，偏概念、机制或策略说明的文档更常先建立心智模型。

#### 2. 篇幅和信息密度按主题复杂度变化

文档深度不在同一层级，但这本身不是问题。
复杂领域需要更多原理解释，简单能力则只需要讲清楚使用方式和少量边界条件。

具体表现为：

- `02-ioc-container.md` 和 `04-logging.md` 是体系化长文，覆盖概念、用法、原理、FAQ、扩展点。
- `01-configuration.md` 是中长篇，覆盖配置系统的主要能力，但语气更教程化。
- `03-app-start-stop.md` 是机制型文档，围绕启动、关闭、Runner、Server、Context 等生命周期展开。
- `05-http-server.md` 和 `06-components.md` 更像短 guide，聚焦常用能力。
- `07-testing.md` 是实践型文档，围绕测试分类、断言和 Mock 展开。

这种差异说明文档没有机械追求统一篇幅，而是根据主题本身决定解释深度。
这是合理的写法，也是后续规则部分的判断基础。

#### 3. 口语化程度不一致

`01-configuration.md` 最口语化，会出现较多面向读者的直接表达，例如“你只需要”“是不是一目了然”“框架帮你处理好了”。

`03-app-start-stop.md`、`05-http-server.md`、`06-components.md`、`07-testing.md` 相对克制，更像工程说明文。

`02-ioc-container.md` 介于两者之间，前半部分教程化较强，后半部分更偏体系化说明。

`04-logging.md` 则更像产品级技术设计文档，强调架构分层、配置模型、性能和生产实践。

#### 4. 视觉标记和提示块不统一

部分文档使用 emoji、对错符号、引用提示或分隔线：

- `01-configuration.md` 使用提示块和 emoji。
- `02-ioc-container.md` 使用对错符号标记对比示例。
- `04-logging.md` 使用大量分隔线、表格和结构图。
- `07-testing.md` 使用提示块说明测试依赖注入的限制。

但 `03-app-start-stop.md`、`05-http-server.md`、`06-components.md` 基本没有这些视觉标记。

这说明现有 guide 的视觉表达还没有完全收敛。
这类差异对内容理解影响不大，但会影响整套文档的稳定感。

#### 5. FAQ 不是通用结构

只有 `02-ioc-container.md` 有明显的“常见问题”章节。
其他文档即使包含边界解释，也通常散落在相关章节内，而不是集中成 FAQ。
这说明 FAQ 不是通用章节，而是只在复杂主题中出现。
短文档或简单能力通常把边界解释放在对应章节内。

#### 6. 原理展开程度按主题需要变化

复杂主题会深入解释内部机制：

- `02-ioc-container.md` 有容器运行流程、核心设计、循环依赖、销毁顺序等内容。
- `04-logging.md` 有日志分层架构、插件注入、生命周期、刷新和适配。
- `03-app-start-stop.md` 有完整启动流程和关闭流程。

而 `05-http-server.md`、`06-components.md` 更偏“如何使用”，原理解释较浅。
这种处理符合主题复杂度：复杂机制需要原理支撑，简单能力只要让读者知道如何接入和配置即可。

#### 7. 标题命名方式存在差异

标题有几种不同风格：

- 名词型：`Bean 定义`、`配置绑定`、`日志级别`
- 动作型：`设置默认配置`、`替换依赖`、`实现 Server`
- 场景型：`为什么需要 IoC 容器？`、`选择哪种注入方式`
- 机制型：`路由机制`、`核心机制`、`运行流程`

这些标题都可读，但尚未形成统一命名约定。
这说明标题命名有收敛空间，尤其是同类章节可以减少命名漂移。

### 逐篇遵守情况

| 文档 | 共同风格遵守度 | 主要特点 | 可变风格表现 |
|------|----------------|----------|----------------|
| `01-configuration.md` | 高 | 教程感强，概念、示例、规则、优先级讲解完整 | 口语化最强，没有独立快速开始，提示块风格与其他文档不完全一致 |
| `02-ioc-container.md` | 高 | 最完整的教程型长文，包含背景、快速开始、用法、原理、FAQ | 领域复杂，因此展开到原理、设计取舍和 FAQ；对错符号不是通用结构 |
| `03-app-start-stop.md` | 中高 | 机制说明清晰，围绕生命周期和运行流程展开 | 没有快速开始，语气更克制，视觉提示较少 |
| `04-logging.md` | 高 | 架构分层、配置表格、生产实践和扩展点最充分 | 日志系统复杂且面向生产实践，因此更像设计文档；分隔线和大量表格不是其他文档的通用模式 |
| `05-http-server.md` | 中 | 短而直接，定位、快速开始、配置项、路由集成完整 | 主题相对收敛，因此以使用方式和集成为主，原理展开较少是合理的 |
| `06-components.md` | 中 | 聚焦 Starter 机制，结构简洁，示例直接 | 主题较集中，因此用短 guide 说明机制和约定即可 |
| `07-testing.md` | 中高 | 实践导向明显，区分单元测试、容器测试、断言和 Mock | 没有快速开始，更多是测试策略说明，不是框架核心机制说明 |

## 第二部分：认可的规则和结论

以下规则是从前面的分析和讨论中沉淀出来的结论。
后续新增或修改 `2.guides` 文档时，应优先遵守这一部分。

### 规则等级

| 等级 | 含义 |
|------|------|
| 必须 | 默认必须遵守。违反时需要有明确理由，并在评审中说明。 |
| 应当 | 推荐遵守。特殊主题可以调整，但应保持同类文档一致。 |
| 可以 | 可选规则。只在主题需要时使用，不为了套模板而添加。 |

规则 ID 前缀含义如下：

| 前缀 | 含义 |
|------|------|
| `GEN` | 总体风格规则 |
| `ORG` | 内容组织规则 |
| `DEP` | 深度控制规则 |
| `TITLE` | 标题命名规则 |
| `EXAMPLE` | 示例规则 |
| `EXP` | 表达规则 |
| `VIS` | 视觉风格规则 |

### 总体规则

| ID | 等级 | 规则 |
|----|------|------|
| GEN-01 | 必须 | `2.guides` 的目标风格是解释型、场景化、示例驱动、带工程判断的中文技术教程。 |
| GEN-02 | 必须 | 文档深度服务于主题复杂度，不以篇幅一致作为目标。 |
| GEN-03 | 应当 | 真正需要统一的是风格边界、标题命名、示例质量和规则表达方式。 |

可以这样写：

```md
# 内置 HTTP Server

Go-Spring 内置 HTTP Server，适用于在应用中直接暴露 HTTP 接口的场景。
它基于标准库 `net/http` 构建，默认随应用一同启动，并统一纳入 Go-Spring 的生命周期管理。
```

不建议这样写：

```md
# HTTP

这里介绍 HTTP 功能。
```

原因：前者说明了能力定位、适用场景和设计边界；后者缺少读者建立心智模型所需的信息。

### 内容组织规则

| ID | 等级 | 规则 |
|----|------|------|
| ORG-01 | 必须 | 文档开篇说明能力定位、适用场景和核心价值。 |
| ORG-02 | 应当 | 面向接入和使用的文档提供 `## 快速开始`，给出最小可运行示例。 |
| ORG-03 | 可以 | 偏概念、机制或策略说明的文档可以省略“快速开始”，但应先建立清晰的心智模型。 |
| ORG-04 | 可以 | 复杂主题可以包含 `核心概念`、`核心机制`、`生命周期`、`运行流程`、`常见问题` 等章节。 |
| ORG-05 | 必须 | 短文档至少包含适用场景、基本用法、配置或扩展点。 |
| ORG-06 | 应当 | FAQ 只用于概念复杂、容易误用或存在明显设计取舍的主题。 |

可以这样写：

````md
## 快速开始

下面的示例展示如何注册一个 HTTP 路由，并让 Go-Spring 接管内置 HTTP Server。

```go
func init() {
	http.HandleFunc("/hello", func(w http.ResponseWriter, r *http.Request) {
		w.Write([]byte("Hello, Go-Spring!"))
	})
}
```
````

不建议这样写：

````md
## 快速开始

用法如下。

```go
http.HandleFunc("/hello", handler)
```
````

原因：快速开始应能让读者理解示例发生在哪个上下文中，而不是只看到孤立调用。

### 深度控制规则

| ID | 等级 | 规则 |
|----|------|------|
| DEP-01 | 必须 | 不为了统一篇幅而增加或删除内容。 |
| DEP-02 | 应当 | IoC、日志、启动生命周期、配置优先级等复杂主题应解释必要原理、流程、生命周期和设计取舍。 |
| DEP-03 | 应当 | 简单能力聚焦何时使用、如何接入、如何配置、关键边界，不做过度原理展开。 |

可以这样写：

```md
Go-Spring 采用按需创建机制。默认情况下，只有 root Bean 会作为依赖注入的入口被触发。
因此，如果某个 Bean 没有被 root Bean 间接依赖，也没有显式标记为 root，它不会被自动创建。
```

不建议这样写：

```md
Go-Spring 会创建 Bean。
```

原因：对于 IoC 这类复杂主题，只给结论不够，必须解释触发条件和使用影响。

对于简单能力，可以这样写：

````md
如果需要关闭内置 HTTP Server，可以设置：

```properties
spring.http.server.enabled=false
```
````

不建议为了简单能力强行展开内部实现细节。

### 标题命名规则

| ID | 等级 | 规则 |
|----|------|------|
| TITLE-01 | 应当 | 标题命名收敛为“名词型 + 必要场景型”。 |
| TITLE-02 | 应当 | 优先使用稳定的名词型标题，例如 `配置项`、`生命周期`、`路由机制`、`核心概念`、`使用方式`。 |
| TITLE-03 | 可以 | 当读者需要做选择、理解原因或避坑时，可以使用场景型标题，例如 `为什么需要 IoC 容器？`、`选择哪种注入方式`。 |
| TITLE-04 | 应当 | 同类章节使用同类标题，减少命名漂移。 |

可以这样写：

```md
## 配置项
## 生命周期
## 路由机制
## 选择哪种注入方式
```

不建议这样写：

```md
## 都有哪些配置可以调
## 它什么时候开始什么时候结束
## 路由到底是怎么回事
## 我应该用哪个
```

原因：前者稳定、可扫描；后者口语化过强，同类章节命名不稳定。

### 示例规则

| ID | 等级 | 规则 |
|----|------|------|
| EXAMPLE-01 | 应当 | 示例尽量接近真实项目，而不是只展示孤立函数。 |
| EXAMPLE-02 | 应当 | Go 示例优先包含必要上下文，例如 `package main`、`import`、结构体定义、构造函数、`init` 或 `main`。 |
| EXAMPLE-03 | 应当 | 配置示例标明格式，例如 YAML、properties、TOML 或 JSON。 |
| EXAMPLE-04 | 必须 | 示例之后解释关键点，尤其是隐含规则、默认行为、优先级、生命周期和错误边界。 |

可以这样写：

````md
```yaml
spring:
  http:
    server:
      addr: ":8080"
```

上面的配置会把内置 HTTP Server 的监听地址从默认的 `:9090` 改为 `:8080`。
````

不建议这样写：

````md
```yaml
addr: ":8080"
```
````

原因：后者缺少完整配置路径，也没有说明默认值和生效对象。

### 表达规则

| ID | 等级 | 规则 |
|----|------|------|
| EXP-01 | 应当 | 先解释为什么，再说明怎么做。 |
| EXP-02 | 应当 | 存在多个选择时，给出推荐路径和选择依据。 |
| EXP-03 | 必须 | 限制、边界行为和不推荐用法要显式说明。 |
| EXP-04 | 应当 | 配置项、优先级、类型支持、策略对比等结构化信息优先使用表格。 |
| EXP-05 | 可以 | 流程、生命周期和执行顺序可以使用列表或文本流程图。 |
| EXP-06 | 应当 | 语气保持亲和但稳定，避免过度口语化。 |

可以这样写：

```md
对于大多数服务端应用，建议优先使用 `gs.Run()`。
它会完成配置加载、日志初始化、容器启动、Server 启动和退出信号监听，适合作为独立服务的标准入口。
```

不建议这样写：

```md
你直接用 `gs.Run()` 就行了，别想太多。
```

原因：前者给出了推荐路径和选择依据；后者口语化过强，也没有说明适用边界。

### 视觉风格规则

| ID | 等级 | 规则 |
|----|------|------|
| VIS-01 | 必须 | 不使用 emoji、对错符号和横向分隔线作为常规表达手段。 |
| VIS-02 | 应当 | 提示、警告或补充说明优先使用朴素引用块、加粗关键句或小标题。 |
| VIS-03 | 应当 | 对比推荐和不推荐写法时，优先用小标题、注释或表格说明，不依赖视觉符号。 |
| VIS-04 | 必须 | 整体视觉风格保持克制、稳定、可维护。 |

可以这样写：

````md
推荐写法：

```go
gs.Provide(NewUserService)
```

不推荐写法：

```go
var service = NewUserService()
```
````

不建议这样写：

````md
对号符号 推荐：
```go
gs.Provide(NewUserService)
```

叉号符号 不推荐：
```go
var service = NewUserService()
```
````

原因：前者不依赖视觉符号，也更符合整套 guide 的克制风格。
