# 2.guides 写作规范结论

本文保存 `go-spring/docs/2.guides` 文档写作风格分析后沉淀出的规则和结论。
推理过程见 [writing-style-analysis.md](writing-style-analysis.md)。

后续新增或修改 `2.guides` 文档时，应优先遵守本文规则。

## 规则等级

| 等级 | 含义 |
|------|------|
| 必须 | 默认必须遵守。违反时需要有明确理由，并在评审中说明。 |
| 应当 | 推荐遵守。特殊主题可以调整，但应保持同类文档一致。 |
| 可以 | 可选规则。只在主题需要时使用，不为了套模板而添加。 |

规则 ID 前缀含义如下：

| 前缀 | 含义 |
|------|------|
| `GEN` | 总体风格规则 |
| `VER` | 版本与状态规则 |
| `ORG` | 内容组织规则 |
| `DEP` | 深度控制规则 |
| `TITLE` | 标题命名规则 |
| `EXAMPLE` | 示例规则 |
| `EXP` | 表达规则 |
| `ECO` | 生态对照规则 |
| `VIS` | 视觉风格规则 |

## 总体规则

| ID | 等级 | 规则 |
|----|------|------|
| GEN-01 | 必须 | `2.guides` 的目标风格是解释型、场景化、示例驱动、带工程判断的中文技术教程。 |
| GEN-02 | 必须 | 文档深度服务于主题复杂度，不以篇幅一致作为目标。 |
| GEN-03 | 应当 | 真正需要统一的是风格边界、标题命名、示例质量和规则表达方式。 |

可以这样写：

```md
# 内置 HTTP Server

Go-Spring 内置 HTTP Server，适用于在应用中直接暴露 HTTP 接口的场景。
它基于标准库 `net/http` 构建，默认随应用一同启动，并统一纳入 Go-Spring 的生命周期管理。
```

不建议这样写：

```md
# HTTP

这里介绍 HTTP 功能。
```

原因：前者说明了能力定位、适用场景和设计边界；后者缺少读者建立心智模型所需的信息。

## 版本与状态规则

| ID | 等级 | 规则 |
|----|------|------|
| VER-01 | 必须 | 当功能存在版本差异、状态差异或迁移风险时，必须标注版本与状态。 |
| VER-02 | 必须 | 实验性、已弃用、计划移除、行为变更类信息必须靠近功能首次出现的位置。 |
| VER-03 | 应当 | 版本标记使用稳定格式，例如“从 v1.2 开始支持”“实验性功能”“已弃用，将在 v2.0 移除”。 |
| VER-04 | 应当 | 如果功能替代了旧用法，应说明旧用法状态、推荐迁移路径和兼容边界。 |

可以这样写：

```md
## 动态配置

从 v1.2 开始支持。

动态配置适用于不重启应用即可刷新配置值的场景。需要注意的是，动态刷新仅适用于使用 `gs.Dync[T]` 声明的字段。
```

```md
## 旧版配置刷新接口

已弃用，将在 v2.0 移除。新代码应使用 `gs.RefreshProperties()`。
旧接口在 v1.x 中仍保持兼容，但不会再增加新能力。
```

不建议这样写：

```md
## 动态配置

动态配置可以刷新配置。
```

原因：后者没有说明功能从哪个版本开始可用，也没有说明状态和兼容边界。对需要维护多个版本的框架文档来说，这会直接影响读者判断。

## 内容组织规则

| ID | 等级 | 规则 |
|----|------|------|
| ORG-01 | 必须 | 文档开篇说明能力定位、适用场景和核心价值。 |
| ORG-02 | 应当 | 面向接入和使用的文档提供 `## 快速开始`，给出最小可运行示例。 |
| ORG-03 | 可以 | 偏概念、机制或策略说明的文档可以省略“快速开始”，但应先建立清晰的心智模型。 |
| ORG-04 | 可以 | 复杂主题可以包含 `核心概念`、`核心机制`、`生命周期`、`运行流程`、`常见问题` 等章节。 |
| ORG-05 | 必须 | 短文档至少包含适用场景、基本用法、配置或扩展点。 |
| ORG-06 | 应当 | FAQ 优先用于复杂主题；短文档如有 2-3 个高频问题，也可以设置 FAQ 小节。 |
| ORG-07 | 应当 | 需要读者具备前置知识、依赖环境或先完成配置时，应在开篇说明前置条件。 |
| ORG-08 | 应当 | 复杂能力、生产常用能力或容易报错的能力，应提供故障排查章节或等价内容。 |
| ORG-09 | 应当 | 文档应按“基础用法 -> 高级用法 -> 生产实践”渐进展开，避免一开始引入过多高级概念。 |

可以这样写：

````md
## 快速开始

下面的示例展示如何注册一个 HTTP 路由，并让 Go-Spring 接管内置 HTTP Server。

```go
func init() {
	http.HandleFunc("/hello", func(w http.ResponseWriter, r *http.Request) {
		w.Write([]byte("Hello, Go-Spring!"))
	})
}
```
````

不建议这样写：

````md
## 快速开始

用法如下。

```go
http.HandleFunc("/hello", handler)
```
````

原因：快速开始应能让读者理解示例发生在哪个上下文中，而不是只看到孤立调用。

前置条件可以这样写：

```md
## 前置条件

阅读本文前，你应已经了解 Go-Spring 的 Bean 注册方式，并完成 `spring-core` 的引入。
如果还不了解 Bean 注册，先阅读 [IoC 容器](02-ioc-container.md)。
```

故障排查可以这样写：

```md
## 故障排查

### 路由没有生效

可能原因：应用中同时注册了自定义 `*gs.HttpServeMux`，内置 Server 不再使用 `http.DefaultServeMux`。

处理方式：确认路由注册到了最终交给 `gs.HttpServeMux` 的 `http.Handler` 上。
```

短文档 FAQ 可以这样写：

```md
## 常见问题

### 为什么修改端口后没有生效？

确认配置项使用的是完整路径 `spring.http.server.addr`，并检查是否被命令行参数或环境变量覆盖。

### 如何关闭内置 HTTP Server？

设置 `spring.http.server.enabled=false`，或者在代码中使用 `gs.Web(false).Run()`。
```

## 深度控制规则

| ID | 等级 | 规则 |
|----|------|------|
| DEP-01 | 必须 | 不为了统一篇幅而增加或删除内容。 |
| DEP-02 | 应当 | IoC、日志、启动生命周期、配置优先级等复杂主题应解释必要原理、流程、生命周期和设计取舍。 |
| DEP-03 | 应当 | 简单能力聚焦何时使用、如何接入、如何配置、关键边界，不做过度原理展开。 |
| DEP-04 | 应当 | 同一主题内先讲基础路径，再讲高级能力，最后讲生产实践和边界行为。 |

可以这样写：

```md
Go-Spring 采用按需创建机制。默认情况下，只有 root Bean 会作为依赖注入的入口被触发。
因此，如果某个 Bean 没有被 root Bean 间接依赖，也没有显式标记为 root，它不会被自动创建。
```

不建议这样写：

```md
Go-Spring 会创建 Bean。
```

原因：对于 IoC 这类复杂主题，只给结论不够，必须解释触发条件和使用影响。

对于简单能力，可以这样写：

````md
如果需要关闭内置 HTTP Server，可以设置：

```properties
spring.http.server.enabled=false
```
````

不建议为了简单能力强行展开内部实现细节。

渐进式展开可以这样写：

```md
## 基础用法

使用 `gs.Run()` 启动应用。

## 自定义启动配置

通过 `gs.Configure()` 设置默认配置和 Root Bean。

## 生产实践

为 Runner 和 Server 设计可取消的上下文，并在关闭阶段释放资源。
```

不建议这样写：

```md
## 启动方式

先介绍 Server 的 Ready 信号、Root Context、动态配置刷新和自定义 Provider，然后再介绍 `gs.Run()`。
```

原因：后者把高级机制放在基础入口之前，容易让初级读者在建立基本使用路径前被复杂概念淹没。


## 标题命名规则

| ID | 等级 | 规则 |
|----|------|------|
| TITLE-01 | 应当 | 标题命名收敛为“名词型 + 必要场景型”。 |
| TITLE-02 | 应当 | 优先使用稳定的名词型标题，例如 `配置项`、`生命周期`、`路由机制`、`核心概念`、`使用方式`。 |
| TITLE-03 | 可以 | 当读者需要做选择、理解原因或避坑时，可以使用场景型标题，例如 `为什么需要 IoC 容器？`、`选择哪种注入方式`。 |
| TITLE-04 | 应当 | 同类章节使用同类标题，减少命名漂移。 |

可以这样写：

```md
## 配置项
## 生命周期
## 路由机制
## 选择哪种注入方式
```

不建议这样写：

```md
## 都有哪些配置可以调
## 它什么时候开始什么时候结束
## 路由到底是怎么回事
## 我应该用哪个
```

原因：前者稳定、可扫描；后者口语化过强，同类章节命名不稳定。

## 示例规则

| ID | 等级 | 规则 |
|----|------|------|
| EXAMPLE-01 | 应当 | 示例尽量接近真实项目，而不是只展示孤立函数。 |
| EXAMPLE-02 | 应当 | Go 示例优先包含必要上下文，例如 `package main`、`import`、结构体定义、构造函数、`init` 或 `main`。 |
| EXAMPLE-03 | 应当 | 配置示例标明格式，例如 YAML、properties、TOML 或 JSON。 |
| EXAMPLE-04 | 必须 | 示例之后解释关键点，尤其是隐含规则、默认行为、优先级、生命周期和错误边界；单行示例且意图完全明确时可以简化。 |

可以这样写：

````md
```yaml
spring:
  http:
    server:
      addr: ":8080"
```

上面的配置会把内置 HTTP Server 的监听地址从默认的 `:9090` 改为 `:8080`。
````

不建议这样写：

````md
```yaml
addr: ":8080"
```
````

原因：后者缺少完整配置路径，也没有说明默认值和生效对象。

单行示例可以简化：

````md
关闭内置 HTTP Server：

```properties
spring.http.server.enabled=false
```
````

原因：这个示例的意图已经由前置句说明，且配置项本身没有额外规则需要展开。

## 表达规则

| ID | 等级 | 规则 |
|----|------|------|
| EXP-01 | 应当 | 先解释为什么，再说明怎么做。 |
| EXP-02 | 应当 | 存在多个选择时，给出推荐路径和选择依据。 |
| EXP-03 | 必须 | 限制、边界行为和不推荐用法要显式说明。 |
| EXP-04 | 应当 | 配置项、优先级、类型支持、策略对比等结构化信息优先使用表格。 |
| EXP-05 | 可以 | 流程、生命周期和执行顺序可以使用列表或文本流程图。 |
| EXP-06 | 应当 | 语气保持亲和但稳定，避免过度口语化。 |
| EXP-07 | 应当 | 相关概念、相关 API、相关 Guide 应在首次出现或章节末尾进行交叉引用。 |
| EXP-08 | 应当 | 核心术语首次出现时，应链接到统一定义或所属概念文档。 |
| EXP-09 | 应当 | 生产相关能力应说明如何验证功能正常工作、如何观察内部状态、如何开启相关日志或排查信号。 |

可以这样写：

```md
对于大多数服务端应用，建议优先使用 `gs.Run()`。
它会完成配置加载、日志初始化、容器启动、Server 启动和退出信号监听，适合作为独立服务的标准入口。
```

不建议这样写：

```md
你直接用 `gs.Run()` 就行了，别想太多。
```

原因：前者给出了推荐路径和选择依据；后者口语化过强，也没有说明适用边界。

交叉引用可以这样写：

```md
Root Bean 是容器创建依赖图的入口。关于 Bean 注册和 root 语义，见 [IoC 容器](02-ioc-container.md)。
```

章节末尾也可以这样写：

```md
## 相关文档

- [IoC 容器](02-ioc-container.md)
- [应用启停与运行机制](03-app-start-stop.md)
```

不建议这样写：

```md
Root Bean 是容器创建依赖图的入口。
```

原因：如果术语是跨文档核心概念，只解释一句但不给出处，会增加读者跳读和补背景的成本。

可观察性说明可以这样写：

```md
## 验证配置是否生效

启动应用后，访问 `http://localhost:8080/hello`。
如果端口配置生效，请求会返回 `Hello, Go-Spring!`。

如果请求失败，先检查启动日志中的 HTTP Server 监听地址，再确认 `spring.http.server.enabled` 没有被设置为 `false`。
```

## 生态对照规则

| ID | 等级 | 规则 |
|----|------|------|
| ECO-01 | 应当 | 当主题和 Go 原生机制、Java Spring、Spring Boot 或主流 Go 生态存在明显对应关系时，应主动建立对照。 |
| ECO-02 | 应当 | 对照应服务于理解和迁移，说明“哪里相似、哪里不同、使用时该怎么判断”。 |
| ECO-03 | 必须 | 不为了显得熟悉而强行类比；没有稳定对应关系时，不做生态对照。 |
| ECO-04 | 应当 | 对照对象优先选择读者已有心智模型，例如 Go 标准库、Go 原生测试、Java Spring、Spring Boot、Wire、zap、zerolog、Log4j2。 |

可以这样写：

```md
如果你熟悉 Spring Boot，可以把 Go-Spring 的配置加载理解为类似 `Environment` 的统一抽象：
无论原始配置来自 YAML、properties、环境变量还是命令行参数，最终都会转换成统一的 key-value 结构。

不同的是，Go-Spring 不做松散绑定。配置 key 是大小写敏感的，必须和代码中的 path 精确匹配。
```

不建议这样写：

```md
Go-Spring 的配置系统和 Spring Boot 一样。
```

原因：前者说明了相似点和差异点，能帮助读者迁移心智模型；后者过度简化，容易让读者误以为行为完全一致。

## 视觉风格规则

| ID | 等级 | 规则 |
|----|------|------|
| VIS-01 | 必须 | 不使用 emoji 作为主要表达方式，不用对错符号和横向分隔线承担结构表达。 |
| VIS-02 | 应当 | 提示、警告或补充说明优先使用朴素引用块、加粗关键句或小标题；必要时可使用 1-2 个标准化提示 emoji。 |
| VIS-03 | 应当 | 对比推荐和不推荐写法时，优先用小标题、注释或表格说明，不依赖视觉符号。 |
| VIS-04 | 必须 | 整体视觉风格保持克制、稳定、可维护。 |

可以这样写：

````md
推荐写法：

```go
gs.Provide(NewUserService)
```

不推荐写法：

```go
var service = NewUserService()
```
````

不建议这样写：

````md
对号符号 推荐：
```go
gs.Provide(NewUserService)
```

叉号符号 不推荐：
```go
var service = NewUserService()
```
````

原因：前者不依赖视觉符号，也更符合整套 guide 的克制风格。

提示块可以这样写：

```md
> Note:
> `spring.http.server.addr` 的默认值是 `:9090`。
```

必要时可以使用标准化提示 emoji，但不应替代文字说明：

```md
> ⚠️ Warning:
> 关闭内置 HTTP Server 后，所有依赖 `*gs.HttpServeMux` 的路由都不会被自动暴露。
```
