# Go-Spring 编码原则与风格规范

本文档定义了 Go-Spring 项目遵循的编码原则、习惯和风格约定。所有贡献者在修改或添加代码时都应遵守这些准则。

---

## 0. 简单优于聪明

- **清晰优先**: 代码优先追求清晰、直观和可维护性，拒绝炫技式技巧性写法
- **避免过度工程**: 不使用过度抽象、隐式行为，复杂性应当来自问题本身，而非实现方式
- **读者友好**: 倾向最直接、最易理解的方案，代码对读者友好，而非对作者友好
- **重写原则**: 任何需要额外解释才能理解的实现，都应考虑用更简单方式重写

## 1. 项目组织与架构

项目采用清晰的**四层依赖结构**，依赖关系从底层到上层单向流动，上层可依赖下层，下层不可反向依赖上层：

- **第一层 `stdlib/`** - 基础工具层，**绝对零依赖**，是整个项目的基石，提供通用工具函数
- **第二层 `spring-core/`** - 核心容器层，Go-Spring 的灵魂，实现 IoC 容器、依赖注入等核心特性
- **第三层 `starter-*/`** - 扩展集成层，基于核心抽象集成第三方服务（Redis、GORM 等），开箱即用
- **第四层 `gs-*/`** - 工程工具层，提供项目初始化、代码生成等开发工具，提升开发效率

设计原则：
- **单一职责**：每个包聚焦一个功能领域，保持包体积小巧且专注，便于测试和维护
- **清晰可见性控制**：严格区分公开 API（`PascalCase`）和内部实现（`camelCase`）
- **测试伴随生产**：测试文件与生产代码放在同一包目录下，测试本身就是活动的文档
- **零值可用**：优先通过默认值、延迟初始化或内部兜底保证零值安全，避免"未初始化即 panic"。

## 2. 命名约定

- **包名**：全小写，简短且具描述性（`errutil`, `assert`, `gs`, `typeutil`），不使用下划线或驼峰命名，包名不要与包内内容重复
- **标识符**：严格遵循 Go 惯例 - 导出公开标识符使用 `PascalCase`，内部标识符使用 `camelCase`
- **常量**：使用 `UPPER_SNAKE_CASE`，与 Go 标准库保持一致
- **错误变量**：预定义错误以 `Err` 前缀开头（`ErrNotFound`, `ErrInvalidConfig`）
- **接口**：单方法接口以 `-er` 后缀结尾（`Handler`, `Provider`, `Server`），大型接口使用描述性名词
- **变量**：简洁而有意义，避免不必要的长名称
- **方法接收者**：简短一致（通常 1-2 个字母），不使用 `me`/`this`/`that` 这类无意义名称

## 3. 代码格式与组织

- **标准 Go 格式**：严格遵循 `gofmt` 格式化
- **许可证头**：每个源文件必须包含统一的 Apache 2.0 许可证头，模板见 `.claude/LICENSE_HEADER`
- **导入分组**：按逻辑有序分组 → `标准库` → `外部依赖` → `内部依赖`，各组之间用空行分隔
- **函数长度**：偏好短小精悍的函数，一个函数只做好一件事
- **行长度**：保持代码适度紧凑，避免过长单行，但也不刻意限制行数
- **空行使用**：使用空行分隔不同逻辑块，避免不必要的密集堆砌
- **代码清理**：过时代码直接删除，不要保留注释掉的死代码

## 4. 错误处理哲学

项目统一使用 `stdlib/errutil`，独创**双语义错误包装模式**：

> **项目规则**: 不使用标准库 `errors.New`/`fmt.Errorf` 等其他方式构造错误，统一通过 `errutil` 包装

- **解释性包装**（`errutil.Explain`）- 添加业务语义：`failed to connect to database: connection refused`
  ```go
  errutil.Explain(err, "failed to connect to database")
  ```
- **路径包装**（`errutil.Stack`）- 跟踪调用链路：`InitService >> LoadConfig >> file not found`
  ```go
  errutil.Stack(err, "InitService")
  ```
- **快错早返回**：业务逻辑错误尽早返回，初始化阶段不可恢复的编程错误直接 panic
- **始终保持解包链**：`errutil` 内部保证 `%w` 语义，完整支持 `errors.Is()`/`errors.As()` 检视原始错误

## 5. 文档与注释

- **包文档**：每个公开包必须有包注释，阐述**设计理念和使用场景**，重点讲"做什么"和"为什么"，不要过多纠缠实现细节，可附简单示例
- **函数文档**：每个导出函数必须有注释：
  - 一句话功能描述
  - 参数、返回值说明
  - 错误和异常情况说明（如有）
  - 复杂场景可附使用示例
- **代码自文档**：使用清晰命名和简洁结构让代码自解释，不添加不必要的注释
- **AI 协作注释**：需要约束 AI 行为时，可添加特殊格式注释明确要求，例如 `// AI: do NOT refactor this function`

## 6. 测试风格

- **工具库优先**：优先使用 `stdlib/testing` 提供的 `assert`/`require` 断言，避免重复造轮子
  只有当不想为测试引入额外依赖时，才直接使用标准库 `testing`
- **子测试分组**：使用 `t.Run()` 对不同测试场景进行逻辑分组
  ```go
  func TestExplain(t *testing.T) {
    t.Run("nil error", func(t *testing.T) { ... })
    t.Run("multiple nested", func(t *testing.T) { ... })
  }
  ```
- **表驱动测试**：多输入输出场景测试推荐使用表驱动模式，减少重复代码
- **混合灵活使用**：表驱动和子测试并不冲突，哪种方案简洁好用就用哪种
- **测试伴随生产**：测试文件与生产代码放在同一个包目录
- **分层断言风格**：提供 `assert`（失败继续执行）和 `require`（失败立即终止）两种选择

## 7. Go 语言习惯

- **原汁原味 Go**：适配 Spring 理念但不强行面向对象，保持 Go 的自然习惯，不生硬套用其他语言范式
- **正确使用 context**：请求流必须携带 `context.Context`，尽量避免随意使用 `context.TODO()`/`context.Background()`
- **边界检查**：在 API 边界做输入参数验证，尽早发现错误
- **避免过度抽象**：只在真正需要时抽象，不为不确定的未来"预规划"，遵循 YAGNI 原则
- **清晰分离**：公开 API 和内部实现严格分离，暴露必要的，隐藏内部的

## 8. 并发安全设计

- **默认非并发安全**：除非明确设计为支持并发访问，否则默认不保证安全，由调用者决定是否需要同步
- **明确并发安全声明**：如果类型设计为支持并发访问，文档必须明确说明
- **正确使用同步原语**：`sync/atomic` 用于原子标志，`sync.Mutex` 用于保护复杂状态，`channel` 用于通信，选择最合适的工具
- **封装并发细节**：将复杂的同步逻辑封装在干净简洁的 API 后面

## 9. 依赖管理

- **最小依赖原则**：只添加必要的外部依赖
- **核心层零业务依赖**：`spring-core` 不能依赖任何第三方业务包（如 Redis、GORM 等），这些集成放在 `starter-*` 中
- **内部共享工具**：通用工具抽取到 `stdlib`，供跨模块共享
- **模块化**：每个顶层组件有自己的 `go.mod`，松耦合独立演进

---

**总体原则**: 整洁一致的工艺。在保持 Go 自然习语和性能特性的同时，平衡 Spring 熟悉的编程模型。通过直观的 API 和详尽的文档专注于开发者体验。
