# 事实校验规则

本文用于约束 Go-Spring 文档中的事实性内容。写文档时可以组织语言，但不能编造 API、配置、默认值、版本、状态或行为。

## 校验优先级

事实来源按优先级使用：

1. 源码中的导出 API、类型定义、配置默认值和注释。
2. 测试用例中的行为断言。
3. `docs/3.examples/`、`misc/course/`、`misc/articles/` 中可运行示例。
4. 已有文档中的稳定描述。
5. 用户在当前任务中明确提供的事实。

当来源互相冲突时，以源码和测试为准，并在必要时指出文档或示例可能需要同步。

## 必须校验的内容

写入以下内容前必须找到来源：

- API、函数、方法、类型、接口、包名和导入路径。
- 配置 key、默认值、格式、优先级和是否必填。
- 自动注册的 Bean、注入对象和生命周期行为。
- 启动、关闭、刷新、错误处理等运行时行为。
- 版本号、弃用状态、实验性状态、迁移路径。
- 外部依赖名称、模块名、Starter 名和最小接入方式。

## API 校验

写 API 时要确认：

- 名称是否真实存在。
- 是否导出。
- 调用方式是否正确。
- 参数和返回值是否匹配。
- 示例中的包名和导入路径是否正确。

如果只能确认概念，不能确认 API 名称，应避免写具体调用，或明确标注“需要确认具体 API”。

## 配置校验

写配置时要确认：

- 完整 key 路径。
- 默认值。
- 配置格式是否被支持。
- 是否支持环境变量、命令行参数或多配置源覆盖。
- 配置生效时机：启动期、运行期、刷新后或仅初始化时。

配置示例应优先给完整路径，不使用孤立片段让读者猜上下文。

## 示例校验

示例分为两类：

- 可运行示例：应包含必要的 `package`、`import`、结构体、注册逻辑、启动入口和运行命令。
- 片段示例：只展示局部用法，必须通过前后文说明它放在哪里、依赖什么对象、不能直接独立运行。

示例代码应优先采用仓库已有模式。除非用户要求，不为了展示而引入新的第三方库或新的项目结构。

## 版本与状态校验

不要凭经验写版本信息。

可以写：

- 源码、测试、Changelog 或用户明确提供的版本事实。
- 已弃用、实验性、计划移除等在源码或文档中已有标记的状态。

不应写：

- “从 v1.2 开始支持”这类没有来源的版本。
- “将在 v2.0 移除”这类没有来源的计划。
- “推荐迁移到某 API”但该 API 未确认存在。

## 不确定时的写法

当事实没有确认时：

- 不要补想象中的 API 或行为。
- 不要用模糊词掩盖不确定性。
- 可以写“这里需要从源码确认具体 API”。
- 可以先保留概念描述，暂不写具体调用。
- 可以把待确认点列为开放问题，等待用户或后续源码核对。

## 最小核对动作

修改文档前，通常至少执行：

- 用 `rg` 搜索 API、类型、配置 key 或 Starter 名。
- 阅读相关源码或测试。
- 对照已有示例，确认推荐用法是否一致。

只有改错别字、调整排版、移动文件这类非事实变更时，可以跳过完整事实核对。
