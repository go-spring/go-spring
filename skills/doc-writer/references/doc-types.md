# 文档类型规则

本文用于判断 Go-Spring 文档应采用哪种组织方式。先判断文档类型，再决定读取哪份详细规则。

## 适用原则

不同文档承担不同任务，不应全部写成同一种 Guide 风格：

- 面向理解的文档，优先建立心智模型。
- 面向接入的文档，优先给出最短可运行路径。
- 面向集成的文档，优先说明安装、配置、注入对象、验证和排错。
- 面向记录的文档，优先保持事实准确和时间顺序清晰。

当目标文档位于 `docs/0.overview/`、`docs/1.getting-started/`、`docs/2.guides/` 或 `docs/4.integrations/` 时，应同时读取 `writing-style.md`。
当目标文档位于 `docs/3.examples/` 时，应读取 `example-docs.md`。
当目标文档是 `docs/5.faq.md`、`docs/6.contributing.md` 或 `docs/7.changelog.md` 时，应读取 `maintenance-docs.md`。

## `docs/0.overview/`

项目概览用于说明 Go-Spring 的定位、设计目标、核心能力和取舍。

应当包含：

- 项目解决什么问题。
- Go-Spring 与 Go 原生开发、Java Spring、主流 Go 框架之间的关系。
- 核心能力边界，例如配置、依赖注入、日志、生命周期、Starter 生态。
- 设计取舍和不做什么。

不应写成：

- API 使用手册。
- 安装教程。
- 功能列表堆叠。
- 没有边界的宣传文案。

## `docs/1.getting-started/`

快速开始用于让读者尽快跑起第一个可工作的 Go-Spring 应用。

应当包含：

- 环境要求。
- 安装或初始化方式。
- 最小可运行示例。
- 运行命令和验证方式。
- 下一步阅读建议。

写作重点：

- 优先减少认知负担，不提前展开复杂机制。
- 示例必须完整到读者可以复制、运行、验证。
- 每引入一个新概念，只解释当前路径所需的最小含义。

## `docs/2.guides/`

Guide 用于解释某个核心能力的使用方式、机制、边界和生产实践。

常见结构：

- 开篇说明能力定位、适用场景和核心价值。
- 前置条件。
- 快速开始。
- 核心概念或核心机制。
- 基础用法。
- 高级用法。
- 配置项。
- 生产实践。
- 故障排查。
- 常见问题。
- 相关文档。

写作重点：

- 深度由主题复杂度、使用风险和生产影响决定。
- 先讲基础路径，再讲高级能力，最后讲生产实践和边界行为。
- 对 IoC、配置优先级、启动生命周期、日志等复杂主题，应解释必要原理和设计取舍。

## `docs/4.integrations/`

Integration 文档用于说明某个 Starter 或外部组件如何接入 Go-Spring。

应当包含：

- 集成对象和适用场景。
- 安装方式。
- 必要配置和默认值。
- 自动注册或暴露的 Bean。
- 最小使用示例。
- 验证方式。
- 常见故障和排查信号。
- 与相关 Guide 或外部生态文档的链接。

写作重点：

- 不只写 `import _` 和配置片段，还要说明导入后框架提供了什么能力。
- 配置示例必须给完整 key。
- 注入对象、生命周期、关闭行为、连接池或客户端复用等生产相关内容应明确说明。

## `docs/3.examples/`

Example 目录暂不强制套用 `writing-style.md`，但应保证示例能运行、说明能帮助读者理解示例意图。
修改示例索引或示例 README 时，应读取 `example-docs.md`。

应当包含：

- 示例目标。
- 运行命令。
- 关键文件说明。
- 预期输出或验证方式。

不必强求：

- 完整 Guide 结构。
- 大量原理解释。
- 与生产场景无关的扩展说明。

## `docs/5.faq.md`

FAQ 用于聚合跨模块、高频、问题驱动的内容。
修改 FAQ 时，应读取 `maintenance-docs.md`。

应当包含：

- 真实问题作为标题。
- 简短直接的答案。
- 必要时链接到更完整的 Guide。

不应写成：

- 按功能模块重新组织的教程。
- 大段背景介绍。
- 与具体问题无关的概念说明。

## `docs/6.contributing.md`

贡献文档用于说明贡献流程、开发约定和提交要求。
修改贡献文档时，应读取 `maintenance-docs.md`。

应当包含：

- 本地开发准备。
- 测试和格式化要求。
- 提交或 PR 约定。
- 文档、示例、测试的同步要求。

不应套用 Guide 的“快速开始 / 核心概念 / 生产实践”结构。

## `docs/7.changelog.md`

Changelog 用于记录版本变化。
修改 Changelog 时，应读取 `maintenance-docs.md`。

应当包含：

- 版本号或日期。
- 新增、变更、修复、弃用、移除等分类。
- 影响范围和迁移提示。

写作重点：

- 保持事实准确。
- 不写营销化总结。
- 不补没有来源的版本信息。
