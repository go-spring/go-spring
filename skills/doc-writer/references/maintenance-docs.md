# 维护型文档规则

本文约束 `docs/5.faq.md`、`docs/6.contributing.md` 和 `docs/7.changelog.md`。这类文档服务于查找、协作和版本记录，不应套用 Guide 的完整教程结构。

## FAQ

FAQ 用于回答跨模块、高频、具体的问题。

应当这样组织：

- 用真实问题做标题。
- 先给直接答案，再补必要背景。
- 问题需要完整展开时，链接到对应 Guide。
- 同一类问题可以分组，但分组不应替代问题标题。

推荐写法：

```md
### 为什么我的 Bean 没有被创建？

Go-Spring 默认按需创建 Bean。只有 Root Bean 或被 Root Bean 间接依赖的 Bean 会被创建。

如果只是注册了 Bean，但没有任何入口依赖它，它不会自动实例化。关于 Root Bean 语义，见 [IoC 容器](./2.guides/02-ioc-container.md)。
```

不建议：

- 把 FAQ 写成完整功能教程。
- 用“配置管理”“依赖注入”这类模块名代替具体问题。
- 给没有事实来源的性能、稳定性或生产可用性承诺。

## Contributing

贡献文档用于降低协作成本。

应当包含：

- 本地开发准备。
- 依赖安装或生成步骤。
- 测试、格式化和静态检查命令。
- 文档、示例、测试的同步要求。
- Issue、PR、Commit 或 Review 约定。

写作重点：

- 命令必须能在仓库上下文中执行，或说明执行目录。
- 约定应具体到行为，例如“修改 API 时同步测试和文档”，不要只写“保持质量”。
- 不引入当前项目没有采用的流程。

不应包含：

- 项目愿景和功能介绍。
- 面向最终用户的安装教程。
- 没有维护者确认的发布流程。

## Changelog

Changelog 用于记录版本变化和迁移影响。

推荐结构：

```md
## vX.Y.Z - YYYY-MM-DD

### Added

- 新增能力。

### Changed

- 行为变化和影响范围。

### Fixed

- 修复内容。

### Deprecated

- 弃用内容和替代路径。

### Removed

- 移除内容和迁移提示。
```

写作重点：

- 只记录已经确认的变化。
- 行为变更应说明影响范围。
- 弃用和移除应说明替代路径或兼容边界。
- 不把 Changelog 写成发布宣传稿。

如果没有版本号或发布日期来源，不要补写虚构版本；可以先用“未发布”或等待维护者确认。

## 共同要求

- 事实性内容遵守 `fact-checking.md`。
- 链接只指向直接相关文档。
- 保持中文表达稳定、简洁。
- 不强行套用 `writing-style.md` 的 Guide 结构。
