# 示例文档规则

本文约束 `docs/3.examples/` 下的示例索引、示例 README 和示例说明。示例文档的目标不是完整讲解框架机制，而是帮助读者快速理解示例展示什么、如何运行、如何验证。

## 核心目标

- 可运行：命令、路径、依赖和预期行为清楚。
- 可定位：读者能知道关键代码在哪些文件里。
- 可迁移：说明示例展示的 Go-Spring 用法能迁移到什么场景。
- 不膨胀：不把示例 README 写成完整 Guide。

## 示例索引

`docs/3.examples/examples.md` 应作为示例总入口。

应当包含：

- 示例名称和一句话定位。
- 适用场景，例如启动流程、无 Web 应用、HTTP API、分层项目、第三方 Server 集成。
- 示例目录路径。
- 运行方式或指向示例 README。

不应包含：

- 大段机制解释。
- 复制每个示例的完整代码。
- 没有维护价值的营销式描述。

## 单个示例 README

每个示例 README 优先回答四个问题：

- 这个示例展示什么。
- 关键代码在哪里。
- 如何运行。
- 如何确认运行结果符合预期。

推荐结构：

````md
# 示例名称

一句话说明示例目标和适用场景。

## 示例重点

- 关键点 1。
- 关键点 2。

## 关键文件

- `main.go`：说明职责。
- `internal/...`：说明职责。

## 运行

```bash
go run .
```

## 验证

说明预期日志、HTTP 响应、退出行为或测试命令。
````

## 运行说明

运行命令必须以示例目录为上下文。

可以这样写：

````md
在 `docs/3.examples/bookman/` 目录执行：

```bash
go run .
```
````

不建议只写：

````md
运行：

```bash
go run .
```
````

原因：示例目录较多，读者需要明确命令在哪个目录执行。

## 验证方式

示例文档应说明至少一种验证方式：

- HTTP 示例：给出访问地址、请求方法、示例请求体和预期响应要点。
- Runner 示例：说明预期日志或自动执行流程。
- 配置示例：说明修改哪个配置 key 后可以观察到什么变化。
- 退出或生命周期示例：说明如何触发退出，以及预期清理行为。

验证说明不要求覆盖所有细节，但必须让读者知道示例已经正常工作。

## 中英文 README

如果同一示例同时存在 `README.md` 和 `README_CN.md`：

- 两份文档的核心事实应保持一致。
- 代码、命令、路径、配置 key 和接口列表应同步更新。
- 不要求逐句翻译，但不应出现行为描述不一致。

## 示例与 Guide 的关系

示例 README 可以链接到相关 Guide，但不要复制 Guide 的完整解释。

可以这样写：

```md
关于 Runner 和优雅退出的完整说明，见 [应用启停与运行机制](../../2.guides/03-app-start-stop.md)。
```

不建议把 `03-app-start-stop.md` 的生命周期章节完整搬进示例 README。
