# 最终审查清单

完成 Go-Spring 文档修改前，按本文进行最后检查。清单用于发现问题，不要求每篇文档都机械满足所有条目。

## 范围检查

- 是否判断了目标文档类型。
- 如果目标位于 `docs/0.overview/`、`docs/1.getting-started/`、`docs/2.guides/` 或 `docs/4.integrations/`，是否遵守 `references/writing-style.md`。
- 如果目标位于其他目录，是否避免强行套用 Guide 结构。

## 事实检查

- API、类型、函数、方法和导入路径是否已从源码、测试、示例或已有文档确认。
- 配置 key 是否完整，默认值和生效时机是否有来源。
- 自动注册的 Bean、注入对象和生命周期行为是否有来源。
- 版本、弃用、实验性、迁移路径等状态信息是否有来源。
- 是否删除或标注了无法确认的具体事实。

## 结构检查

- 开篇是否说明了当前文档真正需要交代的背景。
- 使用型文档是否给出最小可运行路径或最小可理解路径。
- 复杂主题是否按“基础用法 -> 高级用法 -> 生产实践 / 边界行为”展开。
- Integration 文档是否包含安装、配置、注入对象、验证和故障排查。
- FAQ 是否问题驱动，而不是重新写成教程。

## 示例检查

- 示例是否有足够上下文。
- 可运行示例是否包含必要的包、导入、注册逻辑和启动入口。
- 片段示例是否说明了放置位置和依赖条件。
- 配置示例是否给出完整 key 路径。
- 示例之后是否解释了关键点、默认行为、优先级或错误边界。

## 表达检查

- 术语写法是否统一，例如 Bean、Root Bean、Runner、Provider、Properties。
- API、配置 key、包名、模块名是否使用代码格式。
- 是否避免了无法验证的营销化表达、最高级和泛化承诺。
- 限制、不推荐用法和边界行为是否明确。
- 交叉引用是否指向真正相关、稳定的文档。

## 交付检查

- 是否说明了修改了哪些文件。
- 如果运行了校验命令，是否说明结果。
- 如果没有运行校验命令，是否说明原因。
- 是否留下了需要用户确认的开放问题。
