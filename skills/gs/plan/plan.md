# Plan 环节

PES 循环的第一环。把用户一句话的意图,变成一份**可执行的规划**:判断该不该由 gs 接管、要组合哪些工具、按什么顺序、验收标准是什么。

**核心认知**:一个请求往往需要**多个工具的组合**(如「加个带缓存的接口」= add-http + wire-starter + add-test)。Plan 的职责就是做这个分解与编排,而不是把请求硬塞进单一动作。

## 判定门(先于一切)

被加载 ≠ 要接管。Plan 第一步先判定:

**接得住(白名单)**:意图能分解为 `tools/` 里一个或多个能力的组合。
→ 进入规划。

**接不住**,任一成立即**放手**:

- 意图落不到任何工具上(纯 Go 语法/算法/三方库问题、与本仓库脚手架无关的求助);
- 需要的工具尚未落地;
- 只是话里提到 `gs`/`go-spring`,实际诉求在工具集之外。

放手 = 明确一句「此请求超出 /gs 覆盖范围,按通用方式处理」,**不**套用 PES 与共用约定,直接用通用能力回答。**判定不确定时,倾向放手。**

## 保真度可伸缩(关键)

**阶段不省,产物可省。** Plan 永远走,但产出与任务规模匹配:

- **小需求**(如「跑下测试」「重生成代码」):Plan 可以是一句话、零文档产出,判定命中单个工具即进 Execute。别为琐事产出规划文档,否则仪式感劝退用户。
- **大需求**(跨多工具、有取舍):才产出显式的分解清单,必要时用 TaskCreate 落任务跟踪。

## 工作流程

### 1. 澄清意图

读上下文(`AGENTS.md` / `CLAUDE.md` / `CODING_STYLE.md` / 目录约定 / 已有接口),不凭记忆假设。缺关键信息时用 `AskUserQuestion` 一次问透(优先问影响分解与选型的项):目标与验收、范围边界(明确**不做**什么)、依赖与约束。

### 2. 分解 + 选工具

把意图拆成有序的工具调用序列:

- 每一步对应 `tools/` 里一个能力;
- 有依赖的排好序(如 add-http 改 IDL → gen 重生成 → 写 handler);
- 有多技术路线时给出取舍理由;
- 外部组件**优先查现成 Starter**(见 wire-starter),不从零写。

### 3. 定验收

明确「做完什么样算完成」。验证由 Execute 内建执行(build/test),Plan 只需声明额外的验收点(如某接口手测、某边界必须覆盖)。

### 4. 确认后交棒 Execute

- 大需求:规划交用户确认,**未确认不进 Execute**;
- 小需求:判定清晰可直接进 Execute;
- 用户可随时回头调 Plan(见 SKILL.md 状态机),Plan 是可变状态,不是冻结合同。

## 完成后输出(大需求)

- 意图摘要 + 验收标准;
- 范围边界(做 / 不做);
- 有序工具调用序列(每步标注用哪个工具、为什么);
- 关键选型理由、未决点。

## 关键约束

- **只规划不落地**:本环节不写业务代码;编码在 Execute。
- **边界显式化**:「不做什么」和「做什么」一样重要。
- **不确定即问 / 即放手**:影响分解的未知项用 `AskUserQuestion`;超范围直接放手,不硬套。
