# fix-compile 子流程

定位并修复 Go-Spring 项目里的编译错误 / 测试失败。目标是**找到根因再改**,不是让报错消失。

## 何时使用

- 用户说「编译报错」「修一下」「test 挂了」「build 不过」;
- 其它子流程末尾验证失败,转入本阶段定位。

## 前置检查

- 定位到出错的子 module(有 `go.mod`);多 module 仓库别在根目录盲跑。
- Go 工具链在 PATH。

## 工作流程

### 1. 复现并读全错误

在目标 module 根目录流式执行,拿到**原始**报错:

```bash
go build ./...   # 或 go test ./...
```

- 从**第一个**错误看起,后续错误常是连锁反应;
- 不概括、不截断,读全 `file:line` 与错误类型。

### 2. 定位根因

按错误类型分流,先读相关文件再判断:

- **未定义/找不到符号**:多为生成物过期或未生成 → 先看是否该走 gen 阶段重生成,而非手写。
- **类型不匹配**:IDL 与手写代码脱节,或 DTO/领域类型改动未同步。
- **import 循环/缺失**:检查依赖方向是否反向(分层边界见项目 `CLAUDE.md` 与 `docs/agent-rules/domain-rules.md`)。
- **测试失败**:读断言原文,区分「实现错」还是「用例过时」;不靠删断言/改期望值掩盖。

### 3. 修复

- 生成物问题 → 回 gen 阶段重生成,不手改生成物;
- 手写代码问题 → 改最小必要范围,不顺手重构无关代码;错误包装遵循项目 `errutil` 约定(见 coding-style)。

### 4. 收敛验证

```bash
gofmt -l -w .
go test ./...
go build ./...
```

全绿才算修完;仍失败则回到第 2 步,不要反复重跑同一命令期待变化。

## 完成后输出

- 根因一句话说明(不是「改了 X 就好了」,而是「因为 Y 所以报 X」);
- 改动文件清单;
- 验证结果;
- 若根因涉及生成物/IDL,提示后续同类改动的正确姿势。

## 关键约束

- **禁止**用 `--no-verify`、跳过 hook、删测试/改期望值等方式绕过失败。
- **禁止**手改生成物来消错;生成物错回 gen 阶段。
- **禁止**吞掉或概括后重跑掩盖失败;失败原文交给用户。
- 一次只动根因相关代码,不趁机重构。
