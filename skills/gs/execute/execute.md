# Execute 环节

PES 循环的第二环。按 Plan 定下的工具序列逐步执行,**每步内建验证**,失败按爆炸半径回环 Plan。

**核心认知**:验证不是可选工具,是 Execute 的**内建职责**。任何改动后都要收敛验证(build/test),漏了就是 bug。Plan 不需要显式把验证排进序列。

## 工作流程

### 1. 逐步调用工具

按 Plan 的顺序进入 `tools/` 对应文档,**先读文档再执行**,不凭记忆:

- 尊重步骤间依赖(如 gen 必须在写 handler 前);
- 每个工具自带前置检查,不满足直接终止当前步,回报 Plan。

### 2. 每步收敛验证(内建)

每完成一个会改动代码的工具后,在目标子 module 根目录流式执行:

```bash
gofmt -l -w .
go test ./...
go build ./...   # 仅改测试代码时可跳过
```

验证机制细节见 [`build-test.md`](build-test.md)。**禁止**用 `--no-verify`、删测试、改期望值等方式绕过。

### 3. 失败按半径回环 Plan

执行/验证失败时,**不推倒重来**,按爆炸半径决定 replan 范围:

- **局部失败**(命令挂了、编译错、单测红):先定位根因(见 [`fix-compile.md`](fix-compile.md)),只在**当前步**修复重试,已完成的步骤不动;
- **前提失效**(方案根本行不通、依赖不存在):才回到整体 Plan 重新规划。

默认走局部回环,全量重来是例外。

### 4. 离题旁问不破坏状态

执行中用户临时问无关问题,直接答,答完回到原执行进度。Execute 状态是默认框架,不是牢笼。

### 5. 全部完成 → 交棒 Settle

工具序列跑完且验证全绿,进入 Settle 环节。

## 完成后输出

- 已执行的工具序列与各步结果;
- 每步验证结论(gofmt / test / build);
- 遇到并处理的失败(根因 + 修复);
- 交给 Settle 的改动清单。

## 关键约束

- **验证内建不可省**:每个代码改动后必跑收敛验证。
- **失败局部回环**:不因单步失败作废已完成步骤;不吞失败、不重跑掩盖。
- **改动收敛**:一次只动当前步相关代码,不趁机重构无关部分。
