# 研发文档链:从需求到交付

> 参考文档。每一环都要能"对回上一环":设计对回需求,实现对回设计,交付对回验收标准。

```
PRD(做什么) → 设计文档+接口契约(怎么组织) → 代码+测试(写得对) → 发布/回滚+可观测(交付安全)
```

## 1. 需求层 —— 做什么 / 为什么
- **PRD / 需求文档**:业务目标、用户场景、验收标准(验收标准必须可测)。
- **需求评审记录**:把边界、异常路径、优先级钉死。

## 2. 设计层 —— 怎么组织解决方案(最易缺失,最致命)
- **技术方案 / 设计文档**:架构分层、模块边界、依赖方向、关键数据结构、算法选型、并发/一致性/幂等策略;**决策要带理由**。
- **接口契约(API / IDL / proto)**:跨系统、跨团队的硬边界,先于实现冻结。
- **数据模型 + 迁移方案**:Schema、索引、迁移与回滚。

## 3. 实现层 —— 写得对且一致
- **代码规约 / CODING_STYLE**:形式一致性。
- **测试**:把"实现是否符合设计与验收标准"变成可执行证据。
- **Code Review 记录**:捕捉规约管不到的设计走样。

## 4. 交付层 —— 能否安全上线、出问题怎么办
- **变更记录 / CHANGELOG / commit 说明**:上下文追溯。
- **发布 / 回滚方案 + 部署文档**:灰度、开关、回滚步骤。
- **可观测性说明**:日志、指标、告警。
- **验收 / 上线报告**:对回 PRD 验收标准,闭环。

## 与 PES 控制环的对应

| 文档链层 | PES 环节 |
| --- | --- |
| 需求层 + 设计层 | **Plan**:分解意图、选工具、定验收 |
| 实现层 | **Execute**:调用工具 + 内建验证 |
| 交付层 | **Settle**:交付说明 + 沉淀约定 |

保真度可伸缩:小需求各层可压缩成一句话,但**层永远在,不能跳**。
