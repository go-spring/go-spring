# 文档与代码同步

> 参考文档。核心:同步不是靠"更勤快",而是靠"更少需要同步"。依赖人自觉更新文档必然漂移。

## 层次一:能生成的,绝不手写(消除同步)

让文档从代码/契约派生,不并行维护两份:

- **接口文档** ← IDL / proto / OpenAPI 生成(本 skill 的 `gen` 工具)。
- **配置文档** ← 配置结构体 tag / 注释生成。
- **数据模型** ← migration / schema 生成字段表。

规范:**禁止手改生成物,只改源(IDL/结构体),回 `gen` 重生成。**

## 层次二:必须手写的,绑进变更闭环(让漂移可见)

设计文档、README、架构图这类没法生成的,让"没同步"在流程里暴露:

- **PR 模板加勾选项**:改了接口/配置/行为?对应文档已更新?
- **CI 卡点**:改了 `*.proto` 但生成物没跟着变 → 流水线红。
- **就近原则**:文档离代码越近越不易漂,优先同目录,少放远端 wiki。

## 层次三:用测试兜住"文档说的 = 代码做的"

- 文档示例做成可运行的 example 测试(Go `Example` 函数),文档过期 → 测试挂。
- 文档不是"描述",而是"可执行的断言"。

## 给 RD 的规范:约束动作,不约束态度

1. **改契约先于改实现**:IDL/proto/schema 是唯一真源,先改它再 `gen`,不手改生成物。
2. **文档同步进同一个 PR**:不允许"文档后补"——后补 = 永不补。
3. **能自动化的一律自动化**:新增一类需手工维护的文档前,先问"能否生成或用测试兜住"。
