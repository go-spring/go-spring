# Settle 环节

PES 循环的第三环,收尾闭环:对本轮改动**产出交付说明**,并把值得固化的约定**沉淀**回正确载体。

## 何时进入

- Execute 全部工具跑完、验证全绿后自动进入;
- 用户说「出个变更说明」「交付」「沉淀一下」也可单独触发本环节。

同样遵循**保真度可伸缩**:小需求的 Settle 可以就是一句改动摘要;大需求才产出完整交付说明 + 沉淀。

---

## 一、交付说明

### 前置

用 git 查实际改动,不凭记忆罗列:

```bash
git status
git diff --stat
```

### 摘要结构

- **变更内容**:做了什么,按「为什么/意图」组织,不逐行复述 diff;
- **影响范围**:涉及哪些 module / 接口 / 配置,有无兼容性影响;
- **验证方式**:**实际跑过**的命令与结果(Execute 里的 gofmt/test/build,必要时接口手测);
- **遗留风险**:未覆盖路径、待办、需用户关注点。

接口/配置有变更时,单列 method+path+关键字段 / 配置键的前后差异,方便调用方对齐。

---

## 二、沉淀约定

### 该不该沉淀(先想清楚)

按 `CLAUDE.md` 的三问判断:

1. 新手会自然这么做 → **不记**;
2. 破坏会导致构建失败/评审拒绝/线上事故 → **优先记**;
3. 别处已解释 → **链接而非重复**。

### 载体选择

| 信息类型 | 载体 |
| --- | --- |
| 项目级约定、目录/分层规则、编码风格 | 项目 `CLAUDE.md` / `CODING_STYLE.md` / `docs/agent-rules/` |
| 某个工具的具体做法、踩坑、命令 | `tools/` 对应工具文档 |
| PES 流程 / 判定门 / 工具集调整 | 本 skill `SKILL.md` 或对应环节文档 |
| 跨会话的用户偏好 / 项目背景 | auto memory(不是项目文件) |

### 回写

- 写前先读目标文件查重,有等价条目就**更新**而非新增;
- 简洁、可执行,记「约定」的同时记**为什么**(便于日后判断边界);
- 遵守既有格式(如 Apache License header);
- 改影响面大的文件(CLAUDE.md / SKILL.md)先与用户确认。

## 完成后输出

- 一份可贴进 PR / 交付文档的摘要(变更内容 / 影响范围 / 验证方式 / 遗留风险 + 接口配置差异);
- 沉淀了哪几条、各自落到哪个文件(新增 or 更新);
- 判断为不值得沉淀的信息,说明原因。

## 关键约束

- **据实汇报**:验证方式只写真跑过的;风险不隐藏。
- **不代替提交**:除非用户明确要求,不自动 `git commit` / `push` / 建 PR。
- **沉淀不重复、载体唯一**:一条约定只落一处,能链接就链接。
