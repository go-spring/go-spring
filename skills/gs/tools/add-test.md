# add-test 子流程

为已有代码补单元测试 / 集成测试 / 示例,提升覆盖与可回归性。

## 何时使用

- 用户说「补测试」「加用例」「加 example」「覆盖率不够」;
- 新增/改动逻辑后需要补回归用例。

## 前置检查

- 定位到目标子 module(有 `go.mod`)。
- 读现有测试:沿用项目的测试组织方式与 helper(`assert` / `require`、fake、testdata),不另起一套风格。

## 工作流程

### 1. 圈定待测目标

- 明确要覆盖的函数/方法与关键路径;
- 优先补:核心业务逻辑、易错分支、刚改动过的代码。

### 2. 设计用例

每个目标至少覆盖:

- **正常路径**一条;
- **失败路径**至少一条(参数校验 / 业务错 / 边界值)。

### 3. 写测试

- 断言默认 `assert`,慎用 `require`;不 mock 数据库,用真实依赖或轻量 fake。(测试风格见 coding-style §6,测试策略见 `docs/agent-rules/domain-rules.md`)
- 表驱动优先,用例名表意清晰。
- controller 层测协议无关逻辑,handler 层按需补 HTTP 集成测试。

### 4. 补示例(按需)

用户要 example,或改动引入了新用法时,在 `examples/` 补一条**最小可运行**示例,跟随现有示例结构。

### 5. 验证

在子 module 根目录流式执行:

```bash
gofmt -l -w .
go test ./...
```

用户要看覆盖时再加 `-cover`。测试失败先判断是「用例写错」还是「暴露了真实 bug」,后者转 fix-compile 阶段。

## 完成后输出

- 新增测试/示例文件清单;
- 每个目标覆盖到的正常/失败路径;
- 验证结果(通过数,必要时覆盖率);
- 仍未覆盖的路径与原因。

## 关键约束

- **默认 `assert`,慎用 `require`**(项目约定)。
- **禁止 mock 数据库**(项目约定)。
- **禁止**为凑覆盖率写无断言的空跑用例。
- 示例必须最小可运行,不堆无关代码。
