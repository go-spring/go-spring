# starter-http-client Design

[English](DESIGN.md) | [‰∏≠Êñá](DESIGN_CN.md)

`starter-http-client` is the *client* archetype starter (see
[starter/DESIGN.md](../DESIGN.md) ¬ß2.2, adapted for HTTP) that wires the
runtime transport stack for declarative HTTP clients generated by
`gs-http-gen` ‚Äî Go-Spring's OpenFeign / `@HttpExchange` equivalent.
It never registers a default singleton ‚Äî every `*http.Client` it
publishes is named under `${spring.http-client}` and injected by name.

## 1. Responsibilities & Boundaries

- **In scope:** build a `*http.Client` per configured instance whose
  `Transport` composes tracing + discovery-aware load balancing +
  resilience, and register a destroy hook that releases discovery
  watches and resilience executors.
- **Out of scope:** the interface definitions and code-generation (that
  is `gs-http-gen`); the actual load-balancing algorithms (that is
  `spring/loadbalance`); resilience policies (that is
  `spring/resilience`).

## 2. Transport Composition ‚Äî outer to inner

The chain assembled by `spring/httpx.NewTransport` is:

```
resilience  ‚Üí  discovery + LB (balancedTransport rewrites host)  ‚Üí  otelhttp base
```

- **otelhttp base.** Always innermost so span propagation includes the
  full downstream path (including rewritten host + LB pick).
- **discovery + LB.** When `service-name` is non-empty:
  `discovery.MustGet` + `LiveDialer` + `loadbalance.Pool` ‚Äî the same
  client-side stack every infrastructure client uses. Otherwise a
  `fixedHostTransport` handles the `Addr` case. Both empty ‚Üí the
  request's own host is used.
- **resilience.** Outermost so **retry can re-pick endpoints** on each
  attempt and **circuit-breaker keys by logical service name**
  (`req.URL.Host` = the generated client's Target) rather than by
  per-endpoint dial address.

## 3. Key Decisions

- **Multi-instance only.** No default singleton `*http.Client` bean ‚Äî
  add an instance is a pure-config change. Matches the family rule
  ¬ß2.2 for client starters even though HTTP does not have a specific
  driver registry here.
- **Generated code depends only on stdlib.** `gs-http-gen` output
  imports `net/http` + `spring/httpclt` ‚Äî never a specific starter.
  Declarative dispatch works through generated code (no runtime proxy),
  and the code has a `HTTPClient *http.Client` seam populated by DI.
- **`managedTransport` carries a `closeFn`.** The starter's destroy
  type-asserts the transport to `io.Closer` and releases the discovery
  watch + `resilience.Executor` behind it, so an instance removed at
  refresh time does not leak goroutines.
- **Fail-fast at assembly.** `httpx.NewTransport` returns an error if a
  ServiceName cannot be resolved to a registered discovery backend or
  the resilience driver is unknown; the boot surfaces the config
  problem, not the first request.
- **In direct-connect mode `Target` is a resource name.** The generated
  client's `Target` becomes the circuit-breaker resource label and
  shows up in logs, but does not affect routing ‚Äî `httpx` fully owns
  addressing by rewriting the host.

## 4. Constraints

- **Retries require `GetBody`.** The retry loop replays the request; a
  body that only supports a single read cannot be retried.
- Internal deps resolve through `go.work` ‚Äî do not `go mod tidy`.
- Order matters: swapping resilience and LB (retry *below* LB) would
  break by-service circuit-breaking and let the same endpoint absorb
  every retry.

## 5. Trade-offs / Alternatives Rejected

- **Runtime reflection-based proxy ‚Äî rejected.** Go has no equivalent of
  Java's dynamic proxy; declarative clients ride code generation,
  keeping the runtime cost equal to a hand-written client.
- **Resilience *inside* the LB ‚Äî rejected.** Retries would stick to
  the same endpoint; breakers would key per address, blowing the
  logical-service abstraction.
- **Shared default `http.Client` bean ‚Äî rejected.** Same reason all
  client starters dropped the default singleton: dual registration was
  error-prone and the conditional-singleton semantics were opaque.
