# 从装配应用，到装配研发 —— Go-Spring 面向 AI 时代的 Process as Code 宣言

## 缘起

Go-Spring 一开始的定位是"让 Go 服务更好写"—— 用 IoC、Starter、生命周期这套 Spring 用了二十年的思想,把 Go 应用的运行时装配变得简单。

然而在做 `/gs` skill 的过程中,我发现了一件更大的事情:**同一套思想其实可以往上再推一层,用来装配"研发流程"本身。**

这份文档不是 skill 的使用手册,也不是 `/gs` 的功能规划,更不是一份已经完成闭环的技术设计。它是一份宣言:用一个可能过度乐观、但值得认真验证的视角,回答一个问题:

> Go-Spring 项目未来五年,想要解决的到底是什么问题?

答案是:**软件研发流程的可编码化(Process as Code)。**

## 核心命题

**研发流程也是一个可以被 IoC 装配的应用。**

回顾一下 IoC 在框架里做了什么:

- 把"运行时的组件依赖关系"从代码里提取出来,变成可声明、可替换、可组合的显式配置。
- 结果是:不同项目可以复用同一批组件,不同环境可以切换同一份骨架。

如果我们把主语替换掉:

- 把"研发过程的动作依赖关系"从每个团队的脑子里提取出来,变成可声明、可替换、可组合的显式流程。
- 结果是:不同团队才有机会复用同一批研发能力,不同规模才有机会切换同一份流程骨架。

这不只是比喻,更像一种值得验证的**同构假设**。只要一部分研发动作能被形式化 —— 输入是什么、产出是什么、依赖是什么 —— 它就有机会像 Bean 一样被 IoC 装配。

## 类比:三层同构

| 层 | 装配对象 | 装配单元 | 装配包 | 生命周期 |
| --- | --- | --- | --- | --- |
| Spring / Spring Boot | Java 应用 | Bean | Starter | 应用启动/停止 |
| Go-Spring | Go 应用 | Bean | `starter-*` | 应用启动/停止 |
| **/gs**(本文档主张) | **研发流程** | **Stage 实现** | **Skill Starter** | **Pipeline 执行** |

这三层共享的是相近的思想,只是"应用"的粒度依次放大:

- 第一层管的是**方法调用**如何组织。
- 第二层管的是**服务启动**如何组织。
- 第三层管的是**软件研发本身**如何组织。

Spring 证明了第一层可行,Go-Spring 正在证明第二层可行。这份文档主张:**第三层值得下注,而且可能成为 Go-Spring 项目未来最重要的方向。**

## 为什么"研发流程"能被抽象

"研发流程能被抽象吗?" —— 这是所有质疑的起点。答案倾向于"能",但它不会像抽象 HTTP 服务那么干净。研发流程里有组织、权限、风险偏好和人类判断,这些东西不能被简单抹平;但这不妨碍我们抽象其中稳定、重复、可验证的部分。

许多公司的研发流程,展开看都会反复出现同一组阶段:

1. **需求澄清** —— 把模糊描述变成可落地的任务
2. **设计与拆分** —— 决定模块边界、依赖、组件选型
3. **编码与生成** —— 顺着已有风格改代码
4. **验证** —— 编译、测试、扫描
5. **交付** —— 评审、发布、灰度
6. **沉淀** —— 文档、知识、约定回写

**差异不只在骨架,也在每个阶段用的具体工具和治理卡点**:

- 有的公司需求走 JIRA,有的走 Linear,有的走飞书;
- 有的验证走 Jenkins,有的走 Argo,有的走内部平台;
- 有的公司多一个"安全审批",有的公司多一个"变更公告"。

这跟"所有 HTTP 服务都长得差不多,只是有的用 gin、有的用 echo"并不完全一样,但给了我们一个可迁移的思路:先抽象稳定骨架,再把差异 Starter 化。既然应用运行时能被装配,研发流程中那些可重复、可描述、可验证的部分,也有机会被装配。

## 五个核心对象

要把这套思想落地成可用的东西,需要对以下五个对象进行形式化定义。它们的关系类似 Spring 里的 `Bean` / `BeanDefinition` / `ApplicationContext` / `Lifecycle`。

### 1. Artifact — 阶段间流转的产物

每个 Artifact 是一份**带 schema 校验的结构化数据**。核心 Artifact 至少包括:

- `TaskDescription` — 结构化任务描述(目标 / 边界 / 验收标准)
- `DesignPlan` — 模块边界 + 依赖 + 组件选型
- `ChangeSet` — 代码变更集
- `VerifyResult` — 验证结果(编译 / 测试 / 扫描)
- `ReleaseNote` — 变更摘要 + 风险 + 回滚方案

**Artifact 必须是 schema,不能只是自然语言。** 这是整个体系的地基:AI 的自然语言输出天然发散,schema 化不能消除这种发散,也不能保证严格的"同一输入 → 同一产物";但它能让产物可校验、可比较、可审计、可迁移。这是**编码化**与**纯提示词工程**的分界线。

Artifact 走 semver。不兼容变更走 major,允许发生,但需要迁移指南。

### 2. Stage — 研发流程的最小可寻址单元

Stage 是一个**接口式定义**:名字 + 消费的 Artifact + 产出的 Artifact + 覆盖关系。

```yaml
stage: verify.compile
consumes: [ChangeSet@v1]
produces: [BuildResult@v1]
overrides: default.verify.compile
```

Stage 不允许"顺手做点别的事" —— 依赖必须显式,产物必须声明。如果必须读写外部系统,也应该声明需要的能力、输入边界和副作用。这是 IoC 能成立的前提。

### 3. Skill Starter — Stage 实现的可分发打包

跟 `starter-go-redis` 之于 Go-Spring 一样,Skill Starter 是**一批 Stage 实现的打包**,可以通过 `go get`、插件市场或类似机制分发。

一个典型 Starter 可能声明:

- 我实现了 `verify.compile` / `verify.test` / `verify.scan` 三个 stage;
- 我覆盖了 default 版本;
- 我依赖 `ChangeSet@v1` 和 `BuildResult@v1`;
- 我需要外部配置:CI endpoint / token。

**公司实现覆盖内置实现,沿用 Go-Spring 的"同名条件互斥"策略,不允许隐式覆盖。**

### 4. Pipeline — 阶段图 + 锚点

Pipeline 是**有序阶段图 + 扩展锚点**。默认 Pipeline 是六阶段线性图。锚点允许公司在特定位置插入新阶段(比如 `after:verify` 插入 `security.review`),默认阶段原则上不删除;如果确实需要跳过,也必须通过显式策略和审计记录完成。

阶段默认不跳过,但可以**降档**执行:小改动走"快速档",大改动走"完整档"。降档规则不应该只看输入规模,还要看风险等级、历史约定和人工确认点。

### 5. Runner — 调度器

Runner 是执行引擎。负责:

- 加载 Pipeline 定义;
- 加载注册的 Skill Starter;
- 按阶段图串行 / 并行调度;
- 校验 artifact schema;
- 记录执行环境与版本,支持可复跑、可追踪;
- 记录偏差(实际产物与契约的差异)。

Runner 有 gs 官方默认实现。**默认实现先不依赖任何公司系统**,开源用户不接外部平台也能跑通本地研发闭环;评审、发布、灰度、安全审批等公司流程,通过 Skill Starter 接入。

## 三条底线

不做这三件事,这套东西会退化成"更花哨的提示词工程":

1. **Artifact 是 schema,不是文本。** 自然语言只在阶段内部使用,跨阶段一律走结构化数据。
2. **Stage 依赖显式声明。** 不允许 stage 之间通过"共享上下文""隐式记忆"交换信息。
3. **Runner 默认实现独立可用。** Skill Starter 是增强,不是必需。开源用户零配置也能跑通本地闭环。

违反任一条,整套抽象就漏了 —— 就像 Spring 里到处 `context.getBean()` 硬取一样,IoC 名存实亡。

## 谁获益,怎么获益

这套抽象一旦成立,受益者不止 Go-Spring 社区。

### 大公司

沉淀了几十年的研发经验,历来只以"内部文档 / 内部工具 / 老员工的口传"存在,离职就流失。编码成 Skill Starter 之后:

- 经验变成**可版本化、可回滚、可评审**的产物;
- 新员工上手当天就能得到公司最佳实践,不需要"跟着老同学学半年";
- 底层工具改造成本降低 —— 换掉 CI 平台,只需要重写一个 Starter,流程不动。

### 小公司

原本很难复用成熟团队的研发流程 —— 不是不想用,是学不动、拆不出、装不起来。编码化之后:

- 一句 `gs adopt <某团队>-workflow` 拉进一套可运行的流程骨架;
- 需求拆解、分支策略、变更摘要、上线卡点,先得到可参考的默认做法;
- 不需要一开始理解全部背景,先跑起来,再逐步吃透、删改和替换。

**这是"AI 时代工程化"真正的红利:经验成为可分发的产物。**

### 开源社区

先得到一个面向 Go-Spring / Go 服务研发的**公共流程抽象**。如果它在足够多真实项目中站住脚,再逐步外溢到跨语言、跨行业、跨公司的更大范围。就像 Kubernetes 之于运维、Spring 之于应用装配 —— 抽象本身就有巨大公共价值,但公共价值需要从具体实践里长出来。

## 与现有事物的对标

这件事在业界没有直接对标,但可以参照三种已有思路:

- **构建系统(Bazel / Nx)** —— 把构建过程编码成图。gs 是把研发过程编码成图。
- **CI 流水线(GitHub Actions / Argo Workflows)** —— 声明式管线 + 可扩展 step。gs 阶段图与之形态最近。
- **Spring 框架自身** —— IoC + Starter + 生命周期。gs 的思想直接来自这里。

**三者叠加 = /gs 要做的事**:像 CI 一样声明流程,像 Bazel 一样保留依赖与执行记录,像 Spring 一样可扩展。

## 最小落地闭环

不需要一开始就大干。先做一个能自证的最小闭环:

1. **定义 5 个核心 Artifact 的 JSON schema**:`TaskDescription` / `DesignPlan` / `ChangeSet` / `VerifyResult` / `ReleaseNote`。
2. **定义 6 个核心 Stage 的接口**:澄清 / 设计 / 编码 / 验证 / 交付 / 沉淀。每个给出默认实现。
3. **写 Runner 的最小版**:能加载 Pipeline、加载 Starter、串行调度、校验 artifact。
4. **写 1 个示例 Starter**:把 gs 项目自己的研发规范编码进去,用它跑自己的开发流程 —— 这是 **dogfooding**,也是抽象成立与否的第一次检验。
5. **邀请 1 个外部团队写 Starter**:验证抽象在不同上下文下是否稳定。跑通说明方向有更大机会;跑不通就回过头改契约。

前三步在开源 Go-Spring 仓里就能做。

## 更远处

如果这条路能够走通,Go-Spring 的定位会发生根本性变化:

- **今天的 Go-Spring** = "让 Go 服务更好写"
- **明天的 Go-Spring** = "让软件研发更好做"

前者服务于代码,后者服务于团队。前者的天花板是"另一个好用的框架";后者的天花板是 —— 因为研发流程这件事**至今很少被当成一个可装配的框架来认真对待**,天花板高到无法预估。

一个可能的图景:五年后,一个新公司起步的默认动作是这样的:

```bash
gs init -m github.com/newco/api      # 拉项目骨架
gs adopt team-workflow               # 拉某个团队沉淀出的研发流程
gs adopt security-practice           # 拉某套安全实践
gs run "加个用户查询接口"              # 按流程执行
```

第二、第三行,今天不存在。**让它存在,是这份文档主张的方向。**

## 定位声明

**从这份宣言开始,Go-Spring 试图同时成为两件事:**

1. **Go 应用运行时框架**(现在的 Go-Spring):管应用怎么装配、怎么运行、怎么退出。
2. **软件研发流程框架**(即将扩展的 `/gs`):管研发怎么组织、怎么执行、怎么沉淀。

两者共用同一套思想 —— **依赖注入 + 可插拔组件 + 显式生命周期**。前者是这套思想在应用层的实例化,后者是它在流程层的实例化。

它们不是主辅关系,而是同一套思想在两个层面的展开。至于这种同构能走多远,需要靠实践继续证明。

**Go-Spring 从此站在比"另一个 Go 框架"高一个维度的位置上。**
