# migration 设计
[English](DESIGN.md) | [中文](DESIGN_CN.md)

`migration` 是 stdlib 层零依赖的结构迁移抽象。它让 Go 获得 Flyway / Liquibase 的*效果*
——带版本、带校验和守卫、只向前、恰好一次地应用迁移——而不复刻它们的 XML/DSL 机制。后端
（gorm 之上的 MySQL/PostgreSQL/SQLite）作为独立 starter 贡献一个 `Store`。

## 1. 职责与边界

- 把一次结构变更建模为带编号的 `Migration`，其 `Up` 通过与驱动无关的 `Execer` 发出语句。
- 从 `Source` 收集迁移——嵌入的 `.sql` 目录（单一二进制场景）或代码注册。
- 按版本顺序应用未应用的迁移，并把每条记入版本表，使重跑成为空操作、被改动的历史脚本被捕获。
- 不拥有连接、DDL 方言或回滚。开数据库、跑 SQL 是 `Store` 的职责；自动降级迁移不在范围内
  （只向前，与 Flyway 社区版一致）。

## 2. 关键抽象与缝隙

- **`Store` 接口作为后端缝隙。** 没有全局驱动注册表。后端需要一个活的句柄
  （`*gorm.DB`、`*sql.DB`）来创建版本表并以事务方式运行迁移，而非声明式策略——因此缝隙是
  starter 实现的接口，与 `stdlib/batch`、`stdlib/lock` 相同的选择。`Store` 拥有
  `EnsureVersionTable`、`AppliedRecords`、`Apply`（在一个事务内运行 `Up` + 记录）与
  `MarkApplied`（baseline）。
- **`Execer` 把 `Up` 与驱动解耦。** 迁移的 `Up` 收到一个绑定到该迁移事务的 `Execer`，
  因此同一个 `Migration` 可在任意后端运行，事务范围由 `Store` 决定。
- **`Source` 把「获取」与「执行」分开。** `NewFSSource` 解析 Flyway 风格文件名并对文件内容
  做 SHA-256；`NewSource` 接受代码定义的迁移。Runner 消费 `[]Migration`，从不关心它们来自何处。
- **校验和是防篡改缝隙。** 源为每条迁移计算校验和（FS 用文件哈希，代码由调用方提供）；
  Runner 拿它与记录的校验和比较，漂移即失败。两者都为空表示「不追踪」，是允许的，因此代码
  迁移可以选择退出。

## 3. 约束

- **历史不可变。** 版本一旦应用即被冻结；改动其脚本会改变校验和并使下次运行失败。修复方式是
  新增更高版本，绝不是改动——这正是让一套共享迁移在团队中安全的原因。
- **版本 0 与重复版本被提前拒绝。** `loadSorted` 在应用任何东西之前校验整个集合，因此畸形
  集合在改动数据库之前就失败。
- **默认严格顺序。** 除非设置 `AllowOutOfOrder`，否则低于已应用最高版本的版本会被拒绝；
  默认行为防止一条晚合入的低编号迁移悄悄插队。
- **只向前、失败即停。** `Down` 是 Runner 从不调用的保留字段；`Up` 失败会中止本次运行，
  不再尝试后续迁移，因此数据库不会因越过失败继续而落入半已知状态。

## 4. 取舍与被否决的方案

- **不用驱动字符串注册表。** 迁移不是像 `discovery`、`resilience` 那样的配置期选择；
  没有活的数据库句柄就无法表达「我想要的 store」。此处「接口即缝隙」胜过注册表间接层。
- **不做自动回滚。** 降级迁移在实践中不可靠（删掉的列无法连数据一起复原），Flyway 社区版
  同样省略它。`Down` 保持为供外部工具使用的保留字段而非执行路径，因此框架从不承诺它兑现不了
  的撤销。
- **校验和守卫而非「直接重跑」。** 静默重跑被改动的脚本会让已应用 schema 依赖于运行历史；
  高声失败强制了不可变历史纪律，从而让各环境保持一致。
- **简单语句分割器，而非 SQL 解析器。** `splitStatements` 处理迁移文件承载的 DDL/DML
  （引号/`--` 注释之外的分号）；带内部分号的过程体应放进单语句文件由驱动整体执行，而不足以
  在 stdlib 中引入完整解析器。
