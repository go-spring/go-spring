# go-zero — HTTP/REST (Go-Spring style)

[English](README.md) | [中文](README_CN.md)

A [go-zero](https://go-zero.dev) `Greet` example whose stubs come from a
`.api` IDL via `goctl api go`, then refactored to boot and be configured the
Go-Spring way: `gs.Run()` drives the lifecycle, the business logic is an IoC
bean, and the bind address comes from `conf/app.properties` instead of
hard-coded `main()` wiring.

The service runs on go-zero's **rest.Server** (its HTTP framework). Two
consequences flow from that choice, both intentional:

- **No etcd, no service discovery.** Unlike the sibling `../greet-rpc`, this
  example has no docker-compose and no registry step. `rest.Server` has no
  built-in service discovery — the whole registry story exists only in
  go-zero's zRPC layer — so the consumer talks to a fixed `host:port`.
- **goctl-generated internal/ is a thin layer.** Only `internal/types/*.go`
  and `internal/handler/routes.go` come from goctl and are regenerated by
  `gen.sh`; the rest of `internal/{handler,svc,logic}` is hand-written so the
  Greet business logic can participate in Go-Spring's DI.

This is the HTTP half of the go-zero examples; the zRPC/gRPC half — same
`Greet` service, but generated from `greet.proto` — lives next door in
[`../greet-rpc`](../greet-rpc).

This is a runnable example, **not** a reusable starter module.

## Topology

```
┌────────────┐        HTTP POST /greet      ┌────────────┐
│  provider  │◀─────────────────────────────│  consumer  │
│ gs.Run()   │  {"name":"Hello, go-zero!"}  │ one-shot   │
│  :8888     │──────────────────────────────▶│ assert+exit│
└────────────┘   {"greeting":"Hello, ..."}  └────────────┘
```

## Layout

```
contrib/go-zero/greet-api/
├── greet.api                          # go-zero API IDL
├── gen.sh                             # regenerates goctl-owned files
├── internal/types/types.go            # goctl-owned request/response types (DO NOT EDIT)
├── internal/handler/routes.go         # goctl-owned route table (DO NOT EDIT)
├── internal/handler/greethandler.go   # hand-written; parses request, calls Logic bean
├── internal/svc/servicecontext.go     # hand-written; injected Logic bean surface
├── internal/logic/greetlogic.go       # hand-written; GreetLogic IoC bean
├── provider/handler.go                # HandlerRegister bean wiring routes + Logic
├── provider/server.go                 # RestServer adapter (gs.Server) + Config
├── provider/main.go                   # gs.Run(); long-lived HTTP server
├── consumer/main.go                   # HTTP POST, asserts on response, exits
├── conf/app.properties                # provider configuration
└── check.sh                           # smoke test: build+run provider, run consumer, tear down
```

## How it was generated

```bash
# tool (once)
go install github.com/zeromicro/go-zero/tools/goctl@latest

# scaffold the whole tree from the IDL (goctl variant used here)
goctl api go -api greet.api -dir <tmp>/gen --style gozero
```

`gen.sh` runs the same command inside a scratch workspace whose parent module
is named `greetapi`, then keeps only `internal/types/types.go` and
`internal/handler/routes.go` from the output. Everything else that goctl
scaffolds — `greet.go` (main), `etc/greet.yaml`, `internal/config`,
`internal/svc/servicecontext.go`, `internal/logic/greetlogic.go`,
`internal/handler/greethandler.go` — is intentionally not adopted: Go-Spring
owns the lifecycle and configuration, and the business logic lives in a
Go-Spring bean rather than a per-request `NewGreetLogic()`.

## The refactor: native go-zero → Go-Spring

| Concern         | Stock go-zero REST scaffold                    | Go-Spring version                                                          |
| --------------- | ---------------------------------------------- | -------------------------------------------------------------------------- |
| Startup         | `server.Start()` blocks in `main()`            | `RestServer` implements `gs.Server`; `gs.Run()` drives Run/Stop            |
| Handler wiring  | `handler.RegisterHandlers(server, svcCtx)`     | `gs.Provide(func(*GreetLogic) HandlerRegister { ... })`                    |
| Business logic  | `logic.NewGreetLogic(ctx, svcCtx).Greet(req)`  | `GreetLogic` is a singleton IoC bean; ServiceContext just carries it in    |
| Server enable   | always on                                      | conditional on a `HandlerRegister` bean via `gs.OnBean`                    |
| Config source   | hard-coded `etc/greet.yaml`                    | `${spring.rest.server}` from `conf/app.properties`                         |
| Registration    | none (REST has no discovery)                   | none — see the sibling `greet-rpc` for the etcd story                      |
| Shutdown        | process-owned                                  | graceful shutdown by Go-Spring (SIGTERM → `Stop()` → `rest.Server.Stop()`) |

## Configuration

```properties
# Disable the Go-Spring built-in HTTP server; the provider exposes only the
# go-zero rest.Server bound below.
spring.http.server.enabled=false

# go-zero rest.Server settings; read via the ${spring.rest.server} prefix.
spring.rest.server.name=greet
spring.rest.server.host=0.0.0.0
spring.rest.server.port=8888
```

## Run

Terminal A — start the provider (long-lived):

```bash
go run ./provider
```

Terminal B — start the consumer (direct HTTP call, self-asserts):

```bash
go run ./consumer
```

Expected consumer output:

```
Response from provider: Hello, go-zero!
```

Or run the one-shot smoke test (starts the provider, runs the consumer, tears
it down):

```bash
bash check.sh
```
