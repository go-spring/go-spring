# spring Bug 扫描与修复任务

## 目标

对当前仓库进行系统性代码扫描，发现真实 bug 后逐个记录、修复、验证并提交。

本文档是后续所有 bug 扫描、记录、修复、验证、提交任务的总规约。
凡是修改本仓库代码、测试或相关任务记录，都必须先遵循本文档要求；
如果用户后续临时提出更具体要求，以用户最新要求为准，但不能默认为规模化重构或扩大修改范围。

要求采用“一个 bug 一个闭环”的方式推进，便于逐项检查结果：

1. 发现一个 bug。
2. 为该 bug 创建一个独立子文档并记录问题。
3. 编写或补充能暴露该问题的测试；如果不适合写测试，需要说明原因。
4. 修复该 bug。
5. 运行相关验证。
6. 提交一次 Git commit。
7. 继续扫描下一个 bug。

允许先发现一批 bug、记录一批 bug，再按优先级逐个修复；但修复、验证、提交仍必须保持“一个 bug 一个闭环”。

## 扫描范围

- 重点扫描 `conf/`、`gs/`、`gs/internal/` 等核心代码。
- 同步关注现有测试、示例用法、边界条件、错误处理、并发与生命周期逻辑。
- `vendor/` 目录默认不修改，除非问题明确来自本仓库对依赖的使用方式。
- 文档、注释、命名问题不算 bug，除非会误导用户写出错误代码或导致行为不一致。

## Bug 判定标准

优先处理满足以下任一条件的问题：

- 与公开 API 的预期行为不一致。
- 边界输入、空值、重复调用、错误路径会导致 panic、死锁、资源泄漏或错误结果。
- 配置解析、依赖注入、生命周期、条件判断、动态刷新等核心能力存在行为缺陷。
- 测试覆盖缺口能够隐藏实际错误，而不是单纯追求覆盖率。

判定 bug 时必须写明预期来源，例如公开文档、已有测试意图、API 命名约定、同类代码的一致行为、明确的运行错误或可复现的异常路径。
不能仅凭个人风格偏好、主观设计偏好或“可以更优雅”认定为 bug。

暂不作为本任务处理的内容：

- 大规模重构。
- 借修 bug 的名义改造架构、重排模块边界或批量调整 API。
- 单纯性能优化。
- 风格统一或格式调整。
- 没有明确用户影响的理论问题。

## 工作流要求

- 以省 token 为目标：优先读取必要文件、必要片段和相关测试，不做无意义的大段展开。
- 鼓励分批发现：可以先集中扫描一个有限范围，记录多个候选 bug，再逐个进入修复流程。
- 每个 bug 尽量在新的工作流或新的上下文中处理，降低上下文膨胀带来的遗漏风险。
- 新工作流开始时必须先读取本文档和目标 bug 子文档，再开始修改。
- 如果上下文不足，应优先依赖子文档中的问题描述、复现方式、验证记录和提交状态，而不是重新扫描全部代码。
- 不做规模化重构。除非用户明确批准，否则每次修复只允许修改当前 bug 所必需的最小代码、测试和记录。
- 如果发现某个问题需要较大设计调整，先记录为候选 bug 或风险项，不修复，并在事后汇总告知用户。

## 安全边界

- 开始任何扫描或修复前，必须执行 `git status --short`，确认当前工作区状态。
- 如果存在与当前任务无关的未提交修改，不得回滚、覆盖或混入提交；必要时只记录风险并跳过相关区域。
- 禁止使用破坏性 Git 操作，例如 `git reset --hard`、强制 checkout、删除用户未确认的文件。
- 能用测试复现的问题必须先补测试；不能补测试时，必须在子文档说明原因和替代验证方式。
- 修复当前 bug 时发现其他独立问题，只创建新的 bug 子文档，不顺手修复。
- 不主动更新依赖、不修改 `vendor/`、不引入新工具；除非该 bug 无法在现有工具链下修复，并在子文档中说明原因。
- 如果测试、依赖下载、生成代码或外部工具会修改大量文件，先停止执行并记录原因，避免隐式扩大修改范围。
- 每个子文档必须足够自包含，至少包含预期来源、复现方式、影响范围和当前状态，保证新工作流可以接手。

## 处理流程

每个 bug 独立处理，避免多个修复混在同一个提交中：

1. 定位问题：说明触发条件和影响范围。
2. 复现问题：优先添加失败测试；无法添加测试时记录原因。
3. 修复问题：保持改动范围尽量小，遵循仓库现有代码风格。
4. 验证结果：至少运行相关包测试；必要时运行 `go test ./...` 或项目脚本。
5. 更新记录：在该 bug 的子文档中补全状态、验证命令和提交信息，并在本文档索引中更新摘要。
6. 提交代码：每个 bug 单独 commit，提交信息应能看出修复内容。

## Bug 子文档

每个 bug 使用一个独立 Markdown 子文档记录，建议存放在：

```text
任务记录/bug-N-简短标题.md
```

命名要求：

- `N` 使用递增编号。
- 标题使用简短中文或英文，能反映问题点。
- 文件名避免空格，必要时使用 `-` 分隔。

子文档使用以下模板：

```markdown
# Bug N：简短标题

- 状态：候选 / 已确认 / 修复中 / 已修复 / 已提交 / 暂不修复
- 位置：`path/to/file.go`
- 预期来源：公开文档 / 已有测试 / API 命名 / 一致性行为 / 运行错误 / 其他。
- 问题：描述错误行为。
- 影响：描述用户可见影响或潜在风险。
- 复现：说明复现方式，优先引用测试名。
- 修复：概述修复思路。
- 验证：记录执行过的测试命令和结果。
- 提交：`commit-hash commit message`
- 备注：记录限制、取舍或后续风险。
```

## 提交要求

- 一个 bug 对应一个 commit。
- 提交前确认工作区只包含当前 bug 的相关修改。
- 每个 commit 只能包含当前 bug 的代码、测试、该 bug 子文档和主索引更新。
- commit message 使用简洁英文或中文均可，但必须说明具体修复点。
- 不要把纯扫描记录、多个无关修复或格式化大改混入同一个 commit。

## 验收标准

任务结束时应满足：

- 已扫描的范围清晰可追溯。
- 所有发现的 bug 都有独立子文档。
- 本文档只维护 bug 索引、总流程和全局要求，不承载单个 bug 的完整细节。
- 每个已修复 bug 都有对应验证记录。
- 每个已修复 bug 都有独立 Git commit。
- 当前工作区不遗留未说明的修改。
