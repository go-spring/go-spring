# 领域分层架构（Domain Layered Architecture）

本文介绍 Go-Spring 单服务项目在 domain 形态下的内部领域分层架构。项目顶层通用目录由其他文档说明，本文只聚焦 `internal/` 内的领域模型、四层边界、代码归属、演进约束以及与 AI 协作时需要遵守的规则。

本文回答四个问题：

- **什么时候适合领域分层**：什么项目需要它，什么项目可以保持更轻量。
- **领域代码如何组织**：从协议入口、用例编排、领域规则到基础设施实现，各自归属明确。
- **边界如何长期不走样**：依赖方向、事务、错误、测试、命名和演进触发条件保持一致。
- **如何与 AI 协作**：给 AI 足够上下文，并要求它按分层规则生成、修改和自检代码。

## 阅读方式

建议按以下顺序使用本文：

1. 新项目选型时，先看「架构定位」和「适用范围」，判断是否需要领域分层。
2. 搭建领域分层骨架时，按「领域分层目录结构」组织 `internal/`。
3. 写业务代码时，参考「代码归属速查表」「分层与边界约束」和「端到端流转示例」。
4. 做代码评审时，重点检查「错误处理」「事务与幂等」「测试策略」和「常见误用」。
5. 让 AI 参与实现时，直接套用「与 AI 协作」中的上下文清单和任务模板。

## 架构定位

本架构面向领域复杂度较高的单服务 Go 项目。它保留 DDD 战术设计中最能约束复杂度的部分：聚合根、充血模型、值对象、领域事件、清晰的用例编排和严格的依赖方向；同时结合 Go 隐式接口、Go-Spring IoC 与服务内分层治理，裁剪掉提前抽象和形式化过重的实践。

### 适用范围

| 场景 | 是否适用 | 判断依据 |
|---|---|---|
| 简单 CRUD、后台管理、小工具 | 不优先使用 | 业务规则少，直接分 handler / service / repo 更轻。 |
| 有明确状态机、金额、库存、权限、履约等规则 | 适用 | 业务不变量需要长期集中维护，适合沉到 `domain`。 |
| 一个服务承接多协议入口（HTTP / RPC / MQ / Job） | 适用 | 入向协议需要被隔离在 `api`，避免污染用例与领域模型。 |
| 依赖多个外部系统或第三方 SDK | 适用 | 外部模型和错误语义需要通过 `infra/client` 防腐。 |
| 子域边界逐渐稳定，未来可能拆服务 | 适用 | 服务内先按 BC 或子域组织，后续拆分成本更低。 |
| 需求还在快速试错，领域术语不稳定 | 谨慎使用 | 可以先保持简单结构，等规则沉淀后再迁移到领域分层。 |

核心原则是：复杂性必须来自业务本身，而不是来自架构仪式。**默认不提前创建接口、端口、适配器或跨层抽象**；只有真实变化点已经出现时，才引入新的结构。

### 与 DDD 的关系

本架构采用 DDD 的关键约束，但不追求完整仪式化。凡是能约束领域复杂度的要素，默认保留；仅出于形式或提前抽象的部分，默认省略，等出现真实触发条件后再引入。

| DDD 要素 | 本架构采纳方式 |
|---|---|
| 聚合根与充血模型 | **必须**：业务规则和不变量落在 `domain`；**禁止**把领域对象退化成只有字段的贫血模型。 |
| 值对象 | **必须**：用于表达不可变的领域概念，例如金额、时间窗口、地址、状态组合。 |
| 领域事件 | **推荐按需使用**：聚合产生事件，`application` 在事务边界内收集，并在持久化后发布或交给项目内事件机制处理。 |
| 分层架构 | **必须**：`api`、`application`、`domain`、`infra` 职责分离；**禁止**反向依赖。 |
| Repository 接口 | **默认不预设**：`application` 可直接依赖 `infra/repo` 具体类型；出现多实现、读写分离、缓存装饰等真实变化点后再提取接口。 |
| 防腐层 | **跨 BC 或外部系统必须使用**；同 BC 内部子域协作默认不强制引入防腐层。 |
| 显式接口声明 | **默认不声明**：单实现直接依赖具体类型；出现多态、替换、跨包稳定契约时再声明接口。 |

## 领域分层目录结构

以下目录展示 domain 形态下 `internal/` 内部推荐的基线结构。`order`、`user` 仅作为示例业务域，落地时按项目自身的 BC、子域或核心业务对象替换。

```
internal/
├── api/                                # 入向适配层：协议、路由、请求入口、任务与消息触发入口
│   ├── job/                            # 定时任务、后台 job 等非请求型入口
│   ├── controller/                     # 按业务域纵向拆分的协议入口处理器
│   │   ├── order/                      # 订单业务域
│   │   │   ├── converter/              # IDL 请求/响应模型与 application DTO 的转换器
│   │   │   └── order_controller.go     # 订单业务入口实现
│   │   └── user/                       # 用户业务域
│   │       ├── converter/              # IDL 请求/响应模型与 application DTO 的转换器
│   │       └── user_controller.go      # 用户业务入口实现
│   └── server/                         # 协议入口聚合目录
│       ├── httpsvr/                    # HTTP 协议适配层
│       │   ├── middleware/             # HTTP 专属中间件链
│       │   ├── handler.go              # HTTP handler，组合业务 controller，实现协议接口
│       │   └── httpsvr.go              # HTTP Server 生命周期管理
│       ├── thriftsvr/                  # Thrift RPC 协议适配层
│       │   ├── middleware/             # Thrift 拦截器与链路追踪
│       │   ├── handler.go              # Thrift handler，组合业务 controller，实现协议接口
│       │   └── thriftsvr.go            # Thrift Server 生命周期管理
│       ├── grpcsvr/                    # gRPC 协议适配层
│       │   ├── middleware/             # gRPC Unary / Stream 拦截器
│       │   ├── handler.go              # gRPC handler，组合业务 controller，实现协议接口
│       │   └── grpcsvr.go              # gRPC Server 生命周期管理
│       └── mqsvr/                      # MQ 消费者入口与生命周期管理
│           ├── middleware/             # MQ 消费专属中间件链
│           ├── handler.go              # MQ 消费 handler，组合业务 controller，实现消费接口
│           └── mqsvr.go                # MQ Consumer 注册、启动和关闭
├── application/                        # 应用层：编排领域对象与基础设施能力
│   ├── order/                          # 订单业务域
│   │   ├── assembler/                  # DTO 与 Entity / 聚合的转换器
│   │   ├── dto/                        # application 对外暴露的数据传输对象
│   │   └── order_service.go            # 订单应用服务，用例编排入口
│   └── user/                           # 用户业务域
│       ├── assembler/                  # DTO 与 Entity / 聚合的转换器
│       ├── dto/                        # application 对外暴露的数据传输对象
│       └── user_service.go             # 用户应用服务，用例编排入口
├── domain/                             # 领域层：核心业务规则与业务逻辑
│   ├── order/                          # 订单业务域
│   │   ├── order.go                    # 聚合根与实体
│   │   ├── order_value.go              # 值对象
│   │   └── order_event.go              # 领域事件
│   └── user/                           # 用户业务域
│       ├── user.go                     # 聚合根与实体
│       ├── user_value.go               # 值对象
│       └── user_event.go               # 领域事件
├── infra/                              # 基础设施实现层：访问外部系统与中间件
│   ├── repo/                           # 仓储实现，封装数据库 / 缓存访问
│   │   ├── order/                      # 订单仓储实现
│   │   │   └── order_repo.go
│   │   └── user/                       # 用户仓储实现
│   │       └── user_repo.go
│   ├── client/                         # 外部服务 / SDK 调用，承担防腐层职责
│   │   └── uranus/                     # Uranus 平台服务调用封装
│   └── mq/                             # MQ Producer；消费者入口保留在 api/server/mqsvr
├── pkg/                                # 服务内部通用工具，不承载业务语义
│   ├── stringutil/                     # 字符串工具
│   ├── timeutil/                       # 时间工具
│   └── safego/                         # 安全并发工具
├── consts/                             # 全局共享常量
│   └── errno/                          # 全局错误码定义
└── init.go                             # side-effect import 与注册逻辑聚合入口
```

### 代码归属速查表

当不确定一段代码应该放在哪里时，先按职责判断，再按依赖方向验证。

| 代码内容 | 放置位置 | 判断标准 |
|---|---|---|
| HTTP / RPC / MQ / Job 的入口、路由、协议 handler | `api/server/*`、`api/job` | 只处理入向协议和生命周期，不写业务规则。 |
| 请求参数校验、协议模型与 DTO 转换 | `api/controller/<biz>/converter` | 输入来自 IDL / 协议层，输出是 application DTO。 |
| 一个业务用例的流程编排 | `application/<biz>/*_service.go` | 需要协调聚合、仓储、外部服务、事务或幂等。 |
| DTO 与领域对象转换 | `application/<biz>/assembler` | 转换发生在 application 与 domain 边界。 |
| 状态流转、金额计算、库存扣减、权限判定等业务规则 | `domain/<biz>` | 属于领域不变量，必须由聚合根或领域对象表达。 |
| 不可变领域概念 | `domain/<biz>/*_value.go` | 有业务语义，**必须**保持不可变，不能被外部随意修改。 |
| 领域内已经发生的事实 | `domain/<biz>/*_event.go` | 事件由聚合产生，命名表达过去式事实。 |
| DB / cache 持久化实现 | `infra/repo/<biz>` | 封装 GORM、SQL、Redis 等技术细节。 |
| RPC / SDK / 第三方平台调用 | `infra/client/<system>` | 跨 BC 或外部系统，必须做协议、模型和错误适配。 |
| MQ Producer | `infra/mq` | 出向消息能力；消费入口仍属于 `api/server/mqsvr`。 |
| 无业务语义的工具函数 | `pkg/<func>` | 可以被多个层引用，但不能携带业务概念。 |
| 错误码、全局常量 | `consts/errno`、`consts/*` | 跨层共享且稳定，避免散落在业务实现中。 |

## 分层与边界约束

四层是本架构控制复杂度的核心手段：`api` 负责入向协议适配，`application` 负责用例编排，`domain` 承载业务规则，`infra` 承担出向技术实现。代码归属以职责为准，而不是以当前文件所在位置为准。

### 四层职责

| 层级 | 职责 | 禁止事项 |
|---|---|---|
| `api` | 承接 HTTP / RPC / MQ / Job 等入向流量，完成协议解析、参数校验、请求/响应模型转换。 | **禁止**直接调用 `infra`，**禁止**承载业务规则，**禁止**返回 `domain` 对象。 |
| `application` | 组织用例流程，编排 `domain` 聚合与 `infra` 能力，负责事务边界、幂等控制与领域事件发布。 | **禁止**实现核心业务规则，**禁止**泄漏事务对象，**禁止**把外部模型传入 `domain`。 |
| `domain` | 定义聚合根、实体、值对象、领域事件，承载业务不变量与领域行为。 | **禁止**依赖 `api` / `application` / `infra`，**禁止**感知 DB、RPC、SDK、IDL、DTO。 |
| `infra` | 实现 DB、缓存、RPC、SDK、MQ Producer 等出向能力，屏蔽外部系统差异。 | **禁止**编排业务用例，**禁止**决定业务规则，**禁止**直接面向协议入口。 |

### 依赖方向

依赖方向必须清晰、可检查。`pkg` 与 `consts` 作为无业务语义的通用能力，可被所有层引用，但不能反向引用业务包。

```
                ┌────────────────────────┐
                │       pkg / consts     │
                └───────────▲────────────┘
                            │
     ┌──────────────┬───────┴──────┬──────────────┐
     │              │              │              │
  ┌───────┐    ┌───────────┐    ┌────────┐    ┌────────┐
  │  api  │───▶│application│    │ domain │◀───│ infra  │
  └───────┘    └─────┬─────┘    └────▲───┘    └───▲────┘
                     │               │            │
                     └───────────────┴────────────┘
```

允许依赖关系：

| 来源层 | 允许依赖 | 禁止依赖 |
|---|---|---|
| `api` | `application`、`pkg`、`consts`、协议生成代码 | `domain`、`infra` |
| `application` | `domain`、`infra`、同 BC 内其他 `application` 子域 service（单向无环）、`pkg`、`consts` | 协议生成代码、`api/server` |
| `domain` | 自身包、`pkg`、`consts/errno`、`errutil` | `api`、`application`、`infra`、ORM、RPC SDK、IDL、DTO |
| `infra` | `domain`、`pkg`、`consts`、外部 SDK / DB / MQ 依赖 | `api`、协议入口 handler |

说明：

- `infra/repo` 可以依赖 `domain`，因为仓储实现需要把存储模型转换为领域聚合。
- `application` 可以依赖 `infra` 具体类型，这是 Go-Spring 项目的默认简化策略。
- 同 BC 内 `application` 子域之间可以单向协作，但**禁止**形成循环依赖；一旦出现双向调用，应改用领域事件、只读查询服务或重新划分用例边界。
- `domain` 必须保持最稳定，任何外部协议、存储结构或 SDK 类型进入 `domain`，都意味着边界已经被打穿。

### 跨域协作与防腐层

跨域协作先区分两类边界：

- **同 BC 内子域**：**推荐**通过 `application` 层 service 单向协作，例如 `order` 编排 `user` 的查询服务。**禁止**直接 import 对方 `domain` 模型；一旦出现双向依赖或循环依赖，改用领域事件、只读查询服务或重新划分用例边界。
- **跨 BC 或外部系统**：**必须**通过 `infra/client/` 下的防腐层隔离。业务层只面向本域模型编程，不感知外部协议、字段命名、错误码和 SDK 类型。

防腐层至少承担三件事：

- **协议适配**：把 HTTP / RPC / SDK 调用细节收敛在 `infra/client/<system>`。
- **模型映射**：把外部模型转换为本域可理解的值对象、DTO 或查询结果。
- **错误转换**：把外部错误包装成带上下文的内部错误，必要时映射到 `consts/errno`。

判断是否需要防腐层的标准很直接：只要跨越 BC 边界、外部模型语义与本域不一致，或涉及网络 IO / RPC / 第三方 SDK，就**必须**进入 `infra/client/`。

## 领域建模约定

领域层不是字段定义目录，而是业务规则的落点。写 `domain` 代码时，优先用方法表达行为，而不是让外层随意改字段。

### 聚合根

聚合根是外部修改聚合状态的唯一入口。它负责维护本聚合内的不变量，例如状态流转是否合法、金额是否匹配、库存是否允许扣减。

约定：

- **推荐**聚合根方法用业务动作命名，例如 `Pay`、`Cancel`、`ConfirmReceipt`，避免只暴露 `SetStatus`。
- **必须**由聚合根维护本聚合内的不变量；外部只能通过聚合根方法触发状态变化。
- **禁止**聚合根启动事务、访问仓储或调用外部服务。
- **允许**聚合根产生领域事件，但**禁止**由聚合根负责发布事件。

### 实体和值对象

实体有身份和生命周期，值对象表达不可变的业务概念。

约定：

- **必须**用 ID 区分实体身份，即使字段相同也可能不是同一个对象。
- **必须**保持值对象不可变；需要变化时创建新值。
- **必须**由值对象负责自身合法性校验，例如金额不能为负、时间窗口起止顺序合法。
- **禁止**把 DTO、ORM model、IDL 生成类型当作实体或值对象使用。

### 领域事件

领域事件表达领域内已经发生的事实，通常由聚合根在状态变化时产生。

约定：

- **推荐**事件命名使用过去式事实，例如 `OrderPaid`、`OrderCanceled`。
- **推荐**事件字段只包含订阅方需要的稳定信息，**禁止**暴露整个聚合。
- **必须**由 `application` 在事务边界内收集事件，并在持久化成功后发布或交给项目事件机制处理。
- **推荐**领域事件用于解耦跨聚合、跨子域或异步补偿；**禁止**用事件绕开明确的同步用例编排。

## 端到端流转示例

以「用户下单」为例，一个完整请求依次穿过四层：

```
HTTP 请求
   │
   ▼  api/server/httpsvr/handler.go          协议解析、路由分发
   │
   ▼  api/controller/order/order_controller  参数校验、请求 → application DTO
   │
   ▼  application/order/order_service        用例编排、事务边界、编排 domain 与 infra
   │        │
   │        ├── domain/order                  业务不变量校验，产出领域事件
   │        │
   │        ├── infra/repo/order              持久化订单聚合
   │        │
   │        └── infra/client/uranus           调用外部系统，完成防腐转换
   │
   ▼  application/order/assembler             Entity / 聚合 → DTO
   │
   ▼  api/controller/order/converter          DTO → 协议响应模型
   │
   ▼  HTTP 响应
```

关键流转规则：

- **入参转换**：协议模型在 `api/controller/*/converter` 转成 `application/*/dto`；`domain` 与 `infra` **禁止**感知协议模型。
- **用例编排**：`application` 负责查聚合、调用领域行为、持久化、调用外部服务、处理事务与幂等。
- **业务规则**：所有不变量校验**必须**由 `domain` 聚合根、实体或值对象承担。
- **出参转换**：`domain` 对象在 `application/*/assembler` 转成 DTO，再由 `api/controller/*/converter` 映射成协议响应。
- **外部调用**：`application` 调用 `infra/client` 的本域语义方法，**禁止**直接处理外部协议模型。

## 错误处理分层约定

错误处理沿分层单向流动。每一层只处理自己能识别的错误，跨层时补充上下文，不吞掉原始错误。

| 层级 | 错误来源 | 处理方式 |
|---|---|---|
| `domain` | 业务规则错误，例如状态非法、金额越界、库存不足 | 使用 `consts/errno` 表达业务错误码；需要补充上下文时用 `errutil.Explain`。 |
| `infra` | 技术性错误，例如 DB 断连、RPC 超时、SDK 返回异常 | 用 `errutil.Stack` 保留调用路径，必要时映射到 `consts/errno` 中的技术错误码。 |
| `application` | 编排失败、事务失败、跨依赖组合错误 | 用 `errutil.Explain` 补上业务上下文，例如订单 ID、用户 ID、外部系统名。 |
| `api` | 参数错误、协议错误、下层返回错误 | 统一把 `errno` 错误码映射为协议响应，例如 HTTP status、Thrift Exception、gRPC status。 |

约定要点：

- 返回错误时**必须**带足够上下文，能定位到失败的用例、关键 ID 或外部依赖。
- `errutil` 负责堆栈与上下文，`errno` 负责错误码定义，二者职责正交。
- `domain` **禁止**依赖 DB、RPC、SDK 的错误类型，也**禁止**根据技术错误做业务判断。
- `api` 只做面向外部的错误呈现，**禁止**在协议层补写业务决策。

## 事务与幂等约定

事务和幂等属于 `application` 的用例边界，不能散落到 `api`、`domain` 或 `infra` 中。

- **事务边界**：一个 `application` service 方法即一个事务单元，事务开启与提交都由该方法内部完成；**禁止**向上暴露事务对象。
- **单聚合强一致**：聚合根内部的多次状态变更在同一事务中提交；`domain` 只操作内存中的聚合，**禁止**感知事务对象。
- **跨聚合不做分布式事务**：涉及多个聚合或多个 BC 的一致性需求，**推荐**通过领域事件、补偿任务或最终一致性处理，避免长事务与跨库两阶段提交。
- **幂等责任在 application**：入口按业务 ID / 请求 ID 做幂等校验；`infra/repo` **推荐**通过唯一索引、乐观锁等方式兜底。
- **禁止跨 service 事务嵌套**：需要复合流程时，**推荐**上升为一个新的编排 service，而不是让多个 service 互相调用后再外包一层事务。

## 扩展与演进约定

默认形态保持简单。只有复杂度真实出现，并且触发条件明确时，才启用对应的演进方式。

| 默认做法 | 触发条件 | 演进方式 |
|---|---|---|
| `application` 直接依赖 `infra/repo` 具体类型 | 出现主从、缓存装饰、读写分离、替换存储等多种 Repository 实现 | 提取 Repository 接口。若接口表达聚合持久化语义，放在 `domain/<biz>`；若只是应用编排端口，放在使用方包。 |
| 同 BC 子域通过 `application` service 协作 | 出现跨团队、多仓库协作，或子域边界频繁变化 | 提取稳定契约，必要时引入防腐层隔离语义。 |
| 保持服务内四层目录 | 子域需要独立部署、独立发布或独立扩容 | 将子域提取为独立服务，沿用原有分层结构。 |
| 测试默认使用 Go-Spring Mock 替换依赖 | 测试框架独立性要求强，或关键依赖有多实现替换诉求 | 仅对真实变化点提取接口，其余依赖保持直接连接。 |
| `infra/client` 直接封装外部系统 | 外部系统模型复杂、调用场景多、错误语义不稳定 | 在 `infra/client/<system>` 内部继续拆分 mapper、transport、error adapter，但不泄漏到上层。 |

接口提取规则：

- **禁止**为“未来可能”提取接口。
- **禁止**只为单元测试提取接口。
- **推荐**普通接口放在使用方所在包；Repository 接口只有在表达领域持久化契约时才放入 `domain`。
- 接口一旦提取，**必须**能描述稳定语义，而不是简单复制某个实现的方法列表。

## 工程组织约定

除了分层与边界，工程侧还需要统一命名、组装、注册入口和生命周期管理。

### 文件命名

文件是否带领域前缀，按“文件中的类型是否会被外部包识别”判断。

- **推荐带前缀**：文件中的类型会被跨包注入、嵌入或构造，例如 controller、service、SDK 实现。示例：`order_controller.go`、`order_service.go`、`order_sdk.go`。
- **推荐不带前缀**：文件只承载包内辅助角色，不会被外部直接引用，例如 converter、assembler、dto。示例：`converter.go`、`dto.go`、`assembler.go`。

### Handler 组装

`api/server/<proto>/handler.go` 采取嵌入组合的方式聚合 `api/controller/` 下的业务类型，本身不承载业务方法体。新增业务方法只需在 controller 包中实现，handler 通过嵌入获得对应方法。这样即使协议接口方法数量上升，handler 文件也不会随业务膨胀。

### pkg 组织

`pkg/` 只承载无业务语义的通用工具，**推荐**按职能维度独立拆包，例如 `stringutil/`、`timeutil/`、`sliceutil/`、`safego/`。**禁止**新增 `common/`、`goutil/`、`helper/` 这类聚合包。

### init.go 导入

`init.go` 只处理需要 side-effect import 的入口或注册逻辑，按流量方向区分：

- **入向入口**：`api/server/*`、`api/job` 等**需要**写入 `init.go`，触发路由、任务或消费者注册。
- **出向依赖且会被直接 import**：**不写入** `init.go`，上层显式 import 时会自然完成初始化。
- **出向能力没有显式 import，但 `init()` 有注册逻辑**：**需要**写入 `init.go`，例如需要自动注册的 MQ producer 或插件。

### 装配与生命周期

所有跨层依赖统一通过 Go-Spring IoC 容器装配，不使用包级单例或全局变量长期持有依赖。

- **Bean 声明就近**：Bean **推荐**在实现所在文件中声明，**不推荐**集中到统一装配文件，避免形成中心依赖。
- **启动顺序交给容器**：业务代码**禁止**显式控制初始化顺序；有硬性顺序时，**推荐**通过 `PostConstruct` / `Destroy` 钩子表达。
- **优雅关闭分层负责**：`api/server/*` 负责入向连接排空与关闭，`infra/*` 负责连接池、生产者、订阅等出向资源释放。
- **禁止绕过容器**：`application`、`infra` 之间的依赖**必须**通过 IoC 注入获得，**禁止**包内 `New*()` 直接构造后长期持有。

## 测试策略分层

测试策略与分层一一对应。每一层使用最贴合职责的测试方式，避免用集成测试覆盖单元逻辑，或用单元测试冒充集成。

| 层级 | 测试类型 | 依赖处理 |
|---|---|---|
| `domain` | 纯单元测试 | 无外部依赖，直接构造聚合、实体和值对象，验证业务不变量与领域行为。 |
| `application` | 单元测试 | 使用 Go-Spring Mock 替换 `infra` 依赖，聚焦用例编排、事务和幂等分支。 |
| `infra/repo` | 集成测试 | 连接真实 DB 或容器化中间件，验证持久化与查询语义。 |
| `infra/client` | 契约测试 | 对第三方 SDK 或外部服务使用替身，重点验证模型和错误映射。 |
| `api` | 端到端测试 | 启动完整 server，通过协议入口发起请求，验证全链路。 |

补充原则：

- `domain` 单测优先级最高，业务不变量**必须**由这一层保证。
- **禁止**为了测试而给 `application` 的所有依赖预留接口；Go-Spring Mock 能替换具体类型时，**推荐**保持具体依赖。
- 集成测试**推荐**重点覆盖 `infra/repo` 与关键 `infra/client`，避免测试金字塔倒挂。
- 每个新用例**至少**补齐领域规则测试；涉及协议入口时，再补 controller 或端到端测试。

## 与 AI 协作

领域分层项目适合让 AI 辅助生成骨架、补齐转换器、编写测试和做依赖边界检查，但**必须**给出清晰上下文。实际有效的经验是：把 AI 当成高效的实现者、审查者和测试补全者，而不是让它替团队决定领域边界。上下文不足时，AI 很容易把业务规则写到 `application`，或提前生成一堆接口、manager、common 包。

### 协作边界

AI 协作的边界要按风险区分：规则越明确、改动越机械，越适合放给 AI；领域判断越重、后果越难回滚，越需要人先定方向。

| 类型 | 建议 | 说明 |
|---|---|---|
| converter、assembler、DTO、测试样例、依赖检查 | **可以放宽** | 输入输出明确，错误容易 review，AI 提效明显。 |
| application 用例编排 | **有条件放宽** | 需要先给出事务边界、幂等键、错误码和外部依赖；缺一项时要求 AI 先提问。 |
| 聚合边界、状态机、不变量、跨 BC 协作方式 | **必须收窄** | 这些属于领域设计决策，AI 可以给方案和风险，但不能直接替团队拍板。 |
| 新接口、抽象层、manager / processor 类结构 | **默认禁止** | 除非已经给出真实触发条件，否则 AI 往往会为了“解耦”制造复杂度。 |
| 大范围重构 | **分阶段执行** | 先让 AI 输出影响范围和迁移步骤，再小步修改并跑测试，不要一次性改穿多层。 |

### 适合交给 AI 的任务

| 任务 | 适合程度 | 约束方式 |
|---|---|---|
| 新增一个用例的 controller / DTO / service / domain 方法骨架 | 有条件适合 | 必须明确业务动作、涉及聚合、事务边界、幂等键和错误码；缺失时先让 AI 提问。 |
| 补 converter / assembler | 很适合 | 明确源模型、目标模型和字段语义，不让 AI 猜外部字段含义。 |
| 编写 `domain` 单元测试 | 很适合 | 给出业务不变量、合法状态流转和反例。 |
| 检查依赖方向 | 很适合 | 要求 AI 枚举 import 违规、DTO 泄漏和跨层调用。 |
| 补充错误包装、日志上下文和测试覆盖 | 适合 | 给出错误码、关键 ID 和运行命令，要求 AI 不吞掉底层错误。 |
| 提取接口或重构边界 | 谨慎 | 必须先说明触发条件，禁止 AI 为了“解耦”主动抽象。 |
| 设计聚合边界或跨 BC 协作方案 | 只适合讨论 | 要求 AI 给出方案、约束和风险；最终边界由人确认后再落地。 |

### 给 AI 的上下文清单

让 AI 修改代码前，**推荐至少**提供以下信息：

- **业务动作**：这次要实现的用例是什么，例如“用户下单”“支付订单”“取消订单”。
- **领域术语**：聚合根、实体、值对象、状态枚举和关键业务规则。
- **边界说明**：哪些属于同 BC，哪些是外部系统或跨 BC 调用。
- **输入输出**：协议请求/响应、application DTO、需要返回的错误码。
- **事务与幂等**：事务范围、幂等键、并发冲突处理方式。
- **外部依赖**：DB 表、缓存 key、RPC / SDK、MQ topic，以及错误映射要求。
- **测试要求**：必须覆盖的正例、反例、边界条件和要运行的测试命令。

如果上述信息缺失，**推荐要求 AI 先列出缺口和假设**。除非任务只是 converter、assembler、测试补全或依赖检查，否则不要让 AI 在业务规则不完整时直接实现。

### AI 任务模板

可以直接使用下面的模板给 AI 派任务：

```text
请在 internal 中实现「<业务动作>」用例。

业务规则：
- 聚合根：<聚合名>
- 状态流转：<从什么状态到什么状态>
- 不变量：<金额、库存、权限、时间窗口等规则>
- 错误码：<errno 名称或新增要求>

分层要求：
- api 只做协议模型与 application DTO 转换，不直接调用 infra。
- application 负责事务、幂等、编排 domain 与 infra。
- domain 承载业务规则，不 import api/application/infra/IDL/ORM/SDK。
- infra 只封装 DB、缓存、RPC、SDK 或 MQ producer。
- 如果业务规则、错误码、事务边界或外部依赖不明确，先提问，不要自行编造。

外部依赖：
- repo：<需要读写的数据>
- client：<需要调用的外部系统>
- mq：<需要发送的消息，如无则写无>

测试要求：
- domain 单测覆盖：<规则列表>
- application 测试覆盖：<编排与错误分支>
- 运行命令：go test ./...

交付时请说明：
- 修改了哪些文件。
- 哪些规则放在 domain，哪些编排放在 application。
- 是否新增接口；如新增，说明触发条件。
- 测试结果。
```

### AI 自检清单

要求 AI 交付前先按文末「落地检查清单」自检，并额外确认以下事项：

- 是否说明了业务规则放在 `domain` 的哪个聚合、实体或值对象中。
- 是否说明了 `application` 只做了哪些编排、事务、幂等或事件发布工作。
- 是否列出了实现中使用的假设；如果有不确定项，是否停止实现并请求确认。
- 是否新增接口；如果新增，是否给出真实触发条件。
- 是否列出测试覆盖点和实际运行结果。

### AI 协作注释

当某段代码有明确约束，且后续可能被 AI 误改时，可以使用局部注释约束行为，例如：

```go
// AI: do NOT move this validation to application; it is a domain invariant.
```

这类注释**只用于关键边界**，**禁止**到处添加。注释应说明“不能做什么”和“为什么”，避免变成重复代码含义的噪音。

## 落地检查清单

代码合入前，按下面清单快速检查：

- 业务规则是否集中在 `domain`，而不是散落在 `api` 或 `application`。
- `api` 是否只做协议适配、参数校验和 DTO 转换。
- `application` 是否清晰表达一个用例，并拥有事务与幂等边界。
- `infra/client` 是否隔离了外部协议、模型和错误。
- `infra/repo` 是否封装了存储细节，没有把 ORM model 泄漏给上层。
- 错误是否用 `errutil.Explain` / `errutil.Stack` 保留上下文。
- 是否避免了无触发条件的接口、manager、common 包和过度抽象。
- 测试是否覆盖了 `domain` 不变量，以及关键编排和基础设施语义。

## 常见误用与反例

规则易记，但架构走坏通常发生在一些“看起来能跑”的写法上。评审时优先排查以下问题：

| 反例 | 问题 | 纠正方式 |
|---|---|---|
| `domain` import `infra/repo`、`gorm`、RPC SDK 或 IDL 类型 | 业务规则耦合到技术实现或外部协议，违反依赖方向。 | `domain` 只保留领域对象和业务规则；持久化、协议和 SDK 由外层转换。 |
| `api/controller` 直接调用 `infra/repo` 或 `infra/client` | 绕过 `application`，用例编排和协议入口混在一起。 | 所有跨层调用都经 `application` service。 |
| `application` 实现价格计算、状态机、库存规则 | `domain` 被架空，变成 CRUD 数据结构。 | 把规则移回聚合根、实体或值对象，`application` 只做编排。 |
| `api` 直接返回 `domain.Entity` | 领域模型泄漏到协议层，后续修改会牵动外部契约。 | 经 `application` DTO 与 `api` converter 两次转换。 |
| `infra/repo` 返回 ORM model 给 `application` | 存储结构泄漏到用例层，领域对象失去意义。 | repo 返回领域聚合或 application 需要的查询结果。 |
| 跨 BC 直接 import 对方 `domain` 模型 | 打破 BC 隔离，后续拆服务困难。 | 通过 `infra/client/` 防腐层调用，双方只面向本域模型。 |
| 为单实现依赖提前定义接口 | 增加认知成本，接口也通常无法预判后续变化。 | 保持具体类型依赖，出现真实多态诉求时再抽取接口。 |
| `pkg/` 下新增 `common/`、`goutil/`、`helper/` | 工具能力失去边界，逐渐堆积难维护。 | 按职能拆包，每个子包只承载一类能力。 |
| 事务对象从 `application` 传到 `domain` 或跨多个 service 传递 | 事务边界外泄，调用链难以理解和测试。 | 由一个 application service 拥有完整事务边界。 |
| AI 生成大量 `manager`、`processor`、`adapter` 但职责不清 | 名称掩盖职责，目录复杂度上升。 | 回到四层职责，用业务动作、聚合或外部系统命名。 |
