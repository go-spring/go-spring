# Go-Spring 编码原则与风格规范

本文档定义 Go-Spring 项目的编码原则、习惯与风格约定，所有贡献者修改或添加代码时都应遵守。

---

## 0. 简单优于聪明

- **清晰优先**：追求清晰、直观、可维护，拒绝炫技式写法
- **避免过度工程**：不用过度抽象和隐式行为，复杂性应来自问题本身而非实现方式
- **读者友好**：选最直接易懂的方案，代码对读者友好而非作者
- **重写原则**：需额外解释才能理解的实现，应用更简单方式重写

## 1. 包与设计原则

- **单一职责**：每个包聚焦一个功能领域，小巧专注，便于测试维护
- **可见性控制**：严格区分公开 API（`PascalCase`）与内部实现（`camelCase`），暴露必要的，隐藏内部的
- **零值可用**：通过默认值、延迟初始化或内部兜底保证零值安全，避免"未初始化即 panic"
- **避免全局状态**：不使用全局变量，通过 Go-Spring 的 IoC/DI 注入获取配置、单例和客户端

## 2. 命名约定

- **包名**：全小写，简短且具描述性（`errutil`, `assert`, `gs`），不用下划线或驼峰，不与包内内容重复
- **标识符**：遵循 Go 惯例，公开用 `PascalCase`，内部用 `camelCase`
- **常量**：使用 `UPPER_SNAKE_CASE`
- **错误变量**：预定义错误以 `Err` 前缀开头（`ErrNotFound`）
- **接口**：单方法接口以 `-er` 后缀结尾（`Handler`, `Provider`），大型接口用描述性名词
- **变量**：简洁而有意义，避免不必要的长名称
- **方法接收者**：简短一致（1-2 个字母），不用 `me`/`this`/`that`

## 3. 代码格式与组织

- **标准 Go 格式**：严格遵循 `gofmt`
- **导入分组**：`标准库` → `外部依赖` → `内部依赖`，各组空行分隔
- **函数长度**：偏好短小函数，一个函数只做好一件事
- **行长度**：适度紧凑，避免过长单行，但不刻意限制行数
- **空行使用**：用空行分隔逻辑块，避免密集堆砌
- **代码清理**：过时代码直接删除，不保留注释掉的死代码

### 3.1 传参

复合表达式实参满足**任一**条件时，提取为语义化具名变量再传入：

- **多层嵌套**（≥2 层，拆开才看清每步意图）
- **被复用**（≥2 次，顺带去重）
- **命名能补充信息**（表达式本身看不出业务含义）

单层、只用一次、函数名已自明的调用可内联，强行提取反增噪音（呼应第 0 节「读者友好」）。例如 `fmt.Sprintf(..., strings.ReplaceAll(name, "/", "_"))` 留内联；`path.Dir(filepath.ToSlash(strings.TrimPrefix(file, "./")))` 该拆。

### 3.2 文件内代码组织

目标：从上往下顺读即可理解，减少跳转。

应遵守（review 会提）：

- **类型自成一片**：类型 + 它的方法 + 只服务于它的私有帮手连续摆放，中间不插无关函数
- **辅助定义就近放**：只被某处用到的小函数、专属常量/变量贴着使用方，不上浮到文件顶部
- **归属冲突优先贴类型**：帮手同时被某类型方法和其他函数用到时，归入该类型那片
- **满足任一条件即整片抽到同名新文件**：可独立测试、被多处复用、或显著超过一屏

倾向（可破例）：

- **大体自顶向下**：入口/主流程靠前，细节靠后
- 与 Go 习惯（bottom-up、集中 const 块）冲突时，以项目一致性优先

本规则无法自动校验，仅作编写与 review 取向，不作硬门禁。

## 4. 错误处理哲学

项目统一使用 `stdlib/errutil` 的**双语义错误包装模式**：

> **项目规则**：不用 `errors.New`/`fmt.Errorf` 直接构造错误，统一通过 `errutil` 包装

- **解释性包装**（`errutil.Explain`）- 添加业务语义：`errutil.Explain(err, "failed to connect to database")`
- **路径包装**（`errutil.Stack`）- 跟踪调用链路：`errutil.Stack(err, "InitService")`
- **快错早返回**：业务错误尽早返回；初始化阶段不可恢复的编程错误直接 panic
- **保持解包链**：`errutil` 内部保证 `%w` 语义，完整支持 `errors.Is()`/`errors.As()`
- **足够上下文**：错误信息应包含足以定位来源的上下文

## 5. 文档与注释

- **包文档**：每个公开包必须有包注释，讲清"做什么""为什么"和使用场景，不纠缠实现细节
- **函数文档**：每个导出函数必须有注释——功能描述、参数与返回值、错误情况（如有），复杂场景可附示例
- **代码自文档**：用清晰命名和简洁结构让代码自解释，不加不必要的注释
- **AI 协作注释**：需约束 AI 行为时可加特殊注释，例如 `// AI: do NOT refactor this function`

## 6. 测试风格

- **工具库优先**：优先用 `stdlib/testing` 的 `assert`（失败继续）/`require`（失败终止）断言，不引入 testify 等第三方库；默认用 `assert`，仅当失败会导致后续代码 panic 或失去意义时才用 `require`（如解引用前的 nil 检查）；不想引入额外依赖时才直接用标准库 `testing`
- **子测试分组**：用 `t.Run()` 对不同场景逻辑分组
- **表驱动测试**：多输入输出场景推荐表驱动模式；表驱动与子测试可混用，哪个简洁用哪个
- **原始断言的边界**：仅在没有对应断言的场景才用 `t.Error`/`t.Fatal` —— 如超时保护、`select` 分支或无法恢复的初始化失败
- **测试伴随生产**：测试文件与生产代码放在同一包目录，测试即活文档

## 7. Go 语言习惯

- **原汁原味 Go**：适配 Spring 理念但不强行面向对象，保持 Go 自然习惯
- **正确使用 context**：请求流必须携带 `context.Context`，尽量避免随意使用 `context.TODO()`/`context.Background()`
- **边界检查**：在 API 边界做输入验证，尽早发现错误
- **避免过度抽象**：只在真正需要时抽象，遵循 YAGNI，不为不确定的未来预规划
- **最小依赖**：只添加必要的外部依赖

## 8. 并发安全设计

- **默认非并发安全**：除非明确设计支持，否则不保证并发安全，由调用者决定是否同步
- **明确声明**：若类型支持并发访问，文档必须说明
- **正确使用同步原语**：`sync/atomic` 用于原子标志，`sync.Mutex` 保护复杂状态，`channel` 用于通信
- **封装并发细节**：将复杂同步逻辑封装在简洁 API 之后

---

**总体原则**：整洁一致的工艺。在保持 Go 自然习语与性能的同时，平衡 Spring 熟悉的编程模型，通过直观 API 和详尽文档专注开发者体验。
